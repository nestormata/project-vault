# Adversarial Review — Story 7.1: Machine User Identity & API Key Management

- **Date:** 2026-07-04
- **Reviewed file:** `_bmad-output/implementation-artifacts/7-1-machine-user-identity-and-api-key-management.md`
- **Reviewer:** bmad-review-adversarial-general

## Findings

- **[HIGH] Audit-payload "safety net" claim is factually wrong for the code path this story mandates.** AC-9 asserts: *"the `FORBIDDEN_AUDIT_KEYS` set in `secure-route.ts` already strips any payload key named `apiKey`, but this route does not rely on that alone."* Verified against the actual code: `sanitizeAuditPayload`/`isForbiddenAuditKey` (`apps/api/src/lib/secure-route.ts:151-160`) is invoked only from `defaultAuditWriter` (`secure-route.ts:210`), which is used exclusively by the *declarative* `security.writeAuditEvent` config path. AC-3/AC-9/AC-13 explicitly mandate `security: { writeAuditEvent: false }` plus a manual `writeHumanAuditEntryOrFailClosed()` call — and that function (`apps/api/src/lib/audit-or-fail-closed.ts` → `apps/api/src/modules/audit/human-entry.ts`) writes `payload` straight into `audit_log_entries` with **zero** sanitization anywhere in that call chain. So for every mutation this story implements, there is in fact **no runtime redaction safety net at all** — AC-15's test requirement is the *only* protection, not a redundant second layer as the story claims. This should be corrected in the story text (or the risk explicitly owned), not asserted as an existing backstop that doesn't apply.

- **[HIGH] `api_keys` RLS design conflicts with Story 7.2's key-hash-only lookup requirement, and the story never surfaces this.** AC-2 gives `api_keys` a standard org-scoped RLS policy (`org_id = current_setting('app.current_org_id')`). But the machine-token exchange endpoint (7.2) must look up a row by `keyHash` alone — before it knows which org the caller belongs to (that's the entire point of authenticating with an opaque key). `architecture.md` documents exactly this chicken-and-egg problem for `sessions`/`refresh_tokens` and explicitly carves them out as "RLS exception tables," accessed only via `withAdminAccess()`. This story — which is the one defining `api_keys`' RLS policy — never mentions this tension or states whether `api_keys` needs the same RLS-exception treatment. This is a foundational schema/security decision being silently deferred to 7.2, risking a late RLS-policy rework on a table this story frames as an already-settled "schema foundation."

- **[MEDIUM] D2's own character-count math contradicts the architecture.md text it claims to follow verbatim.** `architecture.md` (~line 604-606) states the format is *"`pk_` prefix + `crypto.randomBytes(32).toString('base64url')` — 46 chars total."* The story's D2 section instead states *"~49 chars total."* (`pk_` = 3 chars + unpadded base64url of 32 bytes = 43 chars = 46 total, matching architecture.md — the story's own number is arithmetically wrong.)

- **[MEDIUM] AC-9's illustrative response example contradicts the mandated encoding.** The example key is shown as `"pk_9f3a...base64url...=="`, ending in `==` padding characters. Node's `Buffer.toString('base64url')` never emits `=` padding, so this directly contradicts the exact generation method (`pk_` + `randomBytes(32).toString('base64url')`) the story mandates in D2/AC-9.

- **[MEDIUM] AC-4's description-length validation is self-contradictory.** The table lists "description longer than 1024 chars" as the concrete invalid-input trigger, but the same row instructs the developer to "confirm exact value in [credentials.ts's] schema before hardcoding a new one" — i.e., defer to whatever cap `credentials.ts` actually uses. If that cap isn't 1024, the AC's own literal example is wrong, and an integration-test suite that must "cover every AC below" (per the Quick Reference table) could easily hardcode the wrong boundary value from the story text itself.

- **[MEDIUM] `audit_events` vs. `audit_log_entries` naming conflict is handled inconsistently with the story's own methodology.** epics.md's PJ4 explicitly mandates the shared audit table be named `audit_events`; the actual codebase table is `audit_log_entries` (`packages/db/src/schema/audit-log-entries.ts:8`). This is the exact same category of epics.md-vs-reality conflict as D1/D2/D7 (hashing algo, key format, event-name casing) — yet unlike those, it is never elevated to a numbered "Key Design Decision" with its own rationale/citation; it's only mentioned in passing inside the Epic Cross-Story Context table. Given how rigorously the story documents its other conflict resolutions, this asymmetric treatment is an internal inconsistency in the story's own stated methodology.

- **[MEDIUM] No endpoint exists anywhere in this story to deactivate, disable, or delete a machine user, yet AC-11 already builds real enforcement logic around the never-set `deactivatedAt` column.** AC-11's `409 machine_user_deactivated` branch guards against a state that literally cannot occur within this story's scope (nothing sets `deactivatedAt`), making that branch untestable-by-design, while the real operational need — "I created a mistakenly-named/wrong-project machine user, how do I remove it" — has no answer in v1 at all.

- **[MEDIUM] AC-17's concurrency guidance for deciding whether to audit-write is internally ambiguous.** The code hint uses `UPDATE ... SET revoked_at = COALESCE(revoked_at, now())`, where `now()` is generated inside SQL/inside the transaction — yet the same AC says to detect whether *this* transaction caused the change by "comparing the `RETURNING` result's `revokedAt` against what the transaction itself set." That comparison is only meaningful if the timestamp is generated application-side (captured, then compared), not via SQL's `now()` as the snippet shows. This gap in the implementation guidance could easily be implemented incorrectly, producing either duplicate audit rows or a missed one.

- **[MEDIUM] AC-10 covers only `expiresAt` validation for key issuance; there is no equivalent AC for the API key `name` field**, even though `api_keys.name` has the exact same `nameLenCheck` CHECK constraint (AC-2) as `machine_users.name`, and AC-4 exhaustively enumerates empty/omitted/over-length cases for machine-user names. Task 10's integration-test bullet list never mentions "key name validation," so this looks likely to fall through the cracks.

- **[LOW] Rate limiting is scoped per admin+route-template only, not per specific `machineUserId`** (mirrors existing precedent, e.g. `DELETE .../members/:userId` uses a literal, non-interpolated rate-limit key). An admin managing several machine users could exhaust their 10/min key-issuance budget while working on one machine user and be blocked from issuing a key for a completely unrelated machine user for up to 60 seconds. Not a new defect, but "10/min per admin" as phrased in the AC table invites misreading as "per admin per machine user."

- **[LOW] No pagination, filtering, or ordering is specified for `GET .../machine-users` or `GET .../machine-users/:id/api-keys`.** For CI/CD-heavy orgs that create many short-lived keys (explicitly anticipated by AC-17's "overlap during key rotation" scenario), an unbounded list response is a latent scalability gap with no AC coverage.

- **[LOW] `keyHash` has a deliberately non-unique index with no collision defense.** The story dismisses this as cryptographically negligible (true for a *correct* implementation), but there is no DB-level backstop catching an actual bug in the crypto code path (e.g., a broken RNG or an HMAC-secret mixup during rotation) — inconsistent with the story's own stated philosophy elsewhere (AC-1's role CHECK is explicitly framed as "a defense-in-depth backstop, not the primary validation path").

- **[LOW] Proposed RLS policies omit `WITH CHECK`** (`CREATE POLICY ... USING (...)` with no `WITH CHECK` clause), meaning the policy restricts visibility/updates but not what `org_id` a buggy insert path could write. This mirrors an existing systemic gap in migration `0028_monitoring_records.sql`, so it isn't unique to this story, but this story is a fresh opportunity to close it for two brand-new tables and does not.

- **[LOW] AC-3's response body omits `createdBy`** even though the schema (AC-1) defines and persists it. No AC or Dev Note explains why the field is tracked but never surfaced via the API — a gap for any future admin UI or audit-reconciliation need.

- **[LOW] The `machine_user.created` audit payload records only `{ name, role }`, silently omitting `description`.** Since there is no update endpoint, the current row still reflects the original description, so risk is limited — but it's an incomplete audit trail relative to the actual submitted payload, worth flagging given Epic 8 (PJ4) will build compliance tooling directly on top of this table.

- **[LOW] Hard/CASCADE deletion of a project (`ON DELETE CASCADE` from `projects`) silently cascade-deletes all associated `machine_users`/`api_keys` rows with no explicit audit trail of that cascade.** No AC addresses whether project-deletion audit context records "N machine users and M keys were also removed" — a blind spot for a feature meant to feed Epic 8's unified audit trail.

- **[LOW] AC-16's read-path rate-limit default (`{ max: 60, timeWindowMs: 60_000 }`) is asserted without the same rigor of citation/verification the write-path is given** (which explicitly tells the developer to "confirm the exact 429 body shape ... before hardcoding a test assertion"). No equivalent instruction is given for the read-path scoping (per-admin vs. per-IP vs. global), despite the story otherwise being unusually careful about this class of ambiguity.

- **[LOW] The Product Surface Contract documents the missing machine-user UI as "a genuine planning gap" but the risk is never escalated as a blocking dependency for the rest of Epic 7.** Task 11 still requires `web#typecheck` to pick up new generated types for a surface with zero consumers — minor busywork flagged by the story itself as unresolved, without addressing what happens if this planning gap persists past 7.2/7.3 (an admin-only feature permanently operated via raw curl calls).
