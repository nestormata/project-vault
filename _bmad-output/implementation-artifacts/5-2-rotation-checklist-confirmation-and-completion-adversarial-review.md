# Adversarial Review — 5-2-rotation-checklist-confirmation-and-completion

**Date:** 2026-07-01
**Reviewed file:** `_bmad-output/implementation-artifacts/5-2-rotation-checklist-confirmation-and-completion.md`
**Reviewer:** bmad-review-adversarial-general

## Findings

1. **[critical] Rate-limit key is ambiguous and may enable cross-tenant denial of service.** AC-23 specifies the four mutation endpoints use `{ max: 60, timeWindowMs: 60_000, key: '<METHOD PATH>' }` without stating whether the bucket is scoped per-org, per-user, or global across the whole instance. If the key is literally just method+path (as written), a single noisy or malicious tenant can exhaust the shared 60 req/min budget for `confirm`/`fail`/`retry`/`complete`, locking every other org out of rotation-checklist actions. Given the also_consider's explicit focus on tenant isolation, this ambiguity needs to be resolved (e.g., key must include `orgId`) before implementation, not left implicit.

2. **[high] `horizonDays: 0` is not equivalent to `status === 'overdue'`, contaminating the org dashboard's "overdue" bucket.** AC-15 states that calling `computeUpcomingRotations(tx, { horizonDays: 0 })` for the org dashboard "means only include `status: 'overdue'` items." But AC-14's inclusion rule is `nextDueAt <= now + horizonDays`, while its status label is `'overdue'` only when `nextDueAt < now` (strict inequality). A credential whose `nextDueAt` lands exactly at `now` is included by the `horizonDays: 0` filter but labeled `'pending'` by the same function — so the org dashboard's overdue list can silently include a non-overdue item unless the implementation adds an explicit `status === 'overdue'` filter that the spec never actually states as a requirement.

3. **[high] `fail` and `retry` have no idempotent-retry guidance, unlike `confirm`.** AC-3 explicitly instructs clients to treat a `409 already_confirmed` as a successful outcome when re-sending a `confirm` call whose response was lost. No equivalent guidance exists for `fail` (AC-5's `409 invalid_item_status`) or `retry`. A client that loses the response to a `fail`/`retry` call after the transaction has committed (e.g., a network blip) cannot distinguish "my own request actually landed" from "someone else raced me and changed the state" — a real gap for endpoints this story explicitly designs around retry-after-409 semantics elsewhere.

4. **[high] FR65's "upcoming rotation" reference point ignores non-completed rotation history, and this will resurface when Story 5.3 lands.** The reference point for cron computation is "the `completedAt` of the credential's most recent `completed` rotation, or `credentials.createdAt` if none." A credential with one or more `abandoned`/`stale_recovery` rotations (vocabulary this story's own AC-3 already references) that never reached `completed` falls all the way back to `createdAt`, producing a badly stale "next due" date for a credential that in fact has real, recent rotation activity. This is a correctness gap baked into 5.2 today that 5.3 will expose and require reworking.

5. **[medium] Audit fail-closed rollback test coverage is uneven across the five new events.** AC-21 mandates an audit-write-failure rollback test for only `confirm` and `complete` ("at least"), leaving `fail`, `retry`, and `max_retries_exceeded` without a required test proving the fail-closed guarantee actually holds for them — despite the story asserting all five use the identical `writeHumanAuditEntryOrFailClosed` pattern. An untested code path in a fail-closed security control is a meaningful gap, not a nice-to-have.

6. **[medium] `rotation.completed` audit payload omits the retired version IDs.** The payload is `{ credentialId, projectId, checklistItemCount, confirmedCount }` — it does not include `previousVersionId`/`newVersionId`. An incident-review pass over the audit log cannot determine which specific `credential_versions` row was retired by a given completion event without a separate join against `rotations`, which undercuts the "evidentiary record" rationale the story itself invokes elsewhere (AC-3) for why confirm data must never be silently overwritten.

7. **[medium] Rotation-scoped (not item-scoped) advisory lock serializes unrelated, legitimate concurrent work.** ADR-5.2-01/AC-8 make every mutation on any item of a rotation contend for one lock. For the realistic operational case this feature targets — many dependent systems being checked off in parallel by different people during an active rotation — this converts ordinary non-conflicting concurrent activity into a stream of `409`s and client-side retries, not just genuine conflicts. The story documents this as deliberate and tested, but never weighs the operational cost against a per-item locking scheme that would only reject genuine conflicts.

8. **[medium] No error handling specified for a malformed/unparseable `rotationSchedule` in the FR65 computation.** AC-14's cron computation (`CronExpressionParser.parse(rotationSchedule, ...)`) has no documented behavior if parsing throws (e.g., stored data drift, or divergence between write-time validation and read-time parsing). As specified, one bad cron string among many credentials in an org could throw during dashboard aggregation and take down the entire `/api/v1/dashboard` response rather than being skipped and logged.

9. **[medium] Org-dashboard "upcoming rotations" computation is unbounded at the query level.** `computeUpcomingRotations(tx, { horizonDays: 0 })` for the org dashboard scans every credential with a schedule across every project the org can see, computing a cron parse per credential in application code, before truncating to 20 results after the fact. For a large org this is an uncapped per-request computation with no discussed caching, batching, or query-level `LIMIT`, unlike a genuinely paginated list endpoint.

10. **[medium] No secondary verification on `confirm` despite it gating an irreversible action.** A single `member`-role user can self-attest that a dependent system was updated, with nothing but an optional free-text note as evidence. Once all items reach that self-attested state, an `admin`/`owner` can trigger `complete`, which permanently retires the old credential version's protection. Nothing in the AC set contemplates a false or mistaken confirm — the audit trail provides after-the-fact accountability, but there is no in-flow check (e.g., different confirmer than initiator, or any automated verification) protecting FR21/FR22's actual safety goal from a single bad self-attestation.

11. **[low] Advisory-lock key is built via string concatenation, copied "verbatim" from an as-yet-unbuilt pattern.** `'rotation:' || orgId || ':' || rotationId` is validated as UUIDs beforehand per AC-17, so injection risk is low, but the story doesn't explicitly call out that the raw-SQL lock-key expression must use parameterized binding rather than literal interpolation — worth stating explicitly since this exact pattern is being copy-pasted a second time in this story and will likely be copied a third time in 5.3.

12. **[low] The `rotation_checklist_items_pending_total` gauge has no stated refresh interval or query-cost bound.** It's described only as "periodic-query-backed, same pattern as 5.1 AC-18," with no interval specified and no index called out for the `status IN (...)` × `rotations.status = 'in_progress'` join it implies — despite AC-1 explicitly reasoning that no new indexes are needed based solely on this story's transactional queries, a reasoning pass that never considers the gauge's own query shape.

13. **[low] `retryScheduledAt` accepts past timestamps with no validation.** `z.iso.datetime().nullable().optional()` has no `.min()`/future-date constraint, so a caller can persist a `retryScheduledAt` already in the past. Since the field is explicitly operator-visibility-only (AC-4), this produces a confusing, meaningless signal ("retry scheduled" for a time already gone) with no server-side guard.

14. **[low] Inconsistent metric label cardinality.** `rotation_checklist_failures_total` carries no `outcome` label while the other three new counters (`confirmations_total`, `retries_total`, `completions_total`) all do, despite the story elsewhere treating consistency with existing patterns as a design principle. This will likely force a breaking metric/label change if a second `fail` outcome is ever introduced.

15. **[low] FR66's literal route shape is permanently left unresolved against epics.md, with no cross-reference for future traceability audits.** AC-13 deliberately does not implement the flat `GET /api/v1/rotations/:rotationId` path epics.md's AC text literally specifies, resolving the conflict via analogy to 5.1's own precedent. Any conformance/traceability tooling that checks epic ACs against the literal implemented route surface (rather than reading this story's prose resolution) will report FR66 as unimplemented.

16. **[low] No de-duplication or roll-up on `max_retries_exceeded` critical alerts.** Every item that escalates fires its own individually-paged `critical`-severity alert with no rotation-level aggregation. A rotation with many stalled dependent systems (a plausible real-world scenario — e.g., an org-wide credential used by a dozen systems) produces a proportional flood of individually-paged critical alerts rather than a single "N items in rotation X are stuck" summary alert.
