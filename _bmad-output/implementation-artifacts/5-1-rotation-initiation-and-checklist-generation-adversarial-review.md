# Adversarial Review — Story 5.1: Rotation Initiation & Checklist Generation

- **Date:** 2026-07-01
- **Reviewed file:** `_bmad-output/implementation-artifacts/5-1-rotation-initiation-and-checklist-generation.md`
- **Reviewer:** bmad-review-adversarial-general

## Findings

1. **`critical`** — AC-5's "backstop" path is very likely broken as specified. PostgreSQL aborts the *entire* transaction after any statement error (including a unique-constraint violation) until a `ROLLBACK` or `SAVEPOINT`/`ROLLBACK TO SAVEPOINT` is issued. AC-5 requires: "The service layer catches this via `isUniqueViolation()` ... and converts it to the same `409 rotation_in_progress` response (looking up the winning rotation's id for the payload)." That lookup is a `SELECT` run inside the *same* `ctx.tx` transaction after the `INSERT` failed. Without an explicit `SAVEPOINT` wrapping the risky `INSERT`, the follow-up `SELECT` will itself fail with "current transaction is aborted, commands ignored until end of transaction block." The existing codebase precedent for `isUniqueViolation()` (`apps/api/src/modules/credentials/import-service.ts:266-282`, `apps/api/src/modules/credentials/service.ts:362`) never performs a follow-up query after catching the violation — it always rethrows and lets the whole transaction abort. This story is the first to need a "catch, then keep using the same transaction" pattern and never mentions `SAVEPOINT` anywhere in Dev Notes/Key Code Patterns. This risks AC-5(b)'s own required test failing with an opaque "transaction aborted" error instead of the specified `409`.

2. **`high`** — No FK/linkage exists between `rotations` and the `credential_versions` rows it creates/locks. Step 4 inserts a new `credential_versions` row and step 5 locks the superseded one, but `rotations` has no `newVersionId`/`previousVersionId` column recording which version rows correspond to which rotation. Given FR23 requires "rotation history is permanently recorded" and Story 5.2/5.3 will eventually need to clear `rotation_locked_at` on completion/abandonment, they must rediscover the correct version by inference (`credential_id` + `rotation_locked_at IS NOT NULL`) instead of a direct join — fragile if the "one locked version at a time" invariant is ever violated by a bug. Since AC-1/AC-2 explicitly claim to finalize the *whole* Epic 5 schema up front specifically to avoid a second migration, this is a real gap that likely forces exactly the follow-up migration the story set out to avoid.

3. **`high`** — Internal terminology contradiction: the schema comment in AC-1 states rotation rows are "Immutable history once completed/abandoned... no route... ever DELETEs a rotation row," implying immutability. But the same document's "Epic Cross-Story Context" table and AC-3's migration notes state Story 5.2 will `UPDATE status`, `version`, `completedAt` on those same rows, and AC-3 explicitly justifies adding `updated_at` triggers to both tables "because neither table has an append-only invariant" — directly contradicting "immutable." The story conflates "never hard-deleted" with "immutable," which will confuse whoever implements or reviews 5.2's mutations against this schema.

4. **`medium`** — Rotation/checklist history's permanence claim is undermined by inherited FK cascade semantics. `rotations.orgId`/`.projectId`/`.credentialId` are all `onDelete: 'cascade'` (following the repo-wide `orgScoped({onDelete:'cascade'})` convention also used by `credentials`/`projects`). No hard-delete route exists yet for orgs/projects/credentials in this branch, so the risk is latent — but if one ever ships, the entire rotation history including checklist items disappears along with the parent row, contradicting this story's repeated claims that rotation history is "immutable," "permanent," and the eventual audit-UI (Epic 8) source of truth. The story never discusses or scopes this tension.

5. **`medium`** — No domain separation for the two-key advisory lock's hash inputs. `pg_try_advisory_xact_lock(hashtext(credentialId), hashtext(orgId))` shares Postgres's single global 64-bit advisory-lock keyspace with other unrelated consumers already in the codebase (`mfa-login.ts` line 129, `check-failed-auth-threshold.ts` line 229), which hash different domain values into the same two-int4 key format. Nothing in this story (or, apparently, the precedents it cites) folds in a per-feature discriminant constant, so a coincidental `hashtext` collision across features (e.g., a credentialId hashing the same as an unrelated userId, with the second key also colliding) would cause cross-feature false lock contention. Low probability, but entirely unacknowledged.

6. **`medium`** — `hashtext()` collisions *within* the rotation domain are also unaddressed: two different credentials in the same org whose `hashtext(credentialId)` values happen to collide would make one credential's rotation attempt spuriously fail the advisory lock whenever the other's rotation is in flight — an incorrect `409` for two genuinely unrelated credentials. Critically, the partial unique index (AC-1) does **not** backstop this failure mode, since the false lock contention short-circuits before the `INSERT` is ever attempted — undercutting the story's own claim that the unique index is "the actual, durable, connection-independent invariant."

7. **`medium`** — AC-10's validation-error table exercises only the `newValue` field. There is no test case or AC coverage for a `notes` value exceeding the schema's own `.max(1024)` constraint, despite `notes` being part of the same `.strict()` body schema shown in AC-1's prerequisites. Gap between the schema as specified and the acceptance criteria that verify it.

8. **`medium`** — No AC/test covers malformed (non-UUID) path parameters (`projectId`, `credentialId`, `rotationId`) for any of the three routes. `RotationParamsSchema` is named in Task 3 but its validation behavior and resulting error code (400? 422? folded into the generic 404?) is never specified in the Acceptance Criteria, unlike the request-body validation table in AC-10.

9. **`medium`** — AC-14's audit-failure response contract is explicitly hedged rather than pinned down: "The client receives `503 { code: "audit_write_failed", ... }` (or whatever the existing SecureRoute error-mapping produces for this error type — verify against `secure-route.ts`'s catch handling and match it exactly, do not invent a new shape)." An acceptance criterion that tells the implementer to go discover the actual contract themselves, rather than stating it, is a real gap — especially inconsistent with how precisely this story pins down every other response shape down to the byte.

10. **`medium`** — Sealing the vault blocks the two read-only rotation endpoints (`GET .../rotations`, `GET .../rotations/:id`) even though AC-16 itself argues these routes "do not expose credential values" (the justification used for the audit-omission classification). Blocking read access to rotation status/history while sealed removes exactly the visibility an operator would want during an incident (e.g., diagnosing a stuck `in_progress` rotation while the vault happens to be sealed). The story doesn't examine this trade-off, it just asserts "mirrors 2.2/2.4" without asking whether pure-metadata reads should be exempted.

11. **`medium`** — No validation, warning, or even a log line exists for the case where the submitted `newValue` is byte-identical to the value already stored in the current version. The entire premise of this endpoint ("a developer who has updated a credential in its target systems") is that the value actually changed externally; silently accepting a no-op "rotation" produces a checklist and permanent audit trail implying a real rotation happened when it may not have — undermining the audit trail's evidentiary value (FR23) with no detection at all.

12. **`medium`** — The story asserts "MFA enforcement for admin/owner roles is already handled globally by the auth middleware... No route-level `requireMfa` option is needed here" (Dev Notes, Architecture Compliance), but no AC or test in this story verifies that assumption for the three new routes. Rotation initiation writes a new live secret value and is one of the most security-sensitive operations in the system — taking MFA enforcement on faith without a single verifying test is a real audit/security-testing gap, especially given the also_consider's explicit call-out of authn/authz risks.

13. **`low`** — `rotation_checklist_items` has no explicit ordering column, and neither the AC-4 response example nor AC-11's detail response specifies an `ORDER BY` for the `checklistItems` array. Postgres does not guarantee row order without one; the checklist order returned to a client could vary between calls unless the service layer is careful to always sort consistently (e.g., by `createdAt`), and nothing in the story mandates this.

14. **`low`** — `InitiateRotationBodySchema`'s `notes` field uses `.trim()` but not `.min(1)` after trimming, so a whitespace-only string (e.g., `" "`) is silently normalized to an empty string and stored as `notes: ""` rather than `null` — a minor data-quality inconsistency between two different "no notes" representations reaching the DB depending on what the client actually sent.

15. **`low`** — AC-13/AC-19 together document that `rotation_locked_at` is set but never cleared until Story 5.2 ships, meaning any rotation left `in_progress` permanently exempts a version from retention purge. There's no metric or log surfaced anywhere in AC-18 for "count of credential versions currently locked by an in-progress rotation" — useful operational visibility the story otherwise invests effort into for every other outcome (success/conflict/validation/audit-failure counters) but omits for this specific, self-acknowledged gap.

16. **`low`** — No guardrail or test exists for a credential with an unusually large number of non-archived dependencies. AC-4 step 6/8 reads and inserts one checklist row per dependency with no documented ceiling (Story 2.4 doesn't impose one either), so a pathological dependency count would extend the advisory-lock-held transaction's size and duration with no stated bound or test coverage for that scale scenario.
