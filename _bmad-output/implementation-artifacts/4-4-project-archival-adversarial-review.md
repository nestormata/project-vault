# Adversarial Review: Story 4.4 — Project Archival

- **Date:** 2026-07-01
- **Reviewed file:** `_bmad-output/implementation-artifacts/4-4-project-archival.md`
- **Reviewer:** bmad-review-adversarial-general

## Findings

- **[critical] Invitation-acceptance bypasses the "no new members after archive" guarantee.** AC-5's guard table lists `POST /api/v1/projects/:projectId/invitations` (creating a new invite) and `POST /api/v1/projects/:projectId/members` (direct add) as routes that must be guarded, but there is no mention of the invitation-**acceptance** endpoint. A user invited before a project is archived can still accept that invitation after archival and become a full member, silently defeating the AC-5 promise that archived projects "must not receive new credentials, members, invitations, or metadata edits." This is a real, unaddressed path around the guard the story spends an entire AC establishing.

- **[high] AC-8 contradicts its own guidance on the 409 response shape.** The prose in AC-8 explicitly warns: "Do not force the active-rotations body into `ApiErrorSchema` — that would change the wire shape away from the 4.3-compatible contract" and recommends a union type or omitting 409 from the typed response map. Immediately after, the example code declares `schema: { response: { 200: ArchiveResponseSchema, 404: ApiErrorSchema, 409: ApiErrorSchema } }` — exactly the thing just prohibited. Since the same archive route can return **two different 409 bodies** (`{code,message}` for `already_archived` and `{error,rotationIds}` for `active_rotations`), declaring a single `ApiErrorSchema` for 409 risks Fastify/AJV response serialization silently stripping the `error`/`rotationIds` fields it doesn't recognize — breaking the exact byte-compatibility with Story 4.3 that ADR-4.4-04 mandates. The Dev Notes section's SecureRoute example then omits 409 entirely, a third inconsistent variant of the same route's schema.

- **[medium] Authorization check runs after a state-disclosing check, enabling non-member enumeration.** In both AC-2 and AC-6, the handler checks idempotency ("already archived" → 409 / "not archived" → 409) in step 3, *before* the project-owner check in step 4. Since the RLS-scoped project load (step 2) is described as isolating by org, not by project membership, any authenticated member of the org — not just members or owners of the target project — can call archive/unarchive on an arbitrary project id in-org and learn its archival state by distinguishing 409 (already in that state) from 403 (not owner, state unknown) — an access-control ordering / minor information-disclosure gap that a stricter design (authz before business-state checks) would avoid.

- **[medium] TOCTOU window between the active-rotation guard and the archive commit.** The active-rotation guard (AC-4, handler step 5) runs before the atomic `UPDATE ... WHERE archived_at IS NULL` (step 7), but nothing in the story requires locking the candidate rotation rows (e.g., `SELECT ... FOR UPDATE`) or otherwise serializing against concurrent rotation creation. A new `in_progress` rotation could be created by a separate transaction in the gap between the guard check and the archive commit, producing exactly the orphaned-rotation outcome the guard exists to prevent.

- **[medium] Archived slugs permanently block reuse, with no product or UX treatment.** AC-6 notes "archived rows still occupy `idx_projects_org_slug`" as a reason no slug re-check is needed on unarchive — but this also means a user who archives "legacy-billing" and later tries to create a *new* project named "legacy-billing" will hit an opaque slug-uniqueness failure. This consequence is never surfaced in AC-11 (Web App UI requirements) or AC-13 (Out of Scope), leaving a confusing dead end unaddressed by the spec.

- **[medium] No audit trail for denied/blocked archive attempts.** Per the Dev Notes "Return data, do not `reply.send`" rule, the SecureRoute same-tx audit writer only fires on the success path. All rejected attempts (403 non-owner, 409 active-rotations, 409 already-archived/not-archived) are therefore never written to the audit log. For an action the story itself calls "high-impact" and gates behind MFA, the absence of any audit record for repeated unauthorized or blocked archive/unarchive attempts (a plausible privilege-escalation or reconnaissance signal) is a logging/observability gap.

- **[medium] Static rate-limit key raises a DoS/availability question the story never resolves.** Both routes set `rateLimit: { max: 10, timeWindowMs: 60_000, key: 'POST /api/v1/projects/:projectId/archive' }` with a literal, non-parameterized key string. The story never clarifies whether the SecureRoute rate limiter combines this key with caller/org identity or uses it as a single global bucket. If the latter, 10 archive operations per minute for the *entire deployment* across all tenants is a severe, self-inflicted availability ceiling at any real scale.

- **[medium] AC-5's "read-only" guarantee is scoped only to creation, not mutation/deletion of existing resources.** The guard table covers `POST` (create credential, invite, add member, transfer ownership) and `PATCH` project metadata, but says nothing about editing, rotating, or deleting an *existing* credential inside an archived project. Whether such operations should also be blocked is left completely unspecified — a real gap in what "archived projects are read-only" is supposed to mean.

- **[low] Org-owner override is unreviewed as a security decision.** The ownership check silently grants any org `owner` archive/unarchive authority over a project even when that user holds no `project_memberships` row at all. This is a meaningful expansion of authority beyond the epic's stated "project owner" concept, yet it has no dedicated ADR discussing the tradeoff, and the audit event for `project.archived`/`project.unarchived` does not distinguish "acted via project ownership" from "acted via org-owner override" — a missed opportunity for audit clarity on a privileged bypass path.

- **[low] Guard-removal after Epic 5 relies entirely on human process, not a technical control.** ADR-4.4-02 and the pre-mortem both name "the seam never gets replaced" as an anticipated failure mode, but the only mitigations offered are a `deferred-work.md` entry, a QA sign-off hold, and an `rg "rotationsTableExists"` grep — all manual. There is no automated CI check (e.g., a test that fails once the `rotations` table exists but the seam code is still present) enforcing removal, so the very risk the story flags remains only softly mitigated.

- **[low] Notifications/background-job interaction with archived projects is unaddressed.** Dashboard fields like `alertCount`/`expiringCount` and any rotation-reminder or credential-expiry notification jobs are never discussed with respect to archived projects. It's unclear whether these continue to fire for a project a user has deliberately hidden from their active view, risking noisy/confusing alerts for archived work.

- **[low] Dead reasoning in a code comment.** The archive-update fallback comment reads "Either gone (concurrent delete) or already archived by a racing request" — but AC-13 explicitly states there is no delete endpoint in v1, making "concurrent delete" an impossible branch and suggesting either stale copy-paste reasoning or an unstated future concern that should be called out explicitly instead of left as a misleading comment.

- **[low] Underspecified test contract for the MFA-not-enrolled case.** AC-10 lists "403/precondition when MFA is required but not enrolled (requireMfa: true)" as a required scenario, but unlike every other listed test case, it doesn't commit to an actual status code or response body shape ("403/precondition" is not a real contract), leaving the expected behavior to implementer discretion.

- **[low] `deferred-work.md` existence is assumed, not verified.** The story repeatedly instructs the dev to append entries to `_bmad-output/implementation-artifacts/deferred-work.md` (AC-5, AC-13, Task 11, multiple ADRs) but — unlike other dependencies such as `route-exemptions.ts` or `meta/_journal.json`, which the Prerequisites table explicitly requires confirming — never verifies this file currently exists or defines its expected format, risking ad hoc, inconsistent entries across stories.

- **[low] Hardcoded migration-tip fact will likely be stale by the time this story is picked up.** The Prerequisites table states "the active migration tip on this branch is `0012_refresh_tokens_org_id.sql`" as current fact, in a story the document's own "Cross-Epic Sequencing" section recommends deferring until after Story 5.1 ships. By the time 4.4 is implemented, several other stories' migrations will likely have advanced this number, and while the text does caveat "verify, do NOT hardcode," embedding a concrete number that is expected to go stale invites a developer to trust it without re-checking.
