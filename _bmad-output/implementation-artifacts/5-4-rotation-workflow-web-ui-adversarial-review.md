# Adversarial Review: Story 5.4 — Rotation Workflow Web UI

- **Date:** 2026-07-05
- **Reviewed file:** `_bmad-output/implementation-artifacts/5-4-rotation-workflow-web-ui.md`
- **Reviewer:** bmad-review-adversarial-general
- **Also-consider scope:** inconsistencies/contradictions between sections; gaps in acceptance criteria or missing edge cases; technical debt implied by the proposed approach; implementation/delivery risks; security risks (authn/authz, RLS/tenant isolation, secrets handling, injection, input validation, audit/logging gaps).

## Findings

- **[critical] AC-24's central claim — that the four read endpoints (`GET .../rotations`, `GET .../rotations/:id`, `GET .../dependencies`, `GET .../rotations/upcoming`) "are not vault-guarded... so reads still succeed while sealed" — is factually false per direct inspection of `apps/api/src/plugins/vault-guard.ts`.** The `VAULT_GUARD_ALLOWLIST` there contains exactly 7 entries: `GET /health`, `GET /ready`, `POST /api/v1/vault/init`, `POST /api/v1/vault/unseal`, `POST /api/v1/auth/register`, `POST /api/v1/auth/login`, `POST /api/v1/auth/refresh`. None of the rotation/dependency read routes are in this list, and `apps/api/src/main.ts` sets `vaultGuardEnabled: true` unconditionally in production, so the guard's `onRequest` hook (which fails closed with `503 { status: "sealed" }` for anything not in the allowlist) applies to *every* route this story reads from. This means the initiate page's dependency preview (AC-3), the rotation detail page's initial load (AC-7), the history section (AC-18), and the dashboard widget (AC-23) all need sealed-vault handling in their `PageServerLoad` functions — directly contradicting AC-24's explicit scoping of the 503 handling to mutations only. This under-scopes the story's actual error-handling surface.

- **[critical] `403 { code: 'mfa_required' }` is never mentioned anywhere in the story despite being a real, tested error on 5 of the 11 endpoints this story wraps.** Direct inspection of `apps/api/src/modules/rotation/routes.ts` shows `security: { minimumRole: 'admin', requireMfa: true, ... }` on initiate (line 421), break-glass (line 544), complete (line 1111), resume (line 1238), and abandon (line 1296); `apps/api/src/modules/auth/mfa-enforcement.ts`'s `requireMfaEnrollment()` returns exactly `403 { code: 'mfa_required', message: MFA_REQUIRED_MESSAGE }` when an admin/owner hasn't enrolled in MFA (past their grace period). A working UI precedent for this exact error already exists in `apps/web/src/routes/(app)/projects/[projectId]/members/+page.svelte` (`error.code === 'mfa_required'`), yet the story's "Ground-Truth API Surface" table, "Key Code Patterns to Follow" section, and AC-25's "error mapping contract" never cite it, and no AC handles it. This directly contradicts the AC Quick Reference's "Errors" row promise that "every documented error code above is mapped to a specific, non-generic on-screen message."

- **[high] AC-8 cites a nonexistent precedent as justification for its `confirmedBy` display-name design: "matches existing `RecentAccessEvent`-style `actorDisplayName` pattern elsewhere."** A repo-wide search of `apps/web/src` finds zero occurrences of either `RecentAccessEvent` or `actorDisplayName`. `recentAccessEvents` is in fact an unimplemented, empty-array placeholder per `deferred-work.md`'s own "Project dashboard" table row, not a working component with a name-resolution pattern. A real mechanism to resolve a user id to a display name does exist (`displayName` field via `apps/web/src/lib/api/org-users.ts`), but the story never references it. This is exactly the class of claim the story repeatedly boasts about being "confirmed via direct inspection," and it is not accurate.

- **[high] 429 rate-limiting is completely unaddressed anywhere in the story.** Every rotation route enforces a rate limit via `secureRoute`'s `rateLimit` option (default 60/min when unset per `apps/api/src/lib/secure-route.ts`; break-glass has its own tighter, explicitly-commented limit — "at most a handful of break-glass calls per minute" at routes.ts line ~520). No AC, no Ground-Truth API Surface "Key error codes" column entry, and no error-mapping section mentions a 429 response. This is a real delivery gap, and it's especially risky for break-glass (AC-20/AC-21) — the one flow the story itself frames as needing to "act in seconds" during an incident — where a fumbled `CONFIRM` retry could plausibly trip a low per-minute cap with zero specified UI message.

- **[high] Secrets-handling gap on the break-glass new-value field.** AC-4 specifies "a monospace, non-autocomplete field matching the reveal-value styling precedent" for the normal-path new-value textarea, but AC-20 (the break-glass panel, the higher-blast-radius and arguably higher-risk path) never repeats or cross-references this same input-hardening requirement for its own new-value field. Separately, neither AC mentions clearing the plaintext value from client-side `$state` after a successful/failed submission, or on component teardown/navigation-away — a live credential value can linger in memory/DOM longer than necessary in a form that is, by construction (textarea has no `type="password"` equivalent), rendered in cleartext on screen with no obscure/reveal toggle discussed, which is a real concern for a workflow explicitly designed to be used mid-incident (shoulder-surfing/screen-share exposure risk).

- **[medium] Internal cross-reference error in the persona-journey stub.** Line 57 attributes "Morgan is member and cannot complete — role-gated" to "AC-16/AC-19," but AC-16 is the Stale-Recovery Banner/Resume AC and AC-19 is the Break-Glass entry-point AC — neither addresses the Complete button's role gating. The actual Complete-button AC is AC-11, which only describes the admin/owner-visible path (disabled while items are pending); AC-14 only covers the `viewer` role's full read-only view. **No AC in the document actually specifies what a `member` sees for the Complete button specifically** (omitted, per the D3 "don't render what you can't do" convention used elsewhere, or something else) — this is a genuine, unfilled acceptance-criteria gap, not just a typo, since the behavior it's supposed to describe is never pinned down anywhere else in the document.

- **[medium] `initiatedBy` is nullable in the actual schema, and the story's own citations acknowledge this as a known live-data condition, yet no AC handles it.** `packages/shared/src/schemas/rotations.ts` declares `initiatedBy: z.uuid().nullable()` on both `RotationDetailSchema` and `RotationSummarySchema`. "NULL `initiatedBy`" is explicitly one of the enumerated findings from the Epic 5 retro that the story's own "Epic Cross-Story Context" table cites as bundled into Story 5.5. AC-7 ("initiated-by/at") and AC-18 (history rows, "initiatedAt") both describe rendering this field with no fallback specified for the null case — a data state the story's own references confirm is currently possible.

- **[medium] Numeric inconsistency between the story's citations for Story 5.5's finding count.** The story's "Epic Cross-Story Context" table (line 72) states Story 5.5 bundles "13 backend hardening findings," matching `sprint-status.yaml`'s comment ("13 findings"), but `deferred-work.md`'s own enumerated list under the same citation (the row this story implicitly relies on for its "5.5 has zero overlap" claim) names exactly 12 distinct findings. The story does not reconcile this discrepancy despite treating both source documents as ground truth elsewhere.

- **[medium] Rendering of admin-controlled free text without any escaping/length-display guidance.** `lastFailureReason` (AC-9), `notes` (AC-4/AC-7), and dependency `systemName` (AC-3) are all free-text fields sourced from user input (the code comment at routes.ts line 206 explicitly calls `reason` "admin-controlled free text"). The story never states these are rendered as plain text nodes (relying implicitly on Svelte's default escaping) rather than via `{@html}` or similar — an easy thing for an implementer to get wrong under time pressure for a story this large, and worth an explicit AC/test assertion given the "also consider... injection" review scope, even though Svelte's defaults make actual XSS unlikely here.

- **[medium] Overconfident blanket-verification framing invites under-scrutiny during implementation and review.** Line 34 states: "Where epic/PRD language and actual shipped code differ (there are no such conflicts found for this story — 5.1/5.2/5.3 already resolved all of them)..." This review directly falsifies that claim (the vault-guard and MFA gaps above are both real, verifiable conflicts between the story's prose and the shipped code it claims to have inspected). A story that repeatedly asserts "confirmed via direct inspection, not aspirational" creates a false sense of security for whoever picks it up next (dev, code reviewer, QA) and may cause them to skip independent verification of exactly the claims that turned out to be wrong.

- **[low] AC-9's retry-counter design (correctly, deliberately) omits the max-retries denominator until a 422 is hit, but this leaves the user with zero forward warning before hitting the cap.** This is a disclosed, intentional design decision (the story is explicit that it will not hardcode the max), so it is not a defect, but it is worth flagging as a residual UX rough edge: an operator retrying a stuck checklist item during an active rotation gets no indication they are one attempt away from `max_retries_exceeded` until the error actually fires.

- **[low] The "Explicit Out of Scope" list and the "Anti-Patterns" list both individually repeat several of the same constraints (no Playwright, no new HTTP client, D1/D2 restatement) verbatim in two separate places in the document** — not wrong, but a maintenance/consistency risk: a future edit to one list (e.g., if Playwright is eventually added repo-wide) is likely to update only one of the two locations, leaving a stale contradiction in the other.

- **[low] The story assumes `GET .../rotations?limit=1` ordering ("most-recent-first per 5.1") is sufficient to determine "the" active rotation for AC-2's redirect logic, but does not address what happens if two rotations are simultaneously in a non-terminal state** (e.g., a data anomaly, or a future state-machine change) — the story treats "first item of a `limit=1` query" as unconditionally authoritative for "the" active rotation without an AC covering what the page does if the assumption (at most one non-terminal rotation exists per credential) is ever violated, even transiently.

- **[low] AC-1's "brand-new credential, zero rotations ever" edge case and AC-18's "zero rotations ever" edge case both specify the identical string "No rotations yet." and note it should be "a single source of truth constant in a shared copy file, not duplicated inline," but no task in "Tasks / Subtasks" actually creates this shared copy file/constant** — Task 2 and Task 7 both touch related surfaces but neither explicitly lists creating this shared string constant as a deliverable, so it is likely to end up duplicated inline in practice despite the story's own explicit instruction not to.
