# Adversarial Review — Story 8.7 (Epic 8 Completion: Audit/Compliance Web UI and Technical Debt)

- **Date:** 2026-07-07
- **Reviewed file:** `/home/nestor/Proyects/project-vault/.claude/worktrees/plan+8-7-epic-8-completion-audit-compliance-web-ui-and-technical/_bmad-output/implementation-artifacts/8-7-epic-8-completion-audit-compliance-web-ui-and-technical-debt.md`
- **Reviewer:** bmad-review-adversarial-general

## Findings

- **[critical]** AC-M1's described `piiRetained` shape does not match the schema Story 8.4 already shipped. AC-M1 says: `piiRetained` entries have `table, fields, retentionJustification` (example: "audit_log_entries — actorTokenId reference — audit log integrity (HMAC-protected, append-only)"). But `PiiRetainedEntrySchema` in `apps/api/src/modules/compliance/schema.ts` is actually `{ table: string, reason: string }` — there is no `fields` array on each entry, and the per-row justification field is named `reason`, not `retentionJustification`. The actual `retentionJustification` field in `ErasureReportResponseSchema` is a single top-level string for the whole report, not a per-table value. This directly undercuts the story's own "Background: What Already Exists" promise that "the dev agent does not have to re-read all of 8.1–8.4 to find these facts" — a dev implementing AC-M1 literally will build the wrong table columns for the highest-stakes compliance artifact in the story.

- **[high]** The endpoint inventory table's header asserts "all registered under `/api/v1/org` (confirmed via `apps/api/src/app.ts:243-245`)," but at least two of the table's own rows contradict that claim. `GET /security-alerts` and `POST /security-alerts/:alertId/dismiss` are registered under prefix `/api/v1/security-alerts` (`app.ts:281`), and `PATCH /organizations/:orgId/user-dormancy-settings` is registered under prefix `/api/v1/organizations` (`app.ts:282`) — neither anywhere near the cited lines 243-245. A dev who trusts the table's header and prefixes every listed path with `/api/v1/org` will ship a client wrapper that 404s on the dormancy-threshold and security-alerts calls (AC groups H and I).

- **[high]** AC-G3's access-report CSV download doesn't actually fit either mechanism D3 promises exist. D3 claims exactly two file-download patterns cover this story's needs: a plain `<a href>` GET link relying on server-set `Content-Disposition` (for audit export), or `triggerJsonDownload()` for JSON blobs. But `POST /audit/access-report` with `format: "csv"` is a POST endpoint (so a plain anchor tag can't carry the filter body needed to reproduce the on-screen `asOf`/pagination state), and its handler (`apps/api/src/modules/audit/routes.ts`) never sets `Content-Disposition` — it only sets `Content-Type: text/csv` and streams the body. AC-G3 punts with "confirm the actual response shape/headers at implementation time and follow whichever of D3's two mechanisms actually matches; do not guess" — but neither mechanism matches, silently asking the implementer to invent an undocumented third download pattern that the Anti-Patterns section elsewhere discourages inventing.

- **[medium]** D6's "safe re-POST as a read path" only covers the `pending` status branch of `409 erasure_not_yet_completed`. When the report-status probe returns `status: "in_progress"` (a live execution race, the same scenario AC-L4 handles on the execute side), D6's prescribed re-POST to `POST /users/:userId/erasure-request` hits `createErasureRequest`'s `execution_in_progress` outcome, which returns `409 { code: 'erasure_execution_in_progress', requestId }` with **no** `piiInventory` field (`ErasureExecutionInProgressErrorSchema` has no `piiInventory`, unlike `ErasureAlreadyPendingErrorSchema`). D6 never addresses this branch, leaving the review screen's behavior during an in-flight execution undefined — likely a blank inventory table or an unhandled response shape.

- **[medium]** Discoverability gap for the `admin`+ role on the Forwarding & Retention page. Per D1, the only navigation path to `/settings/audit/forwarding` is via links inside `/settings/audit`'s body (D1: "that page itself links to its two siblings"). But `/settings/audit` is `owner`-only, and AC-B4/AC-N1 specify that non-owner roles (including `admin`) see a page-level "requires the owner role" notice **instead of** the page body — which is where the sibling links live. Since `admin`+ is explicitly the intended audience for `/settings/audit/forwarding` (AC-N1's own example: "admin, `/settings/audit/forwarding`: full read/write access"), an admin has no in-app discovery path to a page they are supposed to be able to use, short of guessing or bookmarking the raw URL. AC-A1/A2 never account for this.

- **[medium]** No AC covers turning forwarding off once configured. `AuditForwardingConfigRequestSchema`'s `type` field only accepts `'webhook' | 's3'` — there is no "disabled"/"none" type and no field to explicitly unset forwarding. AC-E1/AC-E3 only specify "select a type, configure it, submit" flows; nothing in AC group E (or D2's rationale, which is specifically about read-back, not write capability) explains what an admin who wants to stop forwarding is supposed to do, or whether that's even possible via any existing endpoint. This is a distinct gap from D2's documented limitation and isn't covered by it.

- **[medium]** No AC specifies input masking or autocomplete behavior for the secret-bearing fields this story introduces: `secretHeader` (AC-E1), `accessKeyId`/`secretAccessKey` (AC-E3). The story is careful about never *redisplaying* these values after submission, but says nothing about how they're typed in — plain-text visible inputs risk shoulder-surfing, screen-recording exposure, and browser password-manager autofill/capture of live webhook secrets and AWS credentials as they're entered.

- **[medium]** AC-B3's expand-in-place detail panel is explicitly "a pure client-side reveal... no second API call is made." That means inspecting an individual audit event's sensitive detail (actor IP address, resource ID) generates no audit trail entry beyond the single `audit.search_run` event already logged for the original list query. For a story whose entire purpose is compliance/audit tooling, and which is otherwise meticulous about logging every write path, this creates a real audit-logging blind spot: there is no record distinguishing "owner scanned a table of 20 summary rows" from "owner inspected this specific person's IP address in detail," and the story never acknowledges this as a tradeoff.

- **[medium]** `TypedConfirmInput.svelte` (mentioned once, in Dev Notes) is the one genuinely new shared UI component this "reuse everything" story introduces, gating both the pseudonymize (AC group J) and erasure-execute (AC group L) irreversible actions. Unlike `DataTable.svelte`/`ConfirmDeleteButton.svelte`, which get explicit AC-level reuse call-outs, this component has no acceptance-level spec of its own (props, exact match semantics, disabled/enabled state transitions) — its correctness is load-bearing for the story's two highest-stakes destructive actions but is only indirectly exercised via AC-J3/AC-L1's user-facing behavior.

- **[medium]** No AC bounds or times out the export-status polling loop from AC-C1 ("polling `GET /audit/exports/:jobId` (e.g. every 2 seconds)"). If the background export worker is down or an export job never leaves `pending`/`processing`, there's no specified max-attempts cutoff, no user-visible "this is taking longer than expected" state, and no escape hatch — the panel is implied to poll indefinitely with no stated failure mode for a stuck (not failed) job.

- **[low/medium]** No AC validates or even acknowledges the coupling between the two URL segments on `/settings/users/[userId]/erasure/[requestId]`. `GET .../report`'s actual handler (`apps/api/src/modules/compliance/erasure-routes.ts`) parses `userId` from the URL but never uses it — lookup is scoped purely by `requestId`+`orgId`. The story's route design implicitly assumes `userId` and `requestId` are a validated pair (D6's load-function logic depends on `userId` for the re-POST fallback) but never states this assumption or specifies UI behavior if a stale/hand-edited URL supplies a `userId` that doesn't match the `requestId`'s actual target user.

- **[low/medium]** AC-G3's design (JSON view first, then a fully separate CSV request with `format: "csv"` on download) means every CSV download is a second, full `POST /audit/access-report` round trip — including, on the historical `asOf` path, a second full audit-log replay computation (Story 8.3 D2's replay engine, not a lightweight query). The story never discusses the performance cost of duplicating this computation per download, nor considers caching the already-computed result, despite flagging the replay engine elsewhere as a "structurally different" and presumably non-trivial operation.

- **[low]** The story doesn't reconcile its own specified UI request patterns against the backend rate limits it otherwise carefully documents. `GET /audit/verify` is capped at 20/min, `GET /audit/exports/:jobId` at 60/min (AC-C1's 2-second polling alone consumes up to 30/min per open panel — doubled by a second browser tab), and `POST /audit/access-report` at 30/min (also double-counted per finding above for CSV downloads). No AC addresses user-facing behavior when these limits are hit (e.g., a 429 mid-poll), despite the story otherwise treating every other error status as a first-class UX case.

- **[low]** D4/D5's typed-email confirmation is repeatedly framed as adding real protection ("UI-only defense-in-depth... matching the level of care this codebase already applies to its other highest-stakes actions"), but by the story's own description it is a pure client-side string comparison gating a submit button — any direct API caller (or a user with browser dev tools open) bypasses it trivially, since the actual request body it produces (`confirmUserId` auto-populated from the row, or bare `confirm: true`) is unchanged from what an unconfirmed call would send. Calling this "defense-in-depth" overstates its security value; it is misclick protection, not a security control.

- **[low]** Email-matching case sensitivity is unspecified for D4/D5's typed-confirmation gate. The story describes it only as "type the target user's exact email" / "types anything other than the target's exact email" — if implemented as a naive exact-string match (the most literal reading), a legitimate operator who types their own recollection of the address in different letter case than what's actually stored will be permanently stuck with a disabled submit button, with no fallback specified by any AC.

- **[low]** AC-O2's own suggested regression check ("grep the new `.svelte` files for language like 'current configuration' or 'your saved settings'") is described only as a manual verification step, not wired into Task 13's concrete CI checklist (`pnpm turbo typecheck`, `pnpm turbo lint`, `pnpm jscpd`, test suite, `make ci`). As written, this is the one regression check in the entire story with no automated enforcement, making it the easiest AC in the story to silently skip during actual implementation.
