# Adversarial Review: Story 1.14 — Vault KMS Unseal Mode

- **Date:** 2026-07-11
- **Reviewed file:** `_bmad-output/implementation-artifacts/1-14-vault-kms-unseal-mode.md`
- **Reviewer:** bmad-review-adversarial-general
- **Also-consider scope:** inconsistencies/contradictions between sections; gaps in acceptance criteria or missing edge cases; technical debt implied by the proposed approach; implementation/delivery risks; security risks (authn/authz, RLS/tenant isolation, secrets handling, injection, input validation, audit/logging gaps).

## Findings

1. **[high] Unauthenticated, unrate-limited `POST /api/v1/vault/unseal` now triggers billed, quota-limited external AWS API calls.** AC-17 explicitly confirms unseal is gated by "nothing additional" beyond what exists today (a sealed vault has no user session to check). Under `passphrase`/`file`/`envelope` modes, an unauthenticated flood of unseal attempts only costs local CPU (Argon2id) or disk reads. Under `kms` mode, every attempt is a live, metered `Decrypt` call to AWS KMS. The story never proposes a rate limit, backoff, or cost/quota consideration for this new externally-billed, quota-bound attack surface, despite otherwise being extremely detailed about every other failure mode.

2. **[high] AC-6's "derive first, then attempt insert" ordering means every repeated `POST /api/v1/vault/init` against an already-initialized vault performs a real, wasted `GenerateDataKey` call.** This is called out as intentional parity with existing modes, but for `kms` mode specifically it converts a race-guard no-op into a paid AWS API call and consumes part of the per-key `GenerateDataKey` TPS quota. Combined with finding 1, repeated init/unseal probing (even under the bootstrap-token-gated init path, and unconditionally on the ungated unseal path) is a real availability/cost risk this story does not acknowledge or mitigate.

3. **[high] AC-23's core test-methodology decision is explicitly unresolved, yet the story's status is "ready-for-dev."** Open Question 3 states the LocalStack-vs-mocked-KMSClient choice "was not fully confirmed during story creation" and defers it to "whatever the implementing developer finds." An acceptance criterion whose fundamental test strategy is admitted to be unverified is not actually implementable as written — this is exactly the kind of unresolved ambiguity a story is supposed to close before being marked ready, not push downstream.

4. **[high] No AWS `ThrottlingException`/`LimitExceededException` error category is defined anywhere in the AC set.** AC-3/AC-4/AC-5/AC-11/AC-12/AC-13 enumerate unreachable/not-found/permission-denied/key-unavailable, but AWS KMS's `GenerateDataKey` and `Decrypt` both have per-key/per-region TPS quotas that produce a distinct throttling exception under load — made more likely by findings 1 and 2. Without a defined mapping, this exception either falls through to a generic 500 or gets misclassified as "unreachable," misleading operators during an actual incident (the exact failure mode AC-3's own negative example warns against for a *different* pair of errors).

5. **[medium] Task 9's own AC coverage claim is internally inconsistent.** The task reads: "...covering AC-1 through AC-19 (happy paths, all error classes, concurrency, backward compat) (AC-1–AC-20, AC-23)." The prose range ("AC-1 through AC-19") and the parenthetical AC tag list ("AC-1–AC-20, AC-23") disagree on whether AC-20 (backward compatibility) is in scope for this task — a direct contradiction within a single bullet.

6. **[medium] Migration file is hardcoded to a specific next sequence number (`00XX` after `0047`) in a project that visibly runs multiple concurrent story worktrees.** This environment itself demonstrates the parallel-worktree workflow (this very review runs in `.claude/worktrees/plan/1-14-...`). If another in-flight story also claims migration `0048` first, this story's migration will collide or need renumbering at merge time — a delivery risk the story doesn't flag despite flagging nearly every other risk in exhaustive detail.

7. **[medium] No AC covers the data-integrity edge case of `kms_type = 'kms'` with a `NULL kms_encrypted_dek`.** AC-7's own negative example explicitly *defers* the DB-level CHECK constraint that would prevent this state, reasoning that "the row is inserted only after AWS KMS already succeeded... so the invariant holds in practice." But this only holds for the application code path introduced by this story — a future migration bug, manual DB edit, or admin script could still produce this state, and no AC specifies what `unsealVault()` should do when it reads `kms_type='kms'` with a null ciphertext blob (crash, generic error, or a clean, actionable error). This is a real gap the story's own deferred-CHECK decision creates without covering the resulting failure mode.

8. **[medium] The `@aws-sdk/client-kms` dependency version is never actually verified — only assumed "same version family" as `@aws-sdk/client-s3`.** The story otherwise takes pains to state claims are "confirmed by direct code inspection," and even timestamps its own read of the codebase (2026-07-11), but Task 1 and the Dev Notes justification for the AWS-only decision do not confirm that `@aws-sdk/client-kms` exists, is compatible, or resolves cleanly at the assumed version line — an inconsistent level of rigor compared to the rest of the story's evidentiary standard.

9. **[medium] `KmsKeyProvider.decryptDataKey(ciphertextBlob)` bakes an AWS-specific assumption (that `Decrypt` recovers key ID from ciphertext metadata) into the interface meant to support future providers.** AC-21 frames this interface as enabling "a future story adding a second provider... without touching `key-service.ts`'s core init/unseal logic," but GCP KMS's and Vault Transit's decrypt operations typically *do* require the key/key-version identifier to be supplied. The interface's shape already reflects one provider's peculiarity, undermining the extensibility claim it's used to justify.

10. **[medium] AC-15's "no new locking needed" claim hand-waves the effect of introducing a genuine network round-trip into a previously CPU/disk-bound critical section.** The await window for `Decrypt` is likely far larger and more variable than the previous local operations, widening the race window between concurrent unseal calls and any concurrent seal/shutdown path (`zeroKeys()`). The story asserts existing behavior is "unchanged" and needs no new guard, but does not analyze the specific scenario of a seal/SIGTERM occurring *during* an in-flight KMS `Decrypt` call racing a `commitUnsealedKeys()` — a scenario meaningfully more likely with a network-latency window than with the prior synchronous paths.

11. **[medium] No handling is specified for an unmapped/unrecognized AWS SDK exception type.** AC-3/4/5/11/12/13 map five known exception classes to five sanitized error responses, and AC-18 asserts the KMS wrapper "constructs its own sanitized `AppError` messages rather than forwarding raw SDK error text." But no AC addresses what happens when the SDK throws an exception type outside this enumerated set (a realistic possibility given AWS periodically adds new KMS exception classes) — does it fall through to a generic handler that might log `error.message`/`error.stack` verbatim, contradicting AC-18's own no-leak guarantee?

12. **[medium] `ExpiredTokenException`/STS session-expiry is not called out as its own error category despite AC-16 being specifically about credential rotation.** AC-16 discusses credential rotation and narrowed permissions (mapping to `403 kms_permission_denied`) but doesn't address the common STS/IAM-role failure mode of an expired or not-yet-refreshed session token, which AWS SDKs typically surface as a distinct exception from `AccessDeniedException` — a realistic gap in an AC that otherwise claims to be exhaustive about rotation scenarios.

13. **[low] FR109's key-custody-risk alert has no mechanism to flag a weak or overly permissive KMS setup.** AC-19 categorically treats `kms_type='kms'` as never risk-flaggable via `file_kms_with_backup`, but FR109's underlying intent (warn operators about weak master-key custody) is silently defeated for a `kms`-mode deployment whose IAM role or KMS key policy is itself overly broad (e.g., account-wide `kms:Decrypt` grants). The story doesn't need to solve this in v1, but doesn't flag it as an accepted limitation either — unlike its careful treatment of the DB-CHECK deferral (AC-7) and rotation-execution gap (Open Question 2).

14. **[low] No explicit client-side timeout/connection configuration is specified for the `KMSClient`.** AC-3/AC-11 describe a "timeout" error path but never specify what timeout value triggers it, nor confirm whether the existing `S3Client` construction in `storage.ts` (which this story explicitly mirrors) sets an explicit request timeout or relies on the AWS SDK's default (which can be long). An unresponsive KMS endpoint without an explicit short client timeout could hang unseal requests well past what an operator would tolerate during an incident.

15. **[low] Sentinel mismatch under `kms` mode is handled identically to a `passphrase`-mode typo, with no heightened audit signal.** AC-14 explicitly (and correctly) requires mode-independent sentinel verification, but a sentinel mismatch under `kms` mode is a qualitatively different, rarer, and more suspicious event (implying possible tampering of the `vault_state` row's ciphertext/sentinel pairing, since KMS itself already validated the wrapped key) than a passphrase typo. No AC calls for any differentiated audit/alerting signal for this specific combination, despite the story's otherwise strong emphasis on audit-trail value as the story's core motivation ("rely on... the cloud provider's... audit trail").

16. **[low] AC-8's claim that KMS mode is unconditionally "the most-secure option, not a downgraded one" is asserted without qualification.** This justifies skipping any `acknowledge*` risk-acceptance flag (unlike `envelope`/`file`), but KMS mode also introduces a new trust dependency — the entire AWS account's IAM configuration and blast radius — that is a different, not strictly lesser, trust model than a locally-held passphrase. The story doesn't examine or caveat this trade-off anywhere, despite examining nearly every other trade-off (e.g., AC-7's deferred CHECK, Open Question 2's rotation gap) explicitly.

17. **[low] Cross-account/foreign KMS key scenario is only tested at unseal time (AC-13's edge example), not at init time.** AC-1/AC-5 cover the general permission-denied case at init but never construct the cross-account-key scenario at init the way AC-13 explicitly does for unseal, leaving an asymmetric edge-case treatment between the two otherwise-parallel operations.
