### 1. [Medium] `packages/db` RLS-isolation test flake — confirmed real, correctly unowned pending a dedicated story

13 `packages/db` RLS-isolation test files fail with off-by-one row leakage on a freshly migrated,
*empty* database — reproduced by `10-4`'s coordinator via a clean `make docker-down-v && make db-up &&
make db-migrate && make check-rls` cycle (all 49 migrations applied cleanly, RLS coverage check itself
passed) followed by an isolated `apps/api` test run. Traced to `apps/api#test`'s turbo dependency on
`@project-vault/db#test` pulling in the flaky suite as a side effect, not to anything in `10-4`'s own
diff. This is a genuine, not-yet-investigated risk: "off-by-one row leakage" in an RLS isolation suite
is exactly the shape of finding that, if it turns out to be a real cross-tenant data leak rather than a
test-harness ordering artifact, would be security-relevant — the difference between those two
explanations has not yet been determined.
**Status:** Correctly out of `10-4`'s scope (zero diff, reproduces on unrelated fresh DB). **A separate
agent is already creating a dedicated investigation story for this in parallel with this retro** — this
retro does not schedule or duplicate that work, only confirms the finding is accurately characterized
in the record it's citing (`10-4-sonarcloud-new-coverage-buffer.md` lines ~1349–1369) and flags it here
so it isn't lost if that parallel effort stalls.
**Not previously known/tracked in `deferred-work.md`** — see Action Items below for the entry added
this session.

### 2. [Low] 10-1's Task 7.4 (first nightly E2E run) has no confirmed post-merge completion evidence

`10-1`'s own record explicitly states Task 7.4 (triggering `nightly.yml`'s `e2e` job for the first

**Resolution:** Non-blocking. Recommend a human check of the Actions tab for a completed `nightly.yml`

### 3. [Low] 10-3's coverage-buffer verifier has no negative-path test of its own

`apps/web`'s package-local `test:coverage-buffer` 85% verifier (built via TDD in `10-3`) enforces the
epic's own coverage-buffer discipline, but there is no test proving the verifier itself correctly fails
when given a coverage report below threshold — its correctness is currently established by "it was
built via TDD and it currently passes," not by an independent adversarial test of its failure path.
**Resolution:** Low priority; the risk is a silent regression in the gate's own enforcement logic, not
an active defect. Worth a follow-up if/when the coverage-buffer pattern is reused for `packages/db` or
`apps/api`.
