## Executive Summary

Epic 9 shipped four of six stories — encrypted backup/restore (`9-1`), system settings & multi-org (`9-2`), API parity verification & version upgrades (`9-3`), and an operational runbook (`9-5`) — with a fifth (`9-4`, platform operator audit log) in review and a sixth (`9-6`, backup hardening) also in review. The single biggest finding is the same one raised at every epic from 5 through 8: **the platform-operations UI gap was flagged by all five API-only stories (9-1 through 9-4, 9-6) and still had no `deferred-work.md` row going into this retro.** This is Finding 1 [REPEAT 4x] and a G2 gate blocker. A closure story (`9-7`) is scheduled as a backlog entry this session, and a new CI gate (`check-psc-tbd-tracking`) is implemented and wired into `make ci` and CI this session — so this cannot silently recur again.

## What Went Well

1. **P8-1 CI gate held for the first time — zero `Status:` header drift.** Five epics of manual retro sweeps finally have an automated successor. The gate worked.
2. **9-3's dual adversarial review passes caught a critical dollar-quoted-block bypass** in the migration safety guard before it could silently pass destructive migrations in production. Two review passes on one story is the correct call when the first pass misses a critical finding.
3. **The ADR-chain / D-numbered decision handoff worked at scale.** `9-4` picked up exactly where `9-1/9-2`'s forward-references (D6, AC-25/Open Question #5) said it would. Cross-story dependency management via named Dev Notes decisions, not implicit assumptions, is proving reliable.
4. **`9-3`'s contract-test suite caught four real pre-existing pagination gaps** (machine-users, projects, search, notifications) in already-shipped endpoints — bugs that predated this epic and would have remained invisible without a live enumeration test.
5. **`9-5`'s runbook found nine real drifts** between the story ACs and shipped behavior. Writing documentation by reading the code (not the spec) is functioning as a quality gate in its own right.
6. **Adversarial reviews covered all 5 code stories.** 9-6's adversarial review surfaced naming inconsistencies and concurrency edge cases that were incorporated into the story ACs before implementation started.

## What Didn't Go Well / Gaps, Debt & Risks

1. **Epic 9's full platform-operations UI gap was flagged by 5 stories and still had no deferred-work.md entry** — the same pattern as Epics 5/6/7/8 [REPEAT 4x].
2. **P7-1/P8-2 (PSC enforcement mechanism) were asked for twice and never built** until this retro built it. The pattern: a described behavior without a named artifact doesn't get built.

6. **Retro cadence ran behind delivery again** — Epic 9 had advanced to 9-5 done before Epic 8's retro ran (2026-07-07). This retro is running mid-flight (9-4/9-6 in review), which is an improvement, but still not "before the first story starts."

### 1. [Critical] [REPEAT 4x] Epic 9's platform-operations UI gap was not tracked in deferred-work.md

The UI gap covers: a backup/restore admin screen (`GET /admin/backups`, trigger, validate — 9.1/9.6), a system settings admin page (SMTP form, policy fields, backup schedule — 9.2), a resource usage dashboard (org/user/secret limits with 80/90/95% indicators — 9.2), a version-upgrade trigger and API-docs browser within the app shell (9.3), and a Platform Operator Audit Log page visually distinct from the per-org log per PJ9 with maintenance-mode banner (9.4). This is G2 gate condition 2. It recurs from: Epic 5 (P5-2 / A5-2), Epic 6 (Finding #1 / A6-1), Epic 7 (A7-1), Epic 8 (Finding #2 / A8-1/P8-2) — **[REPEAT 4x]**.
**Resolution:** Scheduled as `9-7-epic-9-completion-platform-operations-web-ui` (backlog, `sprint-status.yaml`). `deferred-work.md` "Web UI gaps" row added this session. `epic-9` held `in-progress` until `9-7` lands. `check-psc-tbd-tracking` CI gate implemented this session so this cannot silently recur.

### 2. [High] [REPEAT 2x] P7-1/P8-2 PSC enforcement mechanism was never built

Epic 7's P7-1 asked for "a CI or checklist gate that fails if a PSC TBD deferral isn't accompanied by a `deferred-work.md` entry." Epic 8's P8-2 asked for the same thing more specifically ("create-story should refuse `ready-for-dev` with a TBD PSC entry unless a matching `deferred-work.md` row exists"). Neither was built. No `check-psc*` script existed in `scripts/` and no PSC-related target existed in `Makefile` or `ci.yml` before this session. Finding 1 above (5 violations across one epic) is the direct evidence that prose asks without tooling enforcement don't hold. **[REPEAT 2x]** from P7-1 (Epic 7) and P8-2 (Epic 8).
**Resolution:** Built this session as `scripts/check-psc-tbd-tracking.ts` with 15 passing tests, wired into `Makefile` (`pnpm check-psc-tbd-tracking` in `ci:` target) and `ci.yml` ("Check PSC TBD UI deferrals tracked in deferred-work.md"). The check currently passes after the `deferred-work.md` entry for Epic 9's UI gap was added this session.

### 5. [Medium] `assetsPresentFromTables` never extended for Epic 8's compliance tables (9-1 D8 obligation)

Story 9-1's D8 explicitly states: *"this table list...must be extended if Epic 8 stories add new compliance-relevant tables before Epic 9 closes."* Migration 0037 (`data_subject_erasure`) added `data_erasure_requests`. `apps/api/src/modules/backup/dump-inspect.ts:29`'s `assetsPresentFromTables` still only checks `credentials`, `projects`, `users`, `audit_log_entries` — confirmed by direct inspection. A backup-restore validation that doesn't flag a missing `data_erasure_requests` table silently under-validates compliance data completeness — the `assetsPresent.valid = true` claim is misleading for an Epic 8+ schema.
**Resolution:** A one-line addition to `assetsPresentFromTables` in `dump-inspect.ts` and the `BackupAssetsPresent` type. Rolled into `9-7` as an AC (the closure story already touches backup admin surface — this is a natural fit). If `9-7`'s scope makes it cleaner to fix it standalone, a direct PR is acceptable; the point is it cannot be deferred past `9-7`.

### 6. [Medium] 9-4's adversarial-review high finding on maintenance-mode bypass has no trigger condition

The same adversarial review's findings #8–13 (medium: unbounded pending entries, PSC compliance-washing, no payload size limit, maintenance-mode rate limiting gap, SECURITY DEFINER boundary, concurrency guard either/or) share this pattern.
**Resolution:** Finding #5 rolled into `9-7`'s AC list or tracked in `deferred-work.md` as a named item. If neither happens before `9-7` ships, it must be explicitly added to `deferred-work.md` at `9-7`'s own retro — "accepted but unnamed" is the failure mode; "named debt with a file citation" is the minimum bar.

### 7. [Low] `/settings` placeholder copy is stale relative to the shipped settings API

### 8. [Low] 9-5's AC-25 human deployment verification was explicitly waived
