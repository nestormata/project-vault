## What Went Well

1. **Split-authorization pattern matured (4.2 D1).** The team correctly recognized that "org admin does X" is ambiguous and split every action into org-scoped vs. project-scoped authorization, with an org-owner override where appropriate. This precedent is now reused across 4.3/4.4 and is the right foundation for Epic 5's rotation authorization.

2. **Adversarial review caught two real critical gaps before they shipped unnoticed (4.2).** The formal review of Story 4.2 found that an admin could remove/demote a peer admin or even the sole org owner with no rank check — both were fixed post-review (D9 hierarchy guard, D5 last-org-owner guard). This is exactly what the Epic 1 retro's P5 mandatory-adversarial-review commitment was for, and it worked.

3. **Cross-epic sequencing handled honestly (4.4).** Story 4.4 depends on the `rotations` table that Epic 5's Story 5.1 creates, but Epic ordering says 4 before 5. Rather than blocking or faking it, the team built a `to_regclass()` table-existence seam with a CI guard test that force-fails once the real table exists — a clean, reusable pattern for future out-of-order cross-epic dependencies.

4. **Product Surface Contract discipline carried forward.** All four stories include the `## Product Surface Contract` section required since the Epic 2 retro, and Epic 4 avoided Epic 2/3's mistake of shipping an API with no linked UI story or honest placeholder.

5. **Stubs for unshipped dependencies are self-documenting.** Both the Epic 5 rotation-block stub and the Epic 7 machine-user-key stub are written as clearly-labeled `// TODO: Epic N` code with an explicit "QA must not sign off as complete until Epic N ships" caveat in the story files — this is honest, trackable technical debt rather than silent placeholder logic.

## What Didn't Go Well

1. **A real, verifiable contradiction shipped between two Epic 4 stories.** Story 4.4's spec (matching `epics.md`) requires the active-rotation block to return `{ error: "active_rotations", rotationIds }`. Story 4.3's shipped `checkActiveRotationsForUser` stub (`apps/api/src/modules/org/deactivation.ts`) actually returns `{ code: "active_rotation_blocking", message }` — an `ApiErrorSchema` shape. Both are harmless today because neither stub ever actually blocks (Epic 5 hasn't shipped), but the moment either becomes real, any client coded against one shape will break against the other. This was caught and documented in `deferred-work.md`, but not fixed.

2. **A high-severity finding from 4.3's own adversarial review was never closed: MFA re-enrollment during recovery doesn't regenerate recovery codes.** AC-15 promotes the pending MFA enrollment to confirmed but never calls `generateRecoveryCodes`/`insertRecoveryCodes` the way the normal enrollment path does. A user recovering from "lost device + lost recovery codes" finishes recovery with MFA active and **zero** recovery codes — recreating the exact lockout the flow exists to fix.

3. **Hierarchy-guard asymmetry within the same story.** Story 4.3's deactivation endpoint blocks admin-on-admin (AC-3), but its admin-initiated recovery-send-link endpoint (AC-10) has no equivalent rank check — any admin can trigger a password-reset link for any other admin, including a peer. The story documents every other ambiguity carefully (D1–D7) but doesn't call this one out as a deliberate choice.

4. **The project-role lookup helper deferral is now a 4-time-repeated debt item, not a 3-time one.** Stories 4.1 and 4.2 each deferred extracting a shared `resolveProjectRole()` helper with "if a 4th consumer appears, extract it" — 4.4 is that 4th consumer, and it still wasn't extracted. This is security-relevant authorization logic now duplicated in at least four call sites.

5. **Adversarial review coverage was inconsistent within the epic.** 4.2 and 4.3 got a formal adversarial review (and 4.2's review caught critical bugs). 4.1 and 4.4 — both auth/lifecycle-sensitive stories — did not, and rely only on informal deferred-work notes. Given 4.2's review track record, skipping it for 4.4 (archival, which also touches authorization and TOCTOU-sensitive locking) looks like an inconsistent application of the Epic 1 retro's own P5 commitment.

6. **Story-file-status hygiene regressed from Epic 3's own lesson.** Epic 3's retro explicitly flagged 3.1's file-vs-sprint-status drift as a P3 violation and made "story tracking hygiene" a team agreement. All four Epic 4 story files still say `Status: review` while sprint-status says `done` — the exact same class of drift, now epic-wide instead of one story.

7. **`.env.example` documentation gap repeated.** Story 4.1 explicitly required documenting `INVITATION_TOKEN_HMAC_SECRET` in `.env.example`. Story 4.3 introduces an equivalent hard-fail-on-missing secret (`RECOVERY_TOKEN_HMAC_SECRET`) but has no equivalent documentation requirement — an operator upgrading in place would hit an undocumented boot failure.

## Significant Discovery Alert

1. **Cross-story contract without enforcement is a repeating pattern, not a one-off.** Epic 4 alone has four instances of "Story A tells Story B (in a later epic) what to do" with zero CI/test enforcement. This is a process gap, not a code bug — worth a team agreement change before Epic 5 starts.

2. **A real security regression exists in shipped code, not just a spec-time finding.** The MFA-recovery-codes gap (Finding #2 above) means a specific class of user (lost device + lost codes) can complete "recovery" and land right back in the state that necessitated recovery. This should be treated as a bug, not a backlog nice-to-have.

3. **Adversarial review is applied inconsistently across a single epic.** 4.2 and 4.3 got one; 4.1 and 4.4 didn't. Since 4.2's review found critical issues that would otherwise have shipped silently, skipping it for two of four stories in the same epic is a governance gap worth closing before it becomes normalized.

**Epic update required for Epic 5?** No fundamental architectural change needed — but Story 5.1's opening notes should explicitly enumerate the four Epic-4-sourced requirements above as acceptance criteria, not leave them as prose the implementer must discover independently.

## Team Agreements (proposed)

1. Cross-epic requirements written in one story's Dev Notes for a *future* story to implement must become that future story's explicit acceptance criteria — not prose the author has to go find.
2. A story counting toward "epic done" gets a formal adversarial review if it touches authorization, session lifecycle, or destructive/irreversible state changes — applied consistently across every story in the epic, not selectively.
3. Story file `Status:` header updates are part of the same commit that flips sprint-status — re-affirming the Epic 3 agreement that clearly wasn't carried forward.

## Key Takeaways

1. The split org-role/project-role authorization pattern from 4.2 is solid and should be the template for Epic 5/7's own authorization design.
2. Adversarial review works when applied — 4.2's review caught two critical bugs pre-ship. It needs to be applied uniformly, not to 2 of 4 stories.
3. Epic 4 shipped a real, verifiable contradiction between two of its own stories (4.3 vs 4.4 response shape) — a concrete argument for reconciling cross-story contracts before calling an epic done, not after.
4. The MFA-recovery-codes gap is a shipped security regression, not a hypothetical risk — recommend treating it with the urgency of a bug fix.
5. Three epics (3, 4, 5) are simultaneously "in flight" with none formally closed — worth a deliberate decision on sequencing discipline before Epic 6/7 planning compounds it further.
