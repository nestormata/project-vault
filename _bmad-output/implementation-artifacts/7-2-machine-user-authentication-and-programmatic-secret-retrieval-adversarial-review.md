# Adversarial Review: Story 7.2 — Machine User Authentication & Programmatic Secret Retrieval

- **Date:** 2026-07-04
- **Reviewed file:** `_bmad-output/implementation-artifacts/7-2-machine-user-authentication-and-programmatic-secret-retrieval.md`
- **Reviewer:** bmad-review-adversarial-general

## Findings

- **[critical]** D3's JWT signing mechanism (namespaced `@fastify/jwt` HS256 vs. a hand-rolled `node:crypto` HS256 sign/verify fallback) is left unverified and unresolved at spec time — the story explicitly admits it could not confirm the `@fastify/jwt` `namespace` option's API and instructs the developer to fall back to hand-rolled JWT crypto if the library doesn't support it, with no additional security review gate, vetted-library requirement, or constant-time-comparison mandate specified for that fallback path. This is the single most security-critical primitive in the entire story (machine authentication), and "roll your own JWT crypto if the library doesn't cooperate" is a well-known high-risk pattern (algorithm confusion, non-constant-time signature comparison, malformed compact-serialization edge cases).

- **[high]** AC-8 contains an internally self-contradictory justification for a security-relevant authorization decision: it asserts "`viewer` is a read-capable role for credential values in the human-user model" and in the very same sentence states the analogous human endpoint "gates at `minimumRole: 'member'` for humans" (i.e., viewer is explicitly *not* sufficient for humans). The decision to grant machine users with role `viewer` full plaintext secret-value read access — exceeding what a human `viewer` can do — is justified by this self-contradicting premise rather than a clean rationale.

- **[high]** AC-6's example response JSON (`{ name, value, versionNumber }`) and the AC Quick Reference table both omit the `cacheable` field, yet AC-14 states the response schema is "extended per AC-6" to include `cacheable: boolean`, implying AC-6 already contains it. This is a direct contradiction between AC-6's literal schema/example and AC-14's claim about it — a developer implementing AC-6 to the letter would omit `cacheable`, breaking AC-14's non-cacheable-exclusion logic in the agent.

- **[high]** AC-15's cache-activation beacon mechanism is left as an unresolved "either/or" design choice ("`POST /api/v1/auth/machine-token`'s response... or a lightweight dedicated `POST /api/v1/machine/cache-activated` beacon endpoint"). Unlike every other cross-cutting design question in this story, which is explicitly resolved via D1–D12, this security/audit-relevant plumbing (the only server-side visibility into an offline-fallback activation) is punted to implementer discretion. Consequently Task 15's "8 new route entries" count does not account for this potential new endpoint, and no AC formally specifies its request/response shape, auth model, or rate limit the way every other endpoint in this story does.

- **[high]** The offline agent's cache file (`~/.project-vault/cache.json`) has no documented protection against concurrent-process write races. AC-12/AC-13 specify encryption and permission bits but never address what happens when multiple concurrent CI processes/steps on the same host call `getSecret()` and write to the same shared cache file simultaneously — a scenario directly relevant to the story's own stated primary use case (CI/CD) — which could trigger the very `VaultCacheCorruptedError` path AC-13 defines, causing spurious hard failures in parallel CI jobs.

- **[high]** The AC-4 per-key-hash brute-force rate limit is logically ineffective against the threat it claims to defend. Since HMAC-SHA256 is deterministic, an attacker guessing unknown candidate keys produces a different `keyHash` on nearly every attempt, so the "10 failed attempts per `keyHash` per 60s" bucket is essentially never triggered during a real credential-guessing/brute-force attack (including the "botnet spreading attempts across many IPs" scenario the AC itself cites) — it can only fire when the exact same fabricated string is retried repeatedly, giving a false sense of defense-in-depth.

- **[medium]** AC-18's hourly overlap-revoke job cadence (`0 * * * *`) is coarser than the minimum allowed `overlapMinutes: 1` (permitted per AC-17's 1–1440 range). A key rotated with a short overlap window could remain valid and unrevoked for up to ~59 minutes past its configured expiry purely due to cron granularity, undermining the "old key auto-revoked after window" guarantee for short overlap windows.

- **[medium]** No AC addresses what happens to already-fired (pending/delivered, non-dismissed) `machine_key.dormant` alerts when an admin changes `organizations.machine_key_dormancy_threshold_days` via the new `PATCH .../machine-key-settings` endpoint (D8) — e.g., raising the threshold from 90 to 180 days after an alert already fired leaves a now-stale alert in the admin queue with no defined reconciliation behavior.

- **[medium]** Task 4's deliverable list only mentions `writeMachineAuditEntry`/`writeMachineAuditEntryOrFailClosed`, but AC-18 introduces an entirely new `actorType: 'system'` and a corresponding `writeSystemAuditEntry()` helper (or "system variant") that is never mentioned in D5 — the section explicitly meant to fully resolve audit actor typing — nor listed as a task deliverable. A developer following the Task list and D5 literally could miss this required helper.

- **[medium]** The pre-auth token-exchange flow (AC-2) has a narrow TOCTOU window: key validity (`revokedAt`/`expiresAt`) is checked in step 4, but the `lastUsedAt` update and JWT issuance in steps 5–6 are not re-verified against a concurrent revocation happening in between. Unlike AC-26's explicit `SELECT ... FOR UPDATE` handling for rotation/emergency-revoke concurrency, this race is not analyzed anywhere in the story.

- **[medium]** No rate limiting or anomaly detection distinguishes failed (404/409) vs. successful lookups on `GET .../credentials/:name/value`. A stolen machine JWT (valid up to 1h) could enumerate credential names within its scoped project at up to 300 requests/minute (AC-27) with no separate "failed lookup" throttle, unlike the explicit failed-attempt throttling this story mandates for the key-exchange endpoint (AC-4).

- **[low]** The story mandates two functionally-identical but independently-implemented AES-256-GCM/HKDF codebases (`packages/crypto/src/aes.ts` server-side and `packages/agent/src/cache-crypto.ts`, D11) enforced only by a code comment — no AC or task requires a cross-compatibility test (encrypt with one implementation, decrypt with the other) to prove the claimed "byte-identical envelope format" actually holds, leaving future silent drift undetected until a real decrypt failure in production.

- **[low]** Dismiss and extend-dormancy admin mutations (AC-22) are not explicitly stated to require MFA enrollment/grace, unlike rotate (AC-16) and emergency-revoke (AC-20), which do. Since extend-dormancy can indefinitely snooze detection of a potentially-compromised dormant key, this inconsistency in MFA gating across "sensitive" machine-user admin actions is worth resolving explicitly.

- **[low]** The FR101/FR110 PRD traceability gap is explicitly flagged by the story itself as pre-existing and deliberately not fixed here — real technical debt carried forward rather than resolved, meaning two functional requirements this story implements against have no PRD-level acceptance criteria to validate against, only epics.md's annotations.
