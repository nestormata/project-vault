# Adversarial Review: Story 10.1 — Playwright E2E Test Automation

- **Date:** 2026-07-09
- **Reviewed file:** `_bmad-output/implementation-artifacts/10-1-playwright-e2e-test-automation.md`
- **Reviewer:** bmad-review-adversarial-general

## Findings

1. **[critical]** AC-J2-1's happy-path setup is built on a helper that cannot pass the real gate the flow depends on. AC-I4 designates `registerAndLoginViaApi` (plain register+login, no MFA step) as the standard way J2 reaches its starting state, and AC-J2-1's own "Given" clause uses exactly that helper for the inviting org owner. But the real `POST /:projectId/invitations` handler (`apps/api/src/modules/invitations/routes.ts:163-166`) calls `requireMfaEnrollmentStrict()` unconditionally before any invitation logic — a gate that, unlike `security.requireMfa`'s grace-respecting `requireMfaEnrollment()`, explicitly ignores the 7-day MFA grace period (`MFA_PRIVILEGED_ROLE_GRACE_DAYS`, default 7, `.env.example:74`) for owner/admin callers (confirmed intentional: Story 4.1 D2, `epics.md` AC-E1c, `mfa-policy-matrix.md`). A freshly-registered owner with no MFA enrollment — precisely what `registerAndLoginViaApi` produces — will receive `403 mfa_required` on the very first action AC-J2-1 attempts. Nothing in Group J2's ACs or Dev Notes calls for an MFA-enrollment step before inviting; as literally specified, AC-J2-1 (and everything downstream in J2/J3 that assumes an invite already happened, e.g. AC-J2-3) cannot pass against the real API without a fixture change the story never mentions.

2. **[high]** No AC, task, or Dev Note addresses whether the real, enforced (non-test-bypassed) rate limiters on `/auth/*` routes can survive a full run of the four journeys. `isRateLimitEnforced()` (`apps/api/src/lib/route-helpers.ts`) only bypasses rate limiting under `NODE_ENV=test`; the E2E stack is the real `docker compose` API container, which does not run under that mode, so every limiter is live: a global 60-req/min-per-IP bucket on all `/auth/*` routes (`routes.ts:377-389`) plus route-specific buckets (register: 10/min, mfa/verify-login: 20/min, etc.), all keyed by IP — and every Playwright request in CI/local dev shares one IP. This is not a hypothetical: this exact class of bug (`register-rate-limit.test.ts`, the "register rate-limit fix" `nightly.yml:90` explicitly calls out from Story 3.4) has already bitten this codebase's own Vitest suite once. AC-I3/AC-J3-2 show the author is aware rate limits exist (J3-2 explicitly defers exhausting `MFA_LOGIN_MAX_ATTEMPTS` to avoid this), but the same scrutiny was never applied to the cumulative register/login calls across J1-J4 in one serial run, nor to whether CI should set an E2E-scoped rate-limit override.

3. **[medium]** AC-I5's CI-precedent claim overstates its own citation: "uploads the HTML report + traces on failure (`actions/upload-artifact`, 30-day retention, matching this repo's own existing `db-coverage-*` artifact-upload precedent in `ci.yml`)" — the actual precedent (`ci.yml:116-122`) uses `retention-days: 90`, not 30. Either the AC should cite the real number or justify diverging from it; right now it misrepresents the precedent it claims to match.

4. **[medium]** J3's "PRD source journey(s)" justification in the Journey Selection table is qualitatively weaker than it's presented alongside J1/J2/J4. J1/J2/J4 each cite specific named personas and quoted PRD lines; J3's row instead says "Cross-cutting security control... explicitly named in this story's own creation-task example list" — i.e., not actually grounded in the PRD's `Journey Requirements Summary` table at all. Checked directly: MFA does not appear in any of the five personas' "Requirements revealed" lists (`prd.md#Journey-Requirements-Summary`). The engineering rationale for including MFA (heavily audited, zero browser coverage) is reasonable on its own merits, but the story's framing implies all four journeys were selected by the same "cross-reference PRD journeys against what's shipped" methodology, when J3 was actually selected by a different, unstated criterion. This isn't necessarily the wrong call, but the justification section is not as apples-to-apples as it reads.

5. **[medium]** AC-I2's folder-naming rationale ("architecture.md's illustrative file names... are adapted, not copied verbatim") correctly flags the `SecretDetailPage.ts`/`secrets.spec.ts`-style staleness in the *file names*, but the story never notes that `architecture.md`'s broader routing tree in the same document (line ~1115, `(auth)/mfa/` listed as its own route folder) is also stale in a way directly relevant to J3: the story's own "Confirmed real routes" table states outright that "there is no separate `/login/mfa` URL" (confirmed correct — no `routes/(auth)/mfa` directory exists), which flatly contradicts architecture.md's illustrated tree just a few dozen lines away from the section this story quotes. Given the task description says a parallel effort just reconciled architecture.md's secrets→credentials naming, this specific route-tree inaccuracy (not a naming issue, a route-existence issue) appears to have been out of that reconciliation's scope and is still sitting there uncorrected — worth flagging as a follow-up doc-fix, since this story had to independently re-derive the correct routing via direct source inspection rather than being able to trust the architecture doc here.

6. **[low]** AC-J1-2's implementation-time verification pointer ("verify the exact current rule against `apps/api/src/modules/auth/schema.ts`") sends the implementer to the wrong file. `apps/api/src/modules/auth/schema.ts` only re-exports `RegisterRequestSchema`; the actual password-length rule (`z.string().min(12).max(256)`) is defined in `packages/shared/src/schemas/auth.ts`. A developer following the citation literally will still find the re-export and can trace one hop further, but the citation itself is imprecise about where the rule actually lives.

7. **[low]** AC-J3-2's "Example (edge — session hygiene)" frames whether pending-MFA state survives a `/login` page reload as something to "verify at implementation time; do not assume." This is actually fully determinable from the shipped code without running anything: `LoginForm.svelte` holds `mfaToken` in a plain component-local `$state`, with no persistence to a cookie, `sessionStorage`, or server-rendered prop — a full page reload always re-mounts the component with `mfaToken = null`, so the challenge state can never survive a reload. Framing this as an open question suggests the story's "verify, don't assume" discipline (explicitly invoked in Dev Notes for several other items) wasn't applied here even though the answer was one file-read away; the AC's instruction to assert on shipped behavior is still correct, just less uncertain than presented.

8. **[low]** AC-I3's serial-execution trade-off (`workers: 1`, `fullyParallel: false`) is well-justified against the specific worker-parallelism collision risk (multiple browser contexts racing the same shared dev Postgres), and each spec's per-test unique org/user avoids the `9-7`/`8-6`-style leaked-org accumulation *within* a single run. However, AC-I3 only resets the DB once per `playwright test` invocation, not once per spec — so repeated `make e2e` iterations during active local development (the exact workflow AC-I6's "run twice in a row" example is designed to prove) still accumulate all of J1-J4's orgs/users for the duration of one run, then reset only at the next invocation. That's an accepted, documented trade-off, not a bug, but the story doesn't quantify how many orgs/users one full run creates (rough count from Groups J1-J4: ~7-9 registrations) or that this number will grow every time a 5th+ journey is added — worth a one-line acknowledgment for future-suite-size planning, per the story's own "revisit once serial runtime becomes the bottleneck" note, which currently only discusses runtime, not accumulated per-run row counts.

9. **[low]** AC-I1 specifies `retries: 1` in CI. Combined with AC-I3's design (no per-spec isolation beyond unique org/user names, one shared DB across the whole run), a retried spec re-runs against a DB state that already includes every side effect of its own first (failed) attempt — e.g. a J2 spec that failed after successfully creating an invitation would, on retry, attempt to invite the *same* generated email again if `fixtures/ids.ts`'s uniqueness isn't re-rolled per retry attempt. The story doesn't specify whether Playwright's retry mechanism gets fresh `uniqueEmail()`/`uniqueOrgName()` values (it does, by default, since retries re-execute the whole test function) — this is very likely fine given Playwright's actual retry semantics, but it's the kind of interaction between two independently-reasonable ACs (I1's retries, I3's shared-DB-within-run model) that the story never explicitly connects, and is worth a one-line note in Dev Notes rather than leaving it as an implicit assumption.

10. **[nitpick]** The Journey Selection table's "Explicitly deferred" list gives a clear, checkable reason for each deferred journey (CI-Bot: API-only; Dana: already has component coverage from 8.7; platform admin: Epic 9's territory; monitoring: natural follow-up) — all of which check out against the real `sprint-status.yaml`/`apps/api-contract-tests` state. No issue found here; noted only because the task's review brief specifically asked for scrutiny on cherry-picking and none was found on this particular axis — the four selected journeys are a defensible, PRD-traceable set for J1/J2/J4 (J3's traceability gap is finding #4 above).

## Verified as Accurate (not findings, but scrutinized and confirmed correct)

- The three cited `sprint-status.yaml` DB-pollution incidents (8-6's "DB-state pollution from repeated runs against the same persistent Postgres volume", 8-7's rotation-module concurrency test rewrite, 9-7's "8,563 leaked test orgs") are all verbatim-accurate, not paraphrased or exaggerated.
- `nightly.yml`'s three existing jobs (`mutation`, `flaky-test-repeat`, `trivy-image`) genuinely run on `schedule: cron '0 2 * * *'` + `workflow_dispatch`, not on `ci.yml`'s PR-blocking path — AC-I5's precedent claim is accurate, and `flaky-test-repeat`'s DB-reset step matches AC-I3's description exactly.
- `scripts/docker-smoke.sh`'s readiness-poll pattern matches AC-I3's description.
- AC-J4-3's concurrency-guard claim is accurate: `apps/api/src/modules/rotation/routes.ts` does return `409 rotation_in_progress` when initiating a second rotation while one is active.
- AC-J4-2's minimum-checklist-gate claim is consistent with `epics.md`'s AC-E5a and the real `/complete` route's `422 ChecklistIncompleteResponseSchema`/`AcknowledgementRequiredResponseSchema` handling.
- AC-J1-3's reveal-permission citation is correct, if non-obviously named: the credential detail page's reveal gate (`canReveal = $derived(canCreateCredential(data.orgRole))`) really does reuse the same function AC-J1-3 cites, confirming the two-tier (viewer-denied / member+-allowed) permission boundary the AC describes.
- All ten routes in the "Confirmed real routes" table exist exactly as stated (verified by direct directory listing).
- Rotation completion's `requireMfa: true` gate does **not** block J4's freshly-registered owner, because it's grace-respecting (not strict) and the default 7-day grace period covers the duration of a test run — unlike the invitations route (finding #1), which uses the strict variant.
