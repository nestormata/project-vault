## Findings

- **[HIGH]** The Story section's own restatement of Team Agreement TA-1 silently narrows the source text it claims to quote. `epic-8-retro-2026-07-07.md` Finding 3 and `epic-9-retro-2026-07-08.md` Finding 3 both state TA-1 as: *"A story touching **authorization, session lifecycle**, or destructive/irreversible state changes gets a formal adversarial review"* (emphasis added). Story 8.8's own "Story" section defines TA-1 as *"every story touching destructive/irreversible state changes gets a formal adversarial review"* — dropping the "authorization, session lifecycle" clause entirely. Because AC-2's anti-scope-creep argument depends on the premise that TA-1 = destructive actions only, and AC-8 separately surfaces a real `minimumRole` vs `allowedRoles` authorization-gating inconsistency across the 6 actions, this narrowed restatement could licitly be used to wave off an authorization-lifecycle finding as "out of scope" under AC-2, when the actual TA-1 text this story is supposed to honor explicitly also covers authorization changes. This is a real, checkable contradiction between this story's own paraphrase and its cited sources.

- **[HIGH]** AC-8's "Given" clause misquotes the actual shipped RLS policy SQL and omits an adjacent comment that already answers the exact question the AC poses. The story states the policies read `USING (org_id = current_setting('app.current_org_id', true)::uuid)`. The actual shipped SQL (`packages/db/src/migrations/0029_machine_users_and_api_keys.sql:59-65`, confirmed by direct read) is `USING (org_id = NULLIF(current_setting('app.current_org_id', true), '')::uuid)` — a materially different expression (explicit empty-string handling via `NULLIF`). Worse, lines 57-58 of that same file — immediately above the cited range and excluded from it — already contain a comment stating: *"WITH CHECK defaults to USING for command-less ALL policies (see 0013_projects.sql / 0014_credentials.sql precedent) — omission here is intentional, not a gap."* AC-8 frames "no explicit `WITH CHECK`" as an open fact for the reviewer to note, without ever citing (or asking the reviewer to check for) this pre-existing, intentional-design comment that already settles it. A reviewer following only this AC's citation range risks re-litigating an already-documented decision, or missing that the quoted SQL doesn't match what's actually deployed.

- **[HIGH]** The story's Prerequisites table instructs reviewing "8-6's shipped code as it exists on `main` today," but never accounts for the fact that the exact files this story cites (`apps/web/src/routes/(app)/notifications/+page.svelte` and `+page.server.ts`) have already been modified by a *later* story. Direct inspection shows both files now contain code explicitly commented `// Story 8.7 AC-H1` / `// Story 8.7 extends 8.6's machine-key section`, adding an out-of-scope "Dormant user alerts" section and a `deactivateDormantUser` action that reuse the same generic `dismissDormancyAlert` action and the same underlying dismiss endpoint/audit-event type this story's AC-7/AC-12/AC-13 review. Despite this, the story's own line-number citations for these files (e.g. `+page.server.ts:152-162`, `+page.svelte:83-105, 107-129`) remain accurate against the current state only because 8-7's additions happen to be appended after the cited ranges — a fragile coincidence, not something the story verified or flagged. The story gives the reviewer no instruction on how to distinguish 8-6-authored destructive-action code from 8-7-authored additions sharing the same files/endpoints, creating real risk of scope drift or misattributed findings once the review is actually executed.

- **[MEDIUM]** AC-13's own "Given" clause contains an arithmetic/factual error about its own 6-action scope table. It states: *"4 of the 6 actions call `writeHumanAuditEntryOrFailClosed()` ... with `AuditEvent.*` constants ... while the dismiss handler ... uses the literal string."* That accounts for 4 + 1 = 5 of the 6 in-scope actions. The 6th, extend-dormancy, is uncounted by this sentence — and direct inspection of `routes.ts:742-754` confirms extend-dormancy *also* calls `writeHumanAuditEntryOrFailClosed()` with the constant `AuditEvent.MACHINE_USER_DORMANCY_EXTENDED`, not a literal string. The correct count is 5 of 6 use constants, 1 of 6 (dismiss) uses a literal string. This is a small but concrete internal-consistency defect in a document that otherwise prides itself on precise, verified citations (AC-1's whole premise is "don't trust unverified claims") — and it's exactly the kind of small miscount that erodes confidence in the rest of the scope table's rigor.

- **[MEDIUM]** AC-9 and AC-10 both pre-guide the reviewer toward classifying their most concretely-identified, highest-confidence findings (missing `Promise.all`-style concurrent tests for double-revoke and double-deactivate) as `[MEDIUM]` ("missing test coverage for an otherwise-sound mechanism," explicitly contrasted against `[HIGH]`/`[CRITICAL]` "broken behavior"). Since AC-15 only mandates TDD-fix-now for `[CRITICAL]`/`[HIGH]` findings, this means the two concurrency gaps the story spends the most analytical effort setting up (AC-9, AC-10, and by extension AC-11's cross-action gap) are structurally pre-routed away from ever triggering AC-15's fix-or-track mandate's stronger branch. For a story whose entire reason to exist is "A8-2 promised a real review, not a waiver," this is in tension with AC-14's "not a waiver" spirit: the most certain, most anticipated findings in the whole document are the ones least likely to produce a code change.

- **[MEDIUM]** AC-13 does not ask the reviewer to check whether *user-supplied* free-text or numeric fields persisted into the audit trail could themselves constitute a secrets/PII leak — it only checks that the system's own plaintext API key never appears in a payload. But the dismiss handler's audit payload (`security-alert-actions-routes.ts:72`, confirmed) persists the admin's typed `reason` field verbatim into a permanent audit record with no sanitization or length/content check, and `DismissDormancyAlertForm.svelte` places no constraint on that field beyond `required`. An admin who pastes a secret, token, or other sensitive text into that free-text box (plausible in a dormancy-alert workflow already discussing specific keys) would have it durably logged. AC-13's audit-coverage review area, as written, would not catch this class of gap because it only tests for the *known* secret (the API key), not for unbounded user input flowing into the audit trail.

- **[MEDIUM]** AC-16's Definition of Done requires the artifact to exist, be non-empty, cover every AC-4–AC-13 area, and (for Critical/High findings) show a fix with a corresponding test — but it never requires any evidence that the shipped code was actually *executed* (e.g., running the cited test files, or the reviewer's own new concurrent-test attempts) as part of producing the artifact. A reviewer could satisfy every literal AC-16 bullet by writing well-cited, plausible-sounding prose findings derived purely from reading code, without ever running `pnpm vitest` against a single one of the cited test files. Given the task's own concern — that this retroactive-review story could be "completed" by a superficial pass that never truly exercises the shipped code paths — AC-16 does not close that loophole; it only requires the artifact to *look* complete, not to prove it was empirically grounded.

- **[MEDIUM]** AC-15's triage process is entirely self-certifying: the same single agent/session that discovers a finding also assigns its severity, decides whether a Critical/High finding's deferral justification is legitimate ("requires a schema change out of scope for a review story"), and writes the corresponding `sprint-status.yaml` backlog entry — with no independent second reviewer or sign-off step required anywhere in AC-14/AC-15/AC-16. This is the same unchecked, single-actor process that allowed A8-2 itself to go unbuilt for a full epic cycle (owned by "Dana, QA" but never independently verified as actually done, per `epic-9-retro-2026-07-08.md` Finding 3). A story whose explicit purpose is to stop a promised review from silently becoming a rubber stamp does not add any structural check against the same failure mode recurring inside its own triage step.

- **[LOW]** AC-13's finding about the `security_alert.dismissed` literal-string-vs-constant drift undersells how widespread the drift actually is. Direct inspection of `apps/api/src/modules/org/security-alerts.ts:144` shows the identical literal string `'security_alert.dismissed'` (instead of `AuditEvent.SECURITY_ALERT_DISMISSED`) is also used in the older, out-of-scope `dismissSecurityAlert()` function (Story 6.2's route). AC-13 frames this as "a real, concrete, low-risk-but-real drift" found at a single call site, when in fact the same drift recurs at a second call site the story never mentions — a reviewer following only AC-13's citation would understate the scope of the pre-existing convention violation.

- **[LOW]** AC-8 instructs the reviewer to verify the `minimumRole: 'admin'` vs `allowedRoles: ['owner', 'admin']` inconsistency "against the shared authz helper, don't assume" but never names or cites that helper. Direct inspection shows the relevant logic is `roleRank(auth.orgRole) >= roleRank(minimumRole)` at `apps/api/src/lib/secure-route.ts:214-215`. Every other AC in this story gives a precise file:line citation for the code it wants verified (a strength of the document overall) — this is the one place where the story asks for independent verification without pointing at what to verify against, an isolated gap in an otherwise unusually well-cited document.

- **[LOW]** AC-8 discusses only the `minimumRole` vs `allowedRoles` *type* inconsistency across the 6 actions' authorization gates, but never asks the reviewer to compare their *rate-limit* configurations, which are also inconsistent in a security-relevant way: the 4 machine-users-module actions share `MACHINE_USER_MUTATION_RATE_LIMIT = 10` (per window; `routes.ts:178`), while the dismiss endpoint allows `max: 30` per 60-second window (`security-alert-actions-routes.ts:41`) — three times the throughput. A compromised owner/admin session (or a session riding on a stolen MFA-satisfied cookie) could mass-dismiss dormancy alerts at 3x the rate it could revoke/deactivate/rotate, which is directly relevant to a story whose whole subject is destructive/irreversible admin actions on security-alerting infrastructure. No AC in this story asks the reviewer to notice or flag this numeric asymmetry.

- **[LOW]** The Scope Definition table's citation for the Deactivate UI entry point (`+page.svelte:116-127` for `onDeactivate()`) undercounts the actual function body. Direct inspection shows `onDeactivate()` spans lines 116–131 (including its `finally` block), not 116–127 as cited — a minor drift of ~4 lines in an otherwise remarkably precise citation set (nearly every other file:line citation in this story was independently verified accurate against the current shipped code during this review).

- **[LOW]** Neither AC-8 through AC-13 nor the Dev Notes ask the reviewer to record which git commit / `main` HEAD the review was actually performed against, despite the Prerequisites table's explicit concern that this review must not run "against an in-flight or hypothetical version of 8-6." Given the codebase has already moved past 8-6's original merge (per the HIGH finding above re: Story 8-7's later changes to the same files), a future reader of the eventual findings artifact has no way to independently confirm exactly what state of the repo the review's file:line citations were verified against, which weakens the artifact's reproducibility and its usefulness as a durable, checkable record — the same durability concern AC-3/AC-16 otherwise take seriously.

- **[LOW]** AC-14 requires that any deferred Critical/High finding "name the specific new backlog story key," but does not itself require that key to exist in `sprint-status.yaml` at the time the verdict section is written — only AC-16's final Definition-of-Done gate checks for a "real `sprint-status.yaml` backlog key." This creates a sequencing gap: AC-14 could technically be satisfied by naming a not-yet-created key in prose, with the inconsistency caught only at the very last DoD check (AC-16) rather than being enforced at the point the verdict itself is written.

## Summary

- **Total findings: 14** (3 High, 6 Medium, 5 Low; 0 Critical).
- No finding in this review concerns Story 8.8's ACs being *too permissive of a bad shipped-code outcome* in the sense of missing an obvious real defect in 8-6's code — the story's own predictive "negative examples" (e.g. AC-12's unconditional `extend-dormancy` update with no `revokedAt` check) were independently confirmed accurate against the actual shipped code during this review, and most of the story's ~50+ file:line citations were spot-checked and found precise. The findings above are about the *story's own* internal rigor: a narrowed restatement of the very Team Agreement it exists to enforce, a misquoted/incompletely-cited RLS policy justification, an unacknowledged scope-boundary risk from a later story modifying the same files, an internal arithmetic error, and several structural gaps (self-certifying triage, no execution-evidence requirement, no rate-limit asymmetry check) that could allow a technically-compliant but substantively shallow review to satisfy this story's own Definition of Done.
