# Adversarial Review — Story 5.3: Stale Rotation Recovery & Break-Glass Emergency Rotation

**Date:** 2026-07-01
**Reviewed file:** `_bmad-output/implementation-artifacts/5-3-stale-rotation-recovery-and-break-glass-emergency-rotation.md`
**Reviewer:** bmad-review-adversarial-general

---

## Resolution (2026-07-01)

All `critical`/`high` findings (1-5) were addressed directly in the story file:

- **#1 (critical, duplicate active rotation via `stale_recovery` gap):** 5.1's partial unique index `idx_rotations_one_in_progress_per_credential` is widened, in this story's own migration, to `idx_rotations_one_active_per_credential ON rotations(credential_id) WHERE status IN ('in_progress', 'stale_recovery')`. Added as CR8/ADR-5.3-08, wired into AC-1's migration text, the Conflict Resolution table, the AC Quick Reference, Pre-mortem item 8, and a new Anti-Patterns bullet. Since 5.1 has not shipped yet, this lands as a single widened index from the start rather than a follow-up `ALTER`; 5.1's initiation endpoint must map the resulting `23505` to its existing `409 rotation_in_progress` code (documented in CR8/ADR-5.3-08 and AC-1).
- **#2 (high, AC-5's "at most one row" claim was unenforced):** now a real guarantee under the widened index (#1) — AC-5 step 1's justification text was corrected accordingly.
- **#3 (high, AC-11 resume's unhandled `23505` risk):** eliminated structurally by the same index widening (#1) — a resumed row was already counted in the constrained set, so its own status flip can never violate the constraint. AC-11 now documents why no defensive handling is needed, rather than requiring new guard code.
- **#4 (high, AC-5's blocking `FOR UPDATE` could stall behind a concurrent 5.2 call):** AC-5 step 1 now uses `FOR UPDATE NOWAIT`, mapping lock failure to the existing `409 rotation_lock_contention` shape (AC-6 updated to document this second contention source and a new test).
- **#5 (high, unsanitized `reason` field reaching outbound Slack/email delivery):** AC-7 gained an explicit "Reason-field sanitization" requirement — verify (or add) escaping in `apps/api/src/notifications/templates/index.ts` before wiring the notification payload, with a dedicated test using Slack mrkdwn and HTML-special-character `reason` values. Cross-referenced from "Notification & Alert Routing," Pre-mortem item 10, and a new Anti-Patterns bullet.

Medium/low findings (6-15) were left as-is — none were blocking, and resolving them was out of scope for this pass.

## Findings

- **[critical]** No mechanism anywhere in this story (or 5.1/5.2) prevents a *normal* `POST .../credentials/:credentialId/rotations` initiation while an existing rotation for that same credential is sitting in `stale_recovery`. The only durable backstop against duplicate active rotations is 5.1's partial unique index `idx_rotations_one_in_progress_per_credential`, which is scoped to `WHERE status = 'in_progress'` **only**. Once AC-9's job flips a rotation to `stale_recovery`, that row falls outside the index's constraint set, and nothing in 5.1's AC-4 initiation flow checks for an existing `stale_recovery` row before inserting a fresh `in_progress` one. The result: a credential can end up with two simultaneously "active" rotations (one `stale_recovery`, one `in_progress`), directly contradicting the "at most one active rotation per credential" invariant this story assumes throughout (see AC-5's "at most one row" claim below). Given a 60-minute default stale threshold and no UI yet to guide an admin to `resume`/`abandon`, an operator retrying rotation via script or API call after seeing a "stuck" rotation is a highly plausible path to trigger this.

- **[high]** Direct consequence of the above: AC-5's break-glass supersede query — `SELECT id, new_version_id, previous_version_id FROM rotations WHERE credential_id = $1 AND status IN ('in_progress','stale_recovery') FOR UPDATE` — is documented as returning "at most one row, per 5.1's partial unique index invariant," but that invariant does not cover the `stale_recovery` half of the `IN (...)` clause. If both an `in_progress` and a `stale_recovery` rotation exist for the same credential, this query can return two rows, and AC-5 steps 2-4 (`UPDATE ... WHERE id = <found id>`) are written for a single scalar id — undefined/broken behavior (which row wins? does the implementer loop, or does the code just take the first row and silently strand the second?) is left completely unspecified.

- **[high]** Also a direct consequence of the first finding: AC-11's resume happy path — `UPDATE rotations SET status = 'in_progress', version = version + 1 WHERE id = $rotationId AND status = 'stale_recovery'` — has no defense against violating `idx_rotations_one_in_progress_per_credential` if a second, independently-initiated `in_progress` rotation already exists for the same credential (reachable via the gap above). This would raise an unhandled `23505` unique-violation inside the resume transaction, surfacing as an unhandled 500 rather than the clean, enumerated 409/422 error contract this story establishes everywhere else (`rotation_lock_contention`, `concurrent_modification`, `rotation_not_stale`, etc.). No AC or test in this story anticipates this failure mode.

- **[high]** AC-5's row-acquisition step uses a plain `SELECT ... FOR UPDATE` (no `NOWAIT`/`SKIP LOCKED`) to find the rotation to supersede. Break-glass's own advisory lock is **credential-scoped**, while a concurrent human `confirm`/`fail`/`retry`/`complete` call (5.2's endpoints) acquires a **rotation-scoped** advisory lock — two different key domains that do not serialize against each other. If a human mutation is mid-transaction on the very rotation break-glass is about to supersede, break-glass's `SELECT ... FOR UPDATE` will silently **block** waiting for that transaction's row lock to release, rather than failing fast. This directly contradicts the fail-fast, non-blocking design philosophy this story and 5.1 repeatedly emphasize (`pg_try_advisory_xact_lock` was chosen specifically to avoid blocking waits) and undermines break-glass's stated premise of acting "in seconds during a breach."

- **[high]** The `reason` field on break-glass (free text, up to 1024 chars, fully admin-controlled) is inserted verbatim into the audit payload **and** the enqueued notification payload (AC-7), which is delivered externally via email/Slack using the generic fallback renderer (`[Project Vault] Notification (${templateId})`). No sanitization, escaping, or content-neutralization is specified anywhere for this field before it reaches an outbound notification channel. This is a plausible injection surface (e.g., Slack mrkdwn/`@channel` abuse, HTML/markup injection in the email renderer) via a field explicitly designed to carry free-form incident narrative during a high-stress, fast-moving break-glass event — exactly the scenario where an attacker with compromised admin credentials would have both motive and opportunity to abuse it.

- **[medium]** AC-10's `dispatchDirectUserNotification({ orgId, userId: rotation.initiatedBy, ... })` does not account for `rotation.initiatedBy` being `NULL`. 5.1's schema defines `initiatedBy` with `onDelete: 'set null'` against `users.id` — if the initiating user's account is deleted between rotation-initiation and stale-detection, `initiatedBy` will be null by the time AC-9's job fires, and the direct-notification call receives `userId: null` with no documented fallback (skip silently? error the whole job transaction? notify nobody?).

- **[medium]** The new index `idx_rotations_status_initiated` on `rotations(status, initiated_at)` has no `org_id` leading column, yet AC-9's stale-detection job iterates per-org (`fetchAllOrgIds()` + `runOrgScopedJob`) and relies on RLS to filter each org's rows out of a single, tenant-agnostic index range. For a deployment with many orgs and/or many concurrently in-flight rotations, this means the same global index is effectively re-walked (and RLS-filtered row-by-row) once per org per job cycle — a scalability concern the story itself flags as unusual ("this is a global, org-scoped scan, not filtered by credential, unlike everything else in Epic 5 so far") without addressing the indexing implication.

- **[medium]** AC-21 explicitly requires an audit-write-failure (rollback) test only for the two most consequential *human-initiated* endpoints (`break-glass`, `abandon`). The two background jobs (AC-8 overlap-expiry, AC-9 stale-detection) use a manual, non-fail-closed audit-insert pattern copied from `check-failed-auth-threshold.ts`, but no AC or test specifies what happens if that manual audit insert throws inside a job's per-row/per-rotation transaction — whether the state-transition + lock-clearing writes cleanly roll back together with the audit write (leaving the row to be retried next cycle) or whether a partial-org failure could abort the entire job run partway through iterating other orgs/rows.

- **[medium]** AC-5 step 2 (`UPDATE rotations SET status = 'abandoned', version = version + 1 WHERE id = <found id>`) performs no optimistic-lock CAS check (`AND version = $observedVersion`), unlike every other rotation-state-transition in this story and its predecessors (5.2 AC-8's confirm/fail/retry/complete, this story's own AC-11/AC-12 resume/abandon). It relies solely on the preceding `SELECT ... FOR UPDATE` row lock for correctness. This is an inconsistent concurrency-control strategy within the same story, undocumented as a deliberate exception, and untested as such.

- **[medium]** AC-5 step 5 explicitly defers a real design decision to the implementer: "`supersedingRotationId: null` (filled in after insert, or written as a second audit row post-insert — **implementer's choice, document which**)." Leaving the shape of an audit-log payload ambiguous inside what is meant to be a precise, unambiguous acceptance criterion is a spec gap — two different implementers could produce materially different audit-log schemas for the same event, undermining the "6 new fail-closed/system audit events" claim's precision.

- **[medium]** AC-13/AC-14 modify `revealCurrentValue()` and `listVersionHistory()` — functions on the hot path for *every* credential value reveal in the system, not just rotation-touched credentials — and the story itself calls this "the single highest-risk change in this story." Despite that self-assessment, the story bundles this change into the same release/PR as the break-glass and stale-recovery features with no discussion of a feature flag, staged rollout, or independent deployability. A subtle regression in the added `isNull(credentialVersions.abandonedAt)` clause would silently break `GET .../value` org-wide, and the only stated mitigation is "add a regression test," not any operational safety net.

- **[low]** `POST .../rotations/break-glass` has no idempotency-key or double-submit protection. Since this endpoint's own premise is a panicked admin acting "in seconds" during an active incident (and no UI exists yet to add client-side debounce), a double-click/double-submit is a realistic scenario. Because a rotation in `break_glass_complete` status does not match AC-5's supersede filter (`in_progress`/`stale_recovery`), a second break-glass call would not be caught as "already handled" — it would simply create a second, independent break-glass rotation and consume a second `credential_versions` row silently.

- **[low]** ADR-5.3-06/CR7 already documents that break-glass/stale-recovery alerts route to org owners only by default, but no edge case addresses what happens if the org has **no** owner-role member at the time (e.g., an owner account was deactivated per Story 4.3 and ownership was never reassigned). In that scenario, `enqueueSecurityAlertNotification`'s default routing could resolve to zero recipients for the single highest-severity alert type in this story, with no fallback or explicit test coverage described.

- **[low]** AC-24's metrics list (`rotation_break_glass_total{outcome=...}`) has no counter/label distinguishing a break-glass call that superseded an existing active rotation (AC-5/CR6) from an ordinary one. Given CR6 identifies this as a real, expected interaction, and the first finding above shows the underlying invariant is weaker than assumed, operators have no dashboard signal for how often break-glass is actually colliding with in-flight rotations — exactly the kind of anomaly this feature's operational-visibility requirements (AC-24) are supposed to surface.

- **[low]** No metric or gauge tracks how often the newly-added `abandonedAt IS NULL` filter in `revealCurrentValue()`/`listVersionHistory()` actually changes the "current version" result in production (e.g., a counter for reveals where the highest version was excluded due to `abandonedAt`). Given the story's own pre-mortem calls this the highest-risk change in the story, a leading operational indicator here would materially help catch a regression early — but AC-24 does not request one.
