# Adversarial Review — Story 4.2: Organization User Management

- **Date:** 2026-07-01
- **Reviewed file:** `_bmad-output/implementation-artifacts/4-2-organization-user-management.md`
- **Reviewer:** bmad-review-adversarial-general

## Findings

- **[critical]** No org-role hierarchy check exists on any removal/role-change path. D4 only blocks self-modification (`userId === caller.userId`) and D5 only blocks removing the *sole owner of a project*. Nothing in AC-3 (`DELETE /org/users/:userId`) or AC-4 (`PUT .../role`) compares the **target's current org role** to the **caller's org role**. Since these routes only require `minimumRole: 'admin'`, an org `admin` can remove or otherwise act on the org `owner` (or another `admin`) with no rank check whatsoever — as long as the target isn't the "sole owner of a project." This is exactly the kind of privilege-escalation NFR-SEC10 ("no user may grant permissions exceeding their own role") appears intended to prevent, yet the story's own AC text never closes this hole.

- **[critical]** There is no "last org owner" protection anywhere in the story — D5 explicitly scopes the "last owner" guard to *projects* only ("this story must add it in three places," all three project-scoped). Combined with the previous finding, an org `admin` could remove the sole org `owner` from the organization entirely (the org owner cannot pre-emptively protect themselves since D4 unconditionally blocks *self*-removal, but nothing blocks an admin from removing *someone else* who happens to be the owner). This would leave the organization with no owner at all — an orphaned-governance state with no documented mitigation, guard, or even an acknowledged edge case.

- **[high]** AC-9's "extended project members page" is specified to source its accepted-member list entirely from `GET /api/v1/org/users`, which is gated `minimumRole: 'admin'` at the **org** level (D1). But D1's whole architectural premise — repeated throughout the story — is that project-scoped actions (viewing/managing one project's membership, per AC-5) must be usable by a project `admin`/`owner` who is only an org `member`. Such a user is fully authorized by AC-5 to remove a member from their own project, yet under AC-9's chosen data source they would get a `403` simply trying to fetch the member list needed to render that same page. This contradicts D1's stated design goal and would ship a broken page for exactly the persona the split-authorization model was designed to serve. Notably, AC-9's own test list ("non-admin redirect/access-denied on both pages") does not distinguish this case at all.

- **[high]** Several load-bearing patterns are described as "prior art to mirror" from Story 4.4 (`isProjectOwner || isOrgOwner` check, the "archive idempotency" conditional-update pattern, the in-handler ownership check at `4-4-project-archival.md:164-183`) — but the Prerequisites table states plainly that Story 4.4 is `ready-for-dev`, **not implemented**, and "no archival code exists yet." There is no actual shipped code to copy; only another story's own unimplemented spec text. This risks Task 6 (transfer-ownership) either stalling on a nonexistent reference implementation or independently inventing a pattern that later silently diverges from whatever 4.4 eventually ships.

- **[medium]** D8 defers extracting a shared "resolve my role in project X" helper for a third time, despite two prior stories (4.1, 4.4) already flagging the same deferral and this story adding two more copy-pasted call sites (AC-5, AC-6). This is authorization-critical logic gating who may remove members or transfer ownership — duplicating it three times increases the odds that a future fix (e.g., a missed org-scoping clause) is applied to only some copies, a real security risk disguised as ordinary tech debt.

- **[medium]** The persona journey narrative (line 35) is internally self-contradictory: it first states the removed user's session is "rejected on its next request with `403 account_deactivated`," then corrects itself mid-sentence to "actually `403 { code: 'insufficient_role' }`-equivalent 401/expired-session behavior," then simultaneously claims the effect happens "within 5 minutes (JWT TTL)" and "practically immediate." The exact error shape and the actual latency of session revocation are both left ambiguous/contradictory in the one place meant to sell the reader on the behavior.

- **[medium]** Audit `resourceId` conventions are inconsistent across the four new events: `org.user_removed` and `project.member_removed`/`project.member_role_changed` all key `resourceId` on the affected user's `userId` (even though the latter two are fundamentally project-scoped operations, with `projectId` demoted to a payload field), while `project.ownership_transferred` keys `resourceId` on `projectId` instead. This inconsistency makes "audit history for project X" queries harder to write uniformly across event types.

- **[medium]** AC-7 asserts a fail-closed audit contract for all four mutation routes but only requires "a forced-failure integration test for at least one of the four routes" — leaving the transaction-rollback behavior of the other three routes unverified prior to merge, despite the story treating fail-closed audit as a hard invariant ("no route in this story may complete a mutation without its audit row").

- **[medium]** The route-classification split in AC-7 is unexplained: `PUT .../role` and `DELETE /projects/:projectId/members/:userId` are classified `action: 'mutation'`, while `DELETE /org/users/:userId` and `POST /transfer-ownership` are classified `action: 'security-action'` — yet a role change directly alters access control just as much as removal or ownership transfer does. No rationale is given for treating these four equally-privileged endpoints at two different severity tiers.

- **[medium]** AC-9's decision to reuse the org-wide `GET /org/users` endpoint to populate a single project's member list means opening one project's members page pulls every user and every project-membership row across the entire org over the wire, then filters client-side. This compounds the story's own acknowledged "no pagination" limitation on that endpoint (AC-2) — a page whose data need is O(one project) is turned into an O(org size) fetch.

- **[medium]** AC-3's sole-owner-count query nests a `FOR UPDATE` inside a correlated `NOT EXISTS` subquery underneath an outer `FOR UPDATE OF pm` — a combination the story itself acknowledges has zero precedent in the codebase ("Grepping the entire codebase... returns zero matches") and explicitly steps outside the stated builder-first convention ("this codebase's convention is Drizzle query-builder calls, not raw SQL"). This is genuinely tricky locking logic underpinning a brand-new security guard, yet the AC only asks for functional test coverage of the guard's outcome, not verification that the lock scope is correct under concurrent load — a subtle over- or under-locking bug here would fail silently (either a spurious deadlock or a race that lets two sole-owner removals both succeed).

- **[low]** No notification (email/in-app) is specified anywhere in this story for any of the four privileged actions. A user who is removed from a project, removed from the org outright, demoted, or silently moved from owner to admin by someone else's ownership transfer has no way to learn of it except noticing broken access or reading an audit log they likely cannot see themselves. Given the codebase already has a notification-surface precedent from Epic 3, the total absence of any notification AC here is a plausible, unacknowledged gap.

- **[low]** `GET /api/v1/org/users` returns every org member's email plus their full cross-project role map, is explicitly exempted from audit logging (`writeAuditEvent: false`, justified only as "does not reveal secret values"), and carries a 60-requests/minute rate limit — effectively an unlogged, weakly-throttled org-topology/email-enumeration endpoint, a materially weaker audit posture than the mutation routes receive despite being a meaningful reconnaissance surface for a compromised admin session.

- **[low]** Response-shape asymmetry across the four mutation endpoints: AC-3 and AC-4 return `200` with a `{ data: ... }` envelope, AC-6 also returns `200`/`data`, but AC-5 returns a bare `204 No Content`. No rationale is offered for why one deletion endpoint needs a response body and the structurally similar one doesn't, which is a minor but avoidable API-consistency gap.

- **[low]** The optional performance index (AC-1) instructs developers to "verify `meta/_journal.json` and use the next free number... do NOT hardcode 0026" but offers no coordination mechanism if this optional step and some other in-flight story's migration are picked up concurrently — a minor process risk the story flags awareness of without actually resolving.

- **[low]** No AC addresses data-retention/compliance implications of the hard-delete approach for `org_memberships`/`project_memberships` rows (as opposed to a soft-delete or status flag, which Story 4.3 uses for the adjacent deactivation lifecycle) — historical membership facts become reconstructable only via the audit log's payload, which is never called out as an explicit design tradeoff.
