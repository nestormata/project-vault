# Adversarial Review

- **Date:** 2026-07-06
- **Reviewed file:** `_bmad-output/implementation-artifacts/9-5-operational-runbook-and-deployment-guide.md`
- **Reviewer:** bmad-review-adversarial-general

## Findings

- **[critical]** AC-18's emergency-revoke procedure documents the wrong response contract. Story 7.2's actual AC-20 ("Emergency Revocation — Atomic Revoke + Reissue") atomically revokes the old key **and** issues a replacement key in the same transaction, returning `{ "data": { "revokedKeyId": "...", "newKey": "pk_...", "newKeyId": "..." } }` — the one-time plaintext replacement key is already in that response. Story 9.5's AC-18 instead documents the response as revoke-only (`200 { "revokedAt": "2026-07-06T14:22:01.000Z" }`) and instructs the operator to perform a separate step "(2) issue a replacement key for the machine user's legitimate automated workflows" afterward. An operator following this runbook verbatim during a live compromise incident would not know to capture the `newKey` value the real endpoint already returns, and may attempt a nonexistent separate "create new key" call — exactly the kind of incident-response error this story exists to prevent, in the one scenario (suspected key compromise) the story's own persona journey names explicitly.

- **[high]** AC-9's positive example contradicts Story 9.1's actual AC-9 spec. Story 9.5 documents `POST .../admin/backups/:filename/restore` returning `202 { "jobId": "..." }` (implying an async, pollable job), but Story 9.1's AC-9 (the story 9.5 cites as its own source of truth) specifies a **synchronous** `200 { "restored": true, "filename": "...", "sealedAfterRestore": true }` response with no `jobId` field at all. An operator following AC-25's non-author documentation review would poll for a `jobId` that the real endpoint never returns.

- **[high]** AC-21 misstates the authorization model for `GET /api/v1/org/audit/verify`. Story 9.5 states this endpoint "requires Owner or explicit Audit role," but Story 8.1 — the cited source of truth — defines it as `allowedRoles: ['owner']` only, explicitly noting it is "the codebase's first-ever owner-only endpoint" (no other org role, including any "Audit" role, is admitted; the codebase's org roles are owner/admin/member/viewer with no audit-specific role anywhere). Documenting a nonexistent role as sufficient to reach a security-integrity-verification endpoint is a real authz-documentation defect, directly in the security-review scope this task called out (authn/authz).

- **[high]** D1's premise is independently confirmed and its consequence is understated: a grep of the current worktree finds zero hits for `is_platform_operator`, `requirePlatformOperator`, `platform_audit_events`, `migration-compatibility-check`, or `api-contract-tests` anywhere in `apps/`, `packages/`, or `scripts/`, and Stories 9.1, 9.2, 9.3, and 9.4 are all still `Status: ready-for-dev`. AC-25, the story's sole acceptance mechanism, is therefore currently unexecutable, and every concrete endpoint/response-shape claim in AC-2 through AC-21 is unverified against real running behavior — as this review's own findings (above) demonstrate, at least two of those claims are already wrong relative to their own cited source stories, despite being drafted "using the 9.1–9.4 story files... as the authoritative source." The safeguard the story proposes (D1 resolution point 2: mark unimplemented items with `<!-- PENDING: Story 9.x -->`) is not enforced by any AC or task — nothing greps for stray unmarked content before this story is considered complete.

- **[medium]** AC-8's example response format also contradicts Story 9.1's actual AC-8. Story 9.5 shows `GET /api/v1/admin/backups` returning a bare JSON array (`[{ "filename": ..., "verified": false }]`), but Story 9.1's AC-8 specifies the response is wrapped in an `{ "items": [...] }` envelope, and `verified` is a string enum (`"valid"` / `"unverified"`), not a boolean. Two independent shape errors in the same short example, on top of the AC-9 and AC-18 errors found above — this is a pattern of drift, not an isolated typo.

- **[medium]** AC-16's opening sentence conflates two structurally distinct alert types from Story 9.2: "a `key_custody_risk`-style tiered alert has fired for audit storage." `key_custody_risk` (master-key age/custody) and `audit_storage.warning`/`audit_storage.critical` (storage capacity, surfaced on `/ready` as `warnings: ["audit_storage_critical"]`) are separate `alertType` values in Story 9.2's own spec. A reader could reasonably conclude the alert literally named `key_custody_risk` is what fires for this scenario, undermining AC-1's own stated goal of exact-string alert/section names for reliable cross-referencing.

- **[medium]** No AC or task actually enforces D1's required interim safeguard. D1 says any not-yet-existing command/endpoint "must" be tagged with an inline `<!-- PENDING: Story 9.x -->` comment if this story is drafted before 9.1–9.4 merge, but none of AC-1 through AC-25 tests for the presence of such markers, and Task 10 ("cross-reference audit pass") only describes verifying content *after* 9.1–9.4 merge — there is no gate preventing this story from being merged in the interim with silent, unmarked, factually incorrect content, which is exactly the failure mode the AC-9/AC-8/AC-18/AC-21 findings above already exhibit.

- **[medium]** D7's disclosed gap (manual master-key rotation never clears the `key_rotated_at`-based custody alert) and the related Open Question 1 (no rotation-execution endpoint exists anywhere) are tracked only as prose across two story files (9.2's D8 and this story's D7/Open Questions), with no backlog ticket or owning story number. This is an acknowledged, security-relevant gap (a stale, misleading custody signal an operator is told to distrust) left entirely to informal cross-references rather than a scheduled, trackable follow-up.

- **[medium]** AC-17's break-glass sweep checklist (item 3) requires confirming "the rotation web UI's hardening fixes are functioning... per Story 8.5's hardening finding," but Story 8.5 is itself `Status: ready-for-dev`, not merged — a sixth blocking dependency (alongside 9.1–9.4 and 7.2) that D1's dependency analysis never names. D1 frames the blocking set as "Stories 9.1–9.4," but AC-17 and AC-18 pull in unmerged content from 7.2 and 8.5 as well; the story's own risk disclosure is incomplete about its true dependency surface.

- **[medium]** AC-11's missed-backup triage instructs the operator to "check the pg-boss job logs / structured operational logs... for a `backup.failed` alert or job-error log entry," but no AC or Dev Note anywhere in the document specifies the actual command or location for viewing these operational logs (vs. the container logs referenced elsewhere as `docker compose logs api`). For a runbook whose explicit goal is eliminating "tribal knowledge," this leaves exactly one such gap unaddressed — a non-author reviewer (AC-25) attempting this branch of AC-11 would need to guess or ask, which AC-25's own pass bar treats as a blocking failure.

- **[medium]** The Incident Response section (AC-15 through AC-18) never requires closing an incident by verifying it is captured in both of the two disjoint audit logs described in AC-21's own PJ9 disclosure (org-scoped `audit_log_entries` and platform-scoped `platform_audit_events`). AC-21 documents the two-log boundary as a monitoring/verification concern but none of AC-15–AC-18's "close the incident" steps cross-reference it as a mandatory pre-closure check, leaving a gap between "the boundary is disclosed" and "here is the step where an operator actually checks both logs before declaring an incident resolved."

- **[low]** AC-5/AC-15's escalation rule from "isolated unseal" to "graduate to the Incident Response flowchart" is qualified only by soft language ("recur within a short window," "multiple OOM-kills") with no numeric threshold, unlike the rest of the document's otherwise very precise, numbers-driven style (e.g., 24h backup-missed window, 95% audit-storage threshold, 365-day key-rotation default). This leaves a judgment call to the operator with no guardrail, in tension with the story's own "so that I can recover the vault reliably... without guesswork" framing.

- **[low]** AC-23's CVE/Trivy checklist item cites ".github/workflows/ci.yml's existing CI-enforced 30-day-max window" for `.trivyignore` expiry as an established fact, but — unlike AC-19's careful "verified against the actual code, not aspirational" discipline for the metrics list — this claim is not called out as independently verified against the actual CI file. Given this review already found two verified-against-source-story mismatches (AC-8, AC-9) and one verified-against-code mismatch is plausible here too, this claim should be spot-checked before merge rather than assumed correct by the same standard the story applies elsewhere.

- **[low]** No CI or lint mechanism is proposed to catch future drift between `docs/runbook.md` and the code/stories it describes, despite drift being this story's single largest disclosed risk (D5, D1, AC-24). The codebase already has precedent for static drift-guards (`check-migration-compatibility`, `check-rls-coverage`); relying entirely on each future story's own code-review pass to notice runbook staleness is a thin, easily-skipped control for a docs-only deliverable whose entire value proposition is accuracy.

- **[low]** AC-2's verification step checks `/health` and `/ready` for `{"status":"ready"}`, but the document never shows one single canonical, complete example of a fully healthy `/ready` response (only partial fragments elsewhere show `warnings` arrays in degraded states). A first-time reader following AC-2 to the letter has no single "this is what full success looks like" transcript distinct from the various partial examples scattered across the Incident Response and Monitoring sections.
