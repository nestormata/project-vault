# Adversarial Review: Story 8.1 — Tamper-Evident Audit Log with HMAC Integrity

- **Date:** 2026-07-04
- **Reviewed file:** `_bmad-output/implementation-artifacts/8-1-tamper-evident-audit-log-with-hmac-integrity.md`
- **Reviewer:** bmad-review-adversarial-general

## Findings

1. **[critical]** The AC-14 "dirty database" test fixture (a directly-inserted `audit_log_entries` row with `actor_type='human'` and `actor_token_id=null`) can **never be deleted**, because `audit_log_entries` is append-only — confirmed in `packages/db/src/test-helpers.ts:39-48`, where `withTestOrg()`'s cleanup deliberately swallows the `DELETE` failure on this table as expected, intentional behavior. The new `checkAuditActorTokenCoverage()` query in AC-13/AC-14 is **not org-scoped** — it counts gaps across the *entire* database, not just the fixture's test org. The story never specifies transaction-rollback/savepoint isolation for AC-14's fixture despite its own Dev Notes explicitly warning "`audit_log_entries` rows inserted by tests are never actually deleted." Unless the test database is fully recreated between every single test run (not stated as a hard guarantee anywhere in this story, only implied for the final `make ci` pass in Task 7.3), AC-14's own test will permanently poison AC-13's "clean database passes" assertion and the real CI gate for any developer or CI environment that reuses a Postgres instance across runs.

2. **[critical]** The story never establishes the actual threat model for tamper detection. AC-2's own edge-case note concedes that the *only* way to produce a "tampered" row in this schema is a privileged actor with raw SQL access, since UPDATE/DELETE are blocked by the trigger + grant REVOKE (D1). But the story never asks whether that same privileged actor could also reach the in-memory audit-signing key (derived via HKDF and cached at vault unseal per `key-service.ts`) — if so, they could INSERT a fully self-consistent forged row (correct `hmac`, correct `keyVersion`) that this endpoint would report as "passed," silently defeating the entire tamper-evidence guarantee the epic is built around. A story whose entire purpose is "prove to auditors the log has not been altered" should state explicitly what compromise scenarios it protects against (DB-only access) versus what it does not (API-process/key compromise) — this is currently unaddressed.

3. **[high]** AC-3's verify logic hard-fails any row whose `keyVersion` doesn't equal `currentAuditKeyVersion(tx)` — an unconditional, permanent check with no historical-key allowance. But D1/"Out of scope" explicitly defers key rotation to a future story specifically because "old key versions must be retained in a key_history store" so that old rows can still be verified after rotation. This story's own AC-3 contract (`hmacMatches AND row.keyVersion === currentKeyVersion`) leaves no interface seam for verifying against a retained historical key. Once rotation ships, every pre-rotation row will read as a permanent, false "tampering" failure under this story's logic unless a future story goes back and rewrites this story's core comparison — this is foreseeable technical debt baked directly into the AC text, not merely "deferred cleanly."

4. **[medium]** No mention anywhere of constant-time comparison for the HMAC check. `computeAuditHmac()` (`apps/api/src/modules/audit/write-entry.ts`) returns a plain hex string; the natural, unremarked-upon implementation of "recompute and compare" is an ordinary `===`/`!==` string comparison, which reintroduces the textbook timing side-channel that constant-time MAC comparison exists to prevent. Practical exploitability is limited (no interactive guess-and-check oracle is exposed), but the story brands this as the security-critical centerpiece of the epic while staying silent on this standard mitigation.

5. **[medium]** AC-6's oversized-range guard does a `COUNT(*)` pre-check before the real fetch/recompute, but `audit_log_entries` is continuously growing and append-only, and AC-12 explicitly allows concurrent writes during a verify call. Between the COUNT and the subsequent SELECT-and-recompute, new rows can land in the same range, so the actual number of rows recomputed can exceed `AUDIT_VERIFY_MAX_ROWS` by the time the real query executes. The DoS-prevention mechanism this story adds specifically to bound CPU-bound recomputation has a check-then-act race nowhere addressed in D4 or AC-6.

6. **[medium]** The `failed` array in the response schema (AC-1/AC-17) is unbounded even though `rowsChecked` is capped at 50,000. A systemic bug or bulk-tamper scenario could return tens of thousands of `failed` entries in a single JSON payload with no pagination, truncation, or "N more" summarization — a self-inflicted large-payload/latency problem for precisely the failure scenario this endpoint exists to surface.

7. **[medium]** Inclusive/exclusive boundary semantics for `from`/`to` are never specified in any AC. D4 recommends a compliance officer cover a full year by making "~4-5 calls across sub-ranges," but this only yields gapless, non-duplicated coverage if the boundary convention (e.g. `>= from AND < to`) is fixed and documented. As written, an auditor stitching adjacent 90-day windows together risks either a missed row or ambiguous double-counting exactly at each boundary timestamp — a real correctness gap for the story's own recommended usage pattern.

8. **[medium]** The route is registered with `writeAuditEvent: false` (Task 3.1) and no inline audit write is described anywhere for `verify.ts` — meaning calls to the integrity-verification endpoint are themselves never recorded in the audit trail. For a compliance-focused feature, "who ran an integrity check, and when" is itself often a fact worth auditing (due-diligence evidence, or detecting a compromised owner account probing what has/hasn't been detected). The story never discusses or justifies this omission.

9. **[medium]** D5's rationale for skipping `requireMfa` analogizes this endpoint to `GET /org/security-alerts`, but that route's `allowedRoles` is actually `['owner', 'admin']` (confirmed in `apps/api/src/modules/org/routes.ts:78-93`) — a materially broader-audience endpoint than this story's owner-only endpoint. Borrowing a broader-audience route's MFA-skip precedent to justify skipping MFA on the codebase's first-ever owner-only endpoint is a weaker analogy than the story presents; no independent justification is given for why a non-MFA-enrolled owner mid-grace-period should be trusted with this specific capability.

10. **[medium]** AC-11 justifies the `20/min` rate limit as matching "the sensitive-mutation rate-limit tier used elsewhere," but its own cited read-only precedent (`GET /org/security-alerts`, used elsewhere in the story to justify authz/MFA decisions) has no explicit rate-limit override in the current code — the `max: 20` figure is actually drawn from mutating routes like `POST /users/:userId/deactivate` and `DELETE /users/:userId/sessions`. The story switches which precedent it cites depending on which decision it's justifying, without acknowledging the switch.

11. **[medium]** The backfill/coverage check (AC-13/AC-14) is scoped only to `actor_type = 'human'`. The story's own Epic Cross-Story Context table notes that Stories 7.1/7.2 will introduce `actor_type = 'machine_user'` rows into the same table with the same nullable `actor_token_id` column, yet the coverage check's WHERE clause has no analogous guard for that population. If machine-user actions ever need to be attributable back to a human for GDPR purposes (e.g., an API key acting on a specific owner's behalf), this check silently doesn't cover that case — an implicit scoping decision that isn't explicitly justified as sufficient.

12. **[low]** AC-9's instruction — "add one explicit test... or confirm an equivalent assertion already exists" — is phrased as a soft, either/or instruction rather than a hard requirement. A developer under time pressure could read "confirm an equivalent assertion already exists" as license to add nothing, defeating the AC's stated purpose of guarding against an accidental future `EXCLUDED_TABLES` regression via a *named* failure.

13. **[low]** The Product Surface Contract section concedes a known, unresolved planning gap (no story anywhere scopes the audit-log UI that the PRD's Dana persona journey and epics.md's AC-E8c both assume exists) and defers it to "Epic 8 sprint planning" rather than resolving it or obtaining explicit sign-off that an API-only increment is acceptable. The story proceeds to "ready-for-dev" status without that resolution.

14. **[low]** AC-8 bans the words "HMAC," "hash," and "cryptographic" from the runtime response summary string, and this same response is also the literal example embedded in the auto-generated OpenAPI spec (AC-17) served to third-party auditor/API tooling via `/api/v1/docs`. The tension between "comprehensible to a non-technical compliance officer" and "precise enough for a technical API consumer building against the documented contract" is never examined.

15. **[low]** No AC addresses the zero-width range case (`from === to`). It's unclear whether this is treated as `invalid_range` (only `to < from` is explicitly listed as invalid), a valid empty-range result, or something dependent on the inclusive/exclusive semantics flagged in finding 7 — a missing edge case.

16. **[low]** The AC-6 table's closing statement — "no case reaches the HMAC-recompute code path" — is imprecise: the oversized-row-count case explicitly requires running a `COUNT(*)` query against the database before rejecting the request, so that specific validation path does touch the database, just not the HMAC recompute step. Minor wording inaccuracy, but the kind of imprecision that tends to get copy-pasted into implementation comments verbatim.
