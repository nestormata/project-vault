# Adversarial Review — Story 6.4

**Date:** 2026-07-06
**Reviewed file:** `_bmad-output/implementation-artifacts/6-4-epic-6-completion-monitored-asset-management-ui-and-technical-debt.md`
**Reviewer:** bmad-review-adversarial-general

## Findings

- **[high] Story is an epic wearing a story costume.** Thirteen tasks spanning four independent resource-type CRUD UIs (services, certificates, domains, service endpoints), an alerts panel with snooze/dismiss, a dashboard tile, a copy fix, and an unrelated shared-package type refactor (`AuditEvent` consolidation) are bundled into one "story." This is a major delivery risk: any single blocked task (e.g., the audit-events refactor surfacing an unexpected typecheck break) can stall the entire closure story, and there is no plan for partial/incremental delivery or splitting into sub-stories/PRs.

- **[high] AC-J's regression test scope omits `apps/api` despite citing an `apps/api` call site as evidence.** AC-J2 explicitly names `apps/api/src/lib/audit-or-fail-closed.ts:26` as a real consumer of the audit-event registry, yet Task 11 and AC-J2/J3's "safety net" only require running `packages/shared` and `packages/db` test suites (plus root `typecheck`). If the consolidation subtly changes which strings are assignable where, `apps/api`'s own test suite is never explicitly required to run as a regression gate, despite being the one runtime package cited as proof of "no `AuditEventType` importers outside the file."

- **[high] No guard against SvelteKit's classic dynamic-route component-reuse bug across near-identical resource pages.** This story adds ~4 structurally identical `[projectId]/[resourceId]` detail/edit routes (services, certificates, domains, service-endpoints) plus list pages under `[projectId]`. SvelteKit reuses page components across param changes unless the developer explicitly invalidates/re-keys on navigation. Nothing in the story (ACs, Dev Notes, or Tasks) mentions this risk, even though it is the single most common cause of "stale data from the previous project/resource briefly rendered" bugs in exactly this route shape.

- **[medium] AC-A2's "exactly one project in the org" branch is an unverified assumption about existing load-function data.** The story's own stated rigor elsewhere ("verify directly in schema.ts before assuming") is not applied here: nothing confirms the `/health` page's current load function already has the org's project count/single-project-id available, or whether this requires a load-function change beyond copy/link edits — contradicting the story's claim that this is "a small, additive change."

- **[medium] Missing failure-mode: HTTP 429 / rate limiting is never addressed in any create/update AC.** The story's own References section states `routes.ts` defines "rate limits," and Sam's persona journey has the user registering four resources back-to-back — a plausible rate-limit trigger — yet AC-B3/C1/D1/E3's otherwise-exhaustive failure-mode lists (validation, archived-project 410, cross-org 404, cap-reached 422, SSRF 422) never mention a 429 response or how the UI should surface it.

- **[medium] AC-B4's "must not display stale cached alert-lead-day state" clause is untestable as written.** The same sentence says the edit page "does not need to display `notifiedLeadDays` itself" and then requires it "must not display stale cached alert-lead-day state after a `renewalDate` edit" — if the field is never displayed, there is no observable UI behavior to assert in a test. This reads as a requirement invented to sound thorough without a concrete, checkable outcome.

- **[medium] Unbounded services/certificates/domains lists are accepted without evidence they'll stay small.** Background explicitly forbids pagination for these three resources because "no equivalent cap exists or is needed... at this story's scope," but no server-side cap exists at all (unlike endpoints' 25-cap) — a single project could accumulate thousands of certificate rows over the product's lifetime with no story-level mitigation, only an assumption of smallness.

- **[medium] The two-step delete/dismiss confirm component is explicitly "optional but recommended," directly inviting the exact anti-pattern the Dev Notes warn against.** Dev Notes calls out the risk of five near-identical bespoke implementations, but Project Structure Notes lists `ConfirmDeleteButton.svelte` as "optional." No AC forces convergence on one shared component, so test coverage and behavior consistency across the five delete/dismiss sites are left to individual developer discretion.

- **[medium] No rollback/feature-flag/incremental-shipping strategy for a story this large.** Given the scope called out above, there's no mention of whether Task 11 (AuditEvent consolidation) could/should land as its own PR to de-risk the rest, despite Dev Notes itself noting it has zero functional dependency on Tasks 1–10.

- **[medium] Task 12's mobile-viewport check is a manual, code-review-verified step with no AC number and no CI gate.** "Manual mobile-viewport check... code-review-verified rather than a captured screenshot" is easy to silently skip in a 13-task story, and isn't tied to any specific AC, so its omission wouldn't fail any test or block any AC sign-off.

- **[medium] Stale-role edge case is only handled for alert-dismiss (AC-F3), not uniformly across all other mutation ACs.** AC-F3 explicitly discusses a mid-session role downgrade causing a 403 that must be caught gracefully. AC-B3/B4/B5/C1/D1/E3/E4/E5 never mention the equivalent case for create/edit/delete on services/certificates/domains/endpoints, despite it being the identical failure shape.

- **[low] Persona-journey text contradicts AC-E3's required fields.** Sam's journey claims "Sam did not have to configure anything beyond the URL" when registering the endpoint, but AC-E3 requires `name` as well (it's in the example payload and is a mutation the UI must collect) — a minor but real inconsistency between narrative and AC.

- **[low] "Zero new backend routes, zero new database tables, zero new migrations" framing understates that Task 11 touches a file consumed at runtime by `apps/api`.** `packages/shared/src/constants/audit-events.ts` is imported by API code (per the story's own citation of `audit-or-fail-closed.ts`). Calling this purely a UI-only closure story is slightly misleading; it is UI-only plus a type-level refactor of shared code with real (if type-only) backend-adjacent blast radius.

- **[low] AC-A1's "the sub-nav itself still renders (it's static)" claim isn't reconciled with the existing cross-org 404 pattern.** If a project 404s, does the whole page (including nav) get replaced by the `AccessNotice`-style notice, or does the nav persist alongside it? The AC asserts both "sub-nav still renders" and "page shows the not-found notice" without clarifying whether these coexist, which is exactly the kind of ambiguity that produces divergent implementations.

- **[low] No mention of output encoding / stored-XSS consideration for free-text fields now rendered across many new list/detail views** (`name`, `domain`, `url`, `failureReason`, alert `severity`/`alertType`). Given the "also_consider" ask explicitly includes injection/input validation, the story should at minimum note that these are user-controlled strings rendered in multiple new surfaces, even if Svelte's default auto-escaping is expected to handle it — the story is silent on this entirely.

- **[low] No explicit statement that the frontend must source `projectId` from the authenticated route/session context rather than any client-editable form field**, for the new API wrapper calls. Backend authorization is the real gate (acknowledged elsewhere), but for a story this focused on mirroring security precedent (role-gating helpers, redaction handling), the absence of even one line on this is notable, especially since forms will be constructing bodies as literal objects.

- **[low] No mention of concurrent-edit/last-write-wins behavior** for the edit forms (e.g., two `member`+ users editing the same certificate's `expiresAt` simultaneously). Likely low real-world impact given team sizes, but zero acknowledgment anywhere in ACs or Dev Notes.

- **[low] DNS-rebinding / TOCTOU SSRF risk for recurring endpoint health checks is never acknowledged, even as an accepted residual risk.** AC-E3 is unusually detailed about the create-time SSRF rejection (`url_not_allowed`), correctly noting DNS resolution can't happen client-side — but the story never notes that a URL validated safe at creation time could resolve to a private address by the time the (recurring, server-side) health-check tick runs. This is a backend concern and thus arguably out of this story's scope, but the story's silence is conspicuous given how much space it spends on the create-time SSRF case specifically.

- **[low] AC-J1's refactor instructions leave the canonical type name ambiguous** ("`AuditEventType` should become `AuthAuditEventType` directly... or `AuthAuditEventType` should be renamed to `AuditEventType`... developer's choice on naming") for a type that is by the story's own admission consumed indirectly across the audit-logging surface. Leaving the final name undecided in the story itself risks inconsistent naming choices slipping through review, since "developer's choice" for a public shared-package type name is a bigger decision than the story treats it as.

- **[low] No CSRF-token or double-submit-protection discussion for the ~15 new state-changing UI-triggered requests**, though this is likely an inherited, already-solved app-wide concern rather than new risk introduced by this story — still, given the explicit "also_consider" security prompt, its complete absence from the document is worth flagging.
