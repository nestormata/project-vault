## Findings

1. **[critical]** The story's top-of-file comment, D8, and AC-T3 all claim "`deferred-work.md`'s 'Web UI gaps' table gains a row for this scope as part of this story's own documentation tasks (Task 9) — the worktree this story was authored in branched before the retro's own `deferred-work.md` edit was committed to `main`, so this story adds it directly rather than assuming it already exists," and "no `deferred-work.md` row existed until the Epic 9 retro." This is factually false for the actual state of this worktree: direct inspection of `_bmad-output/implementation-artifacts/deferred-work.md` (line 73, "Web UI gaps" table, under "Deferred from: Epic 2 closure retrospective") shows the row **already exists right now, in this worktree, on this branch** — "Platform operations admin UI (backup/restore admin screen, system settings SMTP/policy/schedule UI, version-upgrade trigger/API-docs browser within app shell, Platform Operator Audit Log distinct from per-org log per PJ9, resource-usage dashboard with threshold indicators) | 9.1/9.2/9.3/9.4/9.6 | ... | **Resolved 2026-07-08 (Epic 9 retro):** scheduled as `9-7-epic-9-completion-platform-operations-web-ui` (backlog, `sprint-status.yaml`) — `epic-9` held `in-progress` until it lands." The row is not a future merge risk to "reconcile" — it is present today. A dev agent trusting the story's "add it directly, it doesn't exist yet" framing could either duplicate the row or become confused when `git status`/a direct read shows it already present.

2. **[critical]** AC-G4's edge case directly contradicts another section of this same story file, and the actual shipped code. AC-G4 states: "confirm via the endpoint inventory (D2's table) that `GET /admin/settings` does **not** require MFA (only the `PUT` does) — the settings page must still load and display current values for an un-enrolled operator; only the *save* action is blocked. Do not gate the entire page behind MFA when only the mutation requires it." But the story's own "Full endpoint inventory" table (Background section) lists `GET /settings`'s MFA column as **`yes`**, not "no" — and direct inspection of `apps/api/src/modules/platform-admin/settings-routes.ts` (lines 59-80) confirms the `GET /settings` handler's `security` block sets `requireMfa: true`, identical to the `PUT` handler. The same pattern holds for `GET /orgs` (`orgs-routes.ts` lines 88-105, `requireMfa: true`) and is documented in the endpoint table as `yes` for `GET /resource-usage`, `GET /audit/events`, and `GET /audit/verify` too. Building AC-G4 exactly as written will produce a settings page that assumes it loads fine for an MFA-unenrolled platform operator, when in reality the page's own `load` call to `GET /admin/settings` will itself 403 with `mfa_required` before any settings ever render.

3. **[high]** Following directly from Finding 2: AC Groups H (Organizations), I (Resource Usage), K (Platform Audit Log search), and L (integrity verify) never specify any handling for an `mfa_required` 403 on their page-load `GET` request, even though the story's own endpoint table documents `GET /orgs`, `GET /resource-usage`, `GET /audit/events`, and `GET /audit/verify` as all requiring MFA (confirmed for `GET /orgs` by direct code inspection). AC-G4 is the *only* AC in the entire 57-AC document that addresses an MFA-gated page load, and (per Finding 2) it describes the wrong endpoint behavior. Four of the seven new pages therefore ship with zero specified behavior for a realistic, common first-visit scenario: an un-enrolled platform operator navigating to `/platform/settings/orgs`, `/platform/settings/resource-usage`, or `/platform/audit`.

4. **[high]** AC-N1 is assigned to two different Tasks with no stated precedence: Task 3's header reads "**Task 3 — Backend: new `GET /platform/maintenance-mode` status route (D2.4)** (AC: M1, N1)" and Task 10's header reads "**Task 10 — Cross-cutting hardening** (AC: A4, N1-N3, O1-O2)." Both formally claim ownership of AC-N1. Since Task 3 (early, backend-only) and Task 10 (late, cross-cutting) are far apart in the task sequence, this creates a real risk that the RLS/EXCLUDED_TABLES test AC-N1 requires is either duplicated or — more likely in practice — skipped by both tasks, each assuming the other covers it.

5. **[high]** `sprint-status.yaml`'s two independent changelog mechanisms disagree about the same session's events. The file-level `#` header comment (line 2) explicitly logs: "...8-8 added (backlog); 8-8 story file created via bmad-create-story (backlog -> ready-for-dev) — 16 ACs...; 9-7 story file created via bmad-create-story (backlog -> ready-for-dev) — 57 ACs across 16 groups (A-O, T) closing the Epic 9 platform-operations web UI gap." The actual YAML `last_updated:` field's own trailing comment (line 38), which otherwise narrates the same chronological session in detail, ends at "...9-7 story file created (backlog -> ready-for-dev)" and never mentions `8-8` at all — despite `8-8` supposedly being created and moved to `ready-for-dev` in the very same session this same field's comment is describing. The file's two self-maintained changelogs are internally inconsistent about whether 8-8's creation was tracked.

6. **[medium]** AC-A4's required new-test work ("Integration test incl. `403` for non-platform-operator (AC-A4)") is written only inside Task 3's subtask bullet list; Task 3's own header omits A4 from its AC list ("AC: M1, N1"). Task 10's header, however, formally claims AC-A4 as one of its own ACs ("AC: A4, N1-N3, O1-O2"). This is the same ownership-mapping defect as Finding 4, affecting a second AC.

7. **[medium]** AC-A2 ("Platform Admin nav item renders only for platform operators") is implemented inside Task 4's own subtask bullets ("`AuthUser` type gains `isPlatformOperator`; nav-model gains conditional item (AC-A2)"), but Task 4's header lists only "(AC: A3, N3)" — omitting A2 from the header-level AC tracking entirely. Combined with Findings 4 and 6, the Tasks/Subtasks section has a recurring, systemic drift between header-level AC ownership and body-level AC references across at least three different ACs (A2, A4, N1), which undermines the header list's usefulness as a completion checklist.

8. **[medium]** `8-8-story-8-6-retroactive-adversarial-review` is filed under the "# Epic 9: Platform Operations, API & Self-Hosting" section of `sprint-status.yaml` (appearing after `epic-9-retrospective`), even though 8-8 has no substantive connection to Epic 9 — it is a retroactive adversarial review of Story 8-6, which itself closed an Epic 7 gap and is filed under the Epic 8 section (line 121). Every other cross-epic "hardening bucket" story in this file (8-5 closing Epic 5 debt, 8-6 closing Epic 7 debt, 8-7 closing Epic 8's own gap) is grouped under the epic section matching its own numeric prefix; 8-8 breaks that established convention by sitting under `epic-9` instead of `epic-8`. `epic-9`'s own explanatory comment (line 126) never mentions 8-8 at all, despite 8-8 being nested inside its block — reinforcing that this placement is inconsistent with the rest of the file's own organizing logic, not an intentional new pattern.

9. **[medium]** The story repeatedly frames itself as the sole remaining gate on Epic 9's closure — e.g. the top comment: "`epic-9` stays `in-progress` until this story lands (Product Surface Contract G2 gate)," and the Story section: "Epic 9 can close without leaving its own operator-facing UI gap untracked for a fifth epic running." Neither statement acknowledges that `sprint-status.yaml`'s own `epic-9:` comment states 9-4 and 9-6 are "also still in review (partial retro)" — two entirely independent stories, unrelated to this story's G2 scope, that must also reach `done` before `epic-9` can close. A reader relying solely on this story file could reasonably (and incorrectly) conclude that merging 9-7 is sufficient to flip `epic-9` to `done`.

10. **[medium]** The already-existing `deferred-work.md` "Web UI gaps" row (see Finding 1) is itself already stale relative to `sprint-status.yaml`: it reads "scheduled as `9-7-...` (backlog, `sprint-status.yaml`)," but `sprint-status.yaml` has already moved `9-7` to `ready-for-dev`, not `backlog`. AC-T3 instructs updating the resolution text from "scheduled as" to "shipped as" but says nothing about the "(backlog, `sprint-status.yaml`)" parenthetical also being inaccurate — a dev agent following AC-T3 literally could leave this stale sub-clause in the merged row.

11. **[medium]** The story documents the MFA asymmetry between backup actions (trigger/restore/validate — no MFA) and settings/orgs/resource-usage/audit/maintenance-mode (MFA required) as "pre-existing and out of scope to change" (Background section) without escalating it anywhere as tracked risk. Restore is the single most destructive action this entire story exposes in a UI for the first time (irreversibly overwrites live data and forcibly reseals the vault), yet it requires no MFA step, while a routine read like `GET /admin/settings` does. Unlike TD9-2 (also "documented, not fixed," per D8/AC-T2), this asymmetry gets no equivalent `deferred-work.md` entry — it is simply asserted as acceptable.

12. **[medium]** No AC specifies rate-limiting behavior for the restore action or the maintenance-mode activate/deactivate toggle. AC-D3 explicitly covers `429` handling for backup *trigger* (`rateLimit: {max: 10, timeWindowMs: 60_000}`), but restore (AC group E) and maintenance-mode toggle (AC group M) — arguably more sensitive actions — have no equivalent AC. If these routes are in fact rate-limited server-side, the UI has no specified `429` copy for them (unlike D3's exact wording for trigger); if they are not rate-limited, this story provides no acceptance criterion that would ever catch that gap.

13. **[low]** The Persona journey stub's phrasing — "She then visits **Settings**, updates the SMTP `from` address, and **separately visits Organizations** to provision a second org for a new client team" — reads as though Organizations is a parallel, sibling destination to Settings. This contradicts D1's explicit routing decision that Organizations is nested *under* Settings at `/platform/settings/orgs`, not a separate top-level area, and is exactly the kind of narrative AC-B1 itself warns against ("do not add six tiles when the route structure specifies four"). Illustrative prose contradicting the story's own prescriptive routing decision is a minor but avoidable internal-consistency slip in a document this heavily cross-referenced.

14. **[low]** AC-E4 gives a worked "Example (positive)" only for case (a), the `restore_in_progress` conflict; case (b), the `backup_in_progress` conflict, has no equivalent worked example (only a tangential edge-case note about message-attribution imprecision inherited from 9.6). Given how consistently every other AC in this document pairs each documented branch with its own explicit example, the asymmetric example coverage here is a minor completeness gap.

15. **[low]** AC-C1 lists "Key Version" as one of the six `DataTable` columns for the backups list, but no AC or example anywhere in the document ever populates or illustrates a `keyVersion` value (confirmed present as `keyVersion: z.number().nullable()` in `apps/api/src/modules/backup/schema.ts`). Every other column named in AC-C1 (Filename, Started, Status, Size, Verified) gets a concrete example value across AC-C1's three examples; Key Version does not, leaving its display format (e.g. how a `null` key version should render) fully unspecified.
