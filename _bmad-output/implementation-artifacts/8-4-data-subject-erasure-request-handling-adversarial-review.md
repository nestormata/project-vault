# Adversarial Review: Story 8.4 — Data Subject Erasure Request Handling

- **Date:** 2026-07-05
- **Reviewed file:** `_bmad-output/implementation-artifacts/8-4-data-subject-erasure-request-handling.md`
- **Reviewer:** bmad-review-adversarial-general

## Findings

- **[CRITICAL] `sessions.ip_address` and `sessions.user_agent` are never erased or even inventoried.** Confirmed in `packages/db/src/schema/sessions.ts`: `sessions` rows have real `ip_address` and `user_agent` text columns. This story's erasure execution only *revokes* sessions via `revokeAllUserSessionsInOrg` (sets `revoked_at`/`session_version`) — it never scrubs these columns, and neither AC-1's PII inventory response nor AC-14's compliance report (`piiRemoved`/`piiRetained`) lists `sessions` at all. IP address is directly-identifying PII; it survives "erasure" indefinitely, completely unaccounted for in the very artifact (the compliance report) meant to prove to a regulator that PII was removed or its retention justified. This undermines the story's core purpose, not a peripheral detail.

- **[HIGH] `original_email_hash` uses a bare, unsalted `SHA-256(normalizeEmail(email))` (D6) — trivially crackable/enumerable.** Email addresses are low-entropy and highly guessable, especially once an attacker (or an over-curious admin with read access to `data_erasure_requests`) knows the org's email domain conventions. A bare hash lets anyone with table access confirm whether a specific individual's email was ever subject to an erasure request by brute-forcing candidate addresses — a re-identification vector GDPR pseudonymization guidance specifically warns against. This should be a keyed hash (HMAC-SHA256 with a server-side secret/pepper), not a plain digest, matching the caution the story itself takes elsewhere (e.g., "never reversible to the original email" language for `users.email`'s replacement value) but fails to apply consistently to the hash.

- **[HIGH] D6's re-invite block patches only `invitations/routes.ts`; a separate self-service `/register` endpoint is left completely unguarded.** Confirmed via `apps/api/src/modules/auth/routes.ts` (`POST /register` → `registerUser`) that direct account self-registration exists independent of the project-invitation flow. Once erasure execution overwrites `users.email` to `erased_<hash>@erased.invalid`, the original email becomes free again against `users.email`'s unique constraint (`packages/db/src/schema/users.ts`). Nothing in this story prevents anyone from immediately self-registering with the erased person's original email via `/register`, which the story never scopes out or even acknowledges as a gap — an inconsistency given how carefully D6 reasons about the invitation path specifically.

- **[HIGH] D2's cross-org guard can make erasure permanently, structurally impossible for a data subject — with no defined remediation path.** The resolution requires the target user's *only* `org_memberships` row (any status) to belong to the requesting org; otherwise execution is blocked with 409 forever, and the story explicitly declares multi-org coordination "out of scope for this story — an ops/process concern." No API, workflow, or runbook is specified for that coordination, and there is no handling for the case where the other org is dissolved, abandoned, or its admins are unreachable. For a user who was ever briefly a member of a second org (even one long deactivated), this story provides no way to ever honor their Article 17 request through the product — a legal/compliance risk understated as a mere "documented limitation."

- **[MEDIUM] No concurrency guard specified for erasure-*request creation* (AC-4), unlike execution's explicit D9 compare-and-set.** Two admins calling `POST .../erasure-request` for the same user at nearly the same time could both pass the "no existing pending row" read before either write commits, producing two `data_erasure_requests` rows for one user. Task 1.1's schema fields list no partial unique index (e.g., on `user_id` where `status IN ('pending','in_progress')`) to prevent this, and AC-4 only exercises the sequential case, not the race.

- **[MEDIUM] No audit event is specified for erasure-*request creation* itself.** D10/AC-12 cover the execution audit event in detail, but creating a request — which exposes a full PII inventory for a named user and flags them for eventual erasure — has no corresponding `audit_log_entries` write anywhere in the ACs. If a request is created but never executed (or executed by a different admin later), there is no audit trail of who initiated the process or when, a gap for a compliance-sensitive action in a story otherwise obsessive about audit completeness.

- **[MEDIUM] `requireMfa: true` applied uniformly to all three routes (AC-21) without addressing the "owner has no MFA enrolled" case.** The story asserts "no cost to requiring it," but doesn't specify what the gate does for an owner account with zero MFA enrollment — hard-block vs. forced-enrollment redirect. For a single-owner org facing a real, time-boxed regulatory erasure deadline, an unaddressed MFA-enrollment lockout on the one authorized executor is an operational risk this story should resolve explicitly rather than assume away.

- **[MEDIUM] Pseudonymization alias collisions are unguarded.** `pseudonymizeUserIdentityToken` (D3) generates `user_<8-random-lowercase-alphanumeric-chars>` with no mentioned uniqueness check, retry-on-collision logic, or DB unique constraint on `display_name`. Low probability is not zero probability, and a collision would surface as two different pseudonymized users sharing an identical audit-log display alias — actively misleading in any audit search UI (Story 8.2) that resolves `actor_display_name`, and silent since nothing would detect it.

- **[MEDIUM] Fixed, identical password-hash sentinel across all erased users creates an easy fingerprinting surface.** Dev Notes call for "a fixed, well-documented constant" reused for every erased account's `password_hash`. Anyone with DB read access can enumerate every erased user with a single `WHERE password_hash = :sentinel` query. This doesn't leak anything not already derivable from `data_erasure_requests`, but it's an unforced, easily avoided fingerprinting surface (a per-row random non-matching value would avoid it) that the story never weighs as a tradeoff.

- **[MEDIUM] AC-20's rate limit for the *execute* endpoint borrows the routine-admin-action precedent (max 20/min) without weighing that execute is uniquely irreversible, owner-only, and rare.** The story explicitly forecloses picking a different, tighter tier "without citing precedent," which means a clearly higher-stakes, one-way-door action inherits a limit sized for reversible, frequent admin operations — 20 permanent erasures per minute per caller is a strange ceiling to accept by default for an action this destructive.

- **[LOW] AC-6's confirmation gate is under-specified at the validation layer.** Only `{confirm: true}` vs. `{confirm: false}`/omitted are discussed; there's no mention of schema-level rejection of non-boolean truthy values (e.g., string `"true"`, `1`) or unexpected extra fields in the body, leaving the exact input-validation contract implicit.

- **[LOW] Timing side-channel on the tenant-isolation 404 (AC-3) is unaddressed.** The story is careful that "user doesn't exist" and "user exists in a different org" both return `404 user_not_found` to avoid an existence leak in the response body, mirroring AC-19's explicit 404-vs-403 discipline — but never considers whether the two code paths (no row found at all vs. row found then org-membership join fails) take measurably different time, which is the same class of leak via a different channel.

- **[LOW] AC-14's `piiRetained` array mixes a real table name with an unresolved, slash-joined placeholder.** The entry `"table": "rotation history / project_invitations.invitedBy"` is not a real, singular schema identifier the way every other row in that array is (`audit_log_entries`, `org_memberships`); no "rotation history" table is named anywhere else in the story or its references, so this line cannot be implemented as literally specified without first resolving what table(s) it actually refers to.

- **[LOW] "8 atomic steps" (AC-5's heading) conflates PII-erasure operations with bookkeeping/audit operations, slightly overstating erasure scope.** Of the 8 numbered steps, only steps 1–5 actually erase/mutate PII-bearing data; step 6 reuses existing session-revocation infrastructure, and steps 7–8 are status bookkeeping and audit-event writing. Not incorrect, but the framing invites a literal-minded implementer to treat all 8 as equivalent erasure operations when they're not, which matters for how "erasure completeness" gets tested.
