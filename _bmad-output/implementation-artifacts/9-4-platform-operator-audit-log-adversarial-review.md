# Adversarial Review — Story 9.4: Platform Operator Audit Log

- **Date:** 2026-07-06
- **Reviewed file:** `_bmad-output/implementation-artifacts/9-4-platform-operator-audit-log.md`
- **Reviewer:** bmad-review-adversarial-general
- **Also-consider scope:** inconsistencies/contradictions between sections; gaps in acceptance criteria or missing edge cases; technical debt implied by the proposed approach; implementation and delivery risks; security risks (authn/authz, RLS/tenant isolation, secrets handling, injection, input validation, audit/logging gaps).

## Findings

1. **[critical] No RLS defined for the two new support tables (`platform_audit_maintenance_state`, `platform_audit_pending_entries`).** D4/AC-3 give `platform_audit_events` a session-variable-gated RLS policy as explicit defense-in-depth "because this codebase's stated architectural principle is RLS enforced at database level." But Task 1.4 only says the two support tables get added to `check-rls-coverage.ts`'s `EXCLUDED_TABLES` "with an explanatory comment" — no RLS policy is specified for either. `platform_audit_pending_entries.intendedFields` can hold a full snapshot of a queued write's payload (potentially before final redaction has been confirmed to succeed, see finding 7), meaning the table most likely to transiently hold sensitive data is the one left without the same DB-level hardening the story insists on elsewhere.

2. **[critical] The story never models the platform operator as an adversary.** Every AC's threat model is "org admin tries to read/tamper" (AC-21) or "generic external tamperer" (AC-2/AC-3), but the Story statement's entire justification is accountability *for platform operators*. D6 explicitly concedes "an attacker with API-process code execution or access to the in-memory signing key could forge a self-consistent row," yet no AC tests, hardens, or even discusses a malicious/compromised operator falsifying their own audit trail — arguably the single most important threat this log exists to counter, left completely untested.

3. **[high] `writePlatformAuditEntryOrFailClosed()` takes `operatorId` as a caller-supplied parameter rather than deriving it from the authenticated request context internally**, and no AC validates it against the authenticated session before persisting to an immutable, compliance-grade log. A caller bug (wrong variable, copy-paste from another handler) could permanently misattribute a privileged action with no correction mechanism, since the table is append-only by design.

4. **[high] Status (`ready-for-dev`) contradicts D1's own instruction** ("Do not start implementation of this story until 9.1's and 9.2's PRs have actually merged"), and the story does not distinguish that Tasks 1–4 (schema, HMAC write path, maintenance mode, new routes) have no technical dependency on 9.1/9.2 code at all — only Task 5 (retrofit) and Task 6.3 (extend `audit-storage-check.ts`) do. As written, a dev agent picking this up faces contradictory guidance: the story is simultaneously "ready" and "blocked entirely," with no task-level sequencing to resolve the ambiguity.

5. **[high] Maintenance-mode's fail-closed bypass is scoped far more broadly than the emergency it claims to address.** D8/AC-15 catch *any* write failure ("HMAC computation failure, DB constraint violation, sealed-vault `VaultSealedError`") while maintenance mode is active, and silently queue it rather than restricting the bypass to specifically "the audit log storage is unavailable." A genuine application bug (e.g., a malformed payload triggering a constraint violation) occurring during a maintenance window would be silently swallowed into `platform_audit_pending_entries` instead of surfacing as a defect — undermining the very fail-closed invariant this story is designed to guarantee.

6. **[high] Redaction of `FORBIDDEN_AUDIT_KEYS` (AC-6) is asymmetric and under-specified.** The story states it should "fail loud in tests/dev, strip silently and log a warning in production" — meaning a real secret-leak bug is caught in dev/test but only silently masked (not blocked) in production, precisely the environment where an unredactable, immutable log leaking a credential matters most. The AC also never states whether the redaction scan is recursive through nested payload objects (e.g., a hypothetical `payload.smtp.password`) or only checks top-level keys, leaving a plausible path for nested secrets to slip through.

7. **[high] Extremely rigid dependency on the exact, not-yet-written shape of Stories 9.1/9.2.** This story hard-codes specific route paths, function names (`requirePlatformOperator()`, `is_platform_operator`), table names (`admin_alerts`), and response shapes from two stories that are both still `ready-for-dev` with zero merged code. If 9.1/9.2 ship with any naming or shape deviation (plausible, since they are developed independently), Task 5's retrofit and AC-7/AC-8/AC-18 become silently stale with no defined re-validation step before this story's dev-story run begins.

8. **[medium] Unbounded growth of `platform_audit_pending_entries` with no forced exit.** If the underlying cause of write failure is not transient (e.g., an unrecoverable vault key, a persistent bug), the pending table grows indefinitely: D10/AC-18's storage monitoring covers only `platform_audit_events`, not the pending table, and AC-16's manual deactivate path itself fails closed with `503` if the log is still unavailable — leaving no operator-controlled way to force an exit from a permanently stuck maintenance mode.

9. **[medium] The Product Surface Contract's claim that the epics.md UI-distinction requirement is "satisfied at the API-contract level"** via the `X-Log-Scope` response header is a stretch bordering on compliance-washing — no human operator will ever see an HTTP response header in the absence of any UI. The actual intent (human-visible distinction between the two logs) is not met, only deferred; framing it as "satisfied" rather than plainly "deferred" risks this gap being lost at Epic 9 retro despite Open Question #1 nominally flagging it.

10. **[medium] No payload size limit is defined anywhere** for the `jsonb payload` column, despite the table being append-only and prunable only after 30–365 days, with monitoring only in the aggregate (a single 5GB default threshold). A single oversized or repeated payload (accidental or operator-driven) could inflate storage disproportionately with no per-row cap to prevent it.

11. **[medium] `POST /api/v1/platform/maintenance-mode` is excluded from AC-13's rate limiting** (only the two GET endpoints are covered). This endpoint toggles a security-sensitive bypass mode and its deactivation path can return `503`; there is no stated protection against rapid activate/deactivate retry loops against it.

12. **[medium] `purge_expired_platform_audit_entries()`'s `SECURITY DEFINER` privilege boundary is unaddressed.** The story asserts it is "callable only by the scheduled job's internal invocation path, not exposed as an HTTP endpoint," but never specifies what actually prevents the application's normal DB role from invoking this `SECURITY DEFINER` function directly — the standard footgun for this Postgres feature.

13. **[medium] AC-16's concurrency guard is specified with an unresolved either/or** ("advisory lock or `SELECT ... FOR UPDATE SKIP LOCKED`") rather than committing to one mechanism, for a subtle, safety-critical path whose entire job is preventing duplicate/lost rows during concurrent drains — left to implementer discretion rather than specified precisely like the rest of the story.

14. **[low] `actionType` naming convention is internally inconsistent.** Most actions follow bare `noun.verb` (`settings.updated`, `org.created`, `maintenance_mode.activated`), but the verify-endpoint's self-audit action is `platform_audit.integrity_verify_run` — redundantly restating the table's own platform scope inside the action name, since every row in this table is inherently platform-scoped already.

15. **[low] `ip_address` is stored as free text with no documented extraction/anti-spoofing mechanism** (e.g., trusted-proxy header handling). A forged forwarding header could pollute an immutable audit column with attacker-controlled data; the story raises this level of paranoia for HMAC/RLS but not for this field.

16. **[low] D2's naming rationale adds a third naming family** (`platform_audit_events` vs. `platform_security_events` vs. `audit_log_entries`) that diverges from epics.md's own literal wording — an acknowledged but unresolved documentation/naming debt, compounding the other doc-drift issues the story itself flags (D6's "chaining" language, the FR103/FR109 mismatch).

17. **[low] Test-helper duplication risk left to chance.** Dev Notes state `createPlatformOperator()` "will need to be added... If Story 9.1's own dev-story run already added this helper, reuse it rather than duplicating" — since 9.1 and 9.4 may be implemented in different sessions with no live coordination, this non-committal instruction risks two divergent helper implementations.

18. **[low] AC-9's filter parameters (`operatorId`, `targetOrgId`, `targetUserId`) never state expected validation behavior** for malformed values (unlike `limit`'s explicit `max(100)` callout), leaving ambiguous whether a malformed filter returns `422` or an unhandled `500`.

19. **[low] No regression guard equivalent to AC-10's route-audit test exists for the RLS session-variable pooling risk** described in AC-3's edge case — the pooling scenario is covered only as a one-off test, not as a durable, systemic guard analogous to the route-shape regression test this story already builds for authorization.

20. **[low] FR103/FR109 are cited as this story's PRD justification despite the story's own admission that these FR numbers "do not appear in `prd.md` itself"** — the ACs trace to epics.md text that may not itself trace to any ratified PRD requirement, a provenance gap the story notes but does not resolve before treating those FRs as authoritative.
