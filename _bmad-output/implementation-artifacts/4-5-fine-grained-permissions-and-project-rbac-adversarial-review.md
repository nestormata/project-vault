# Adversarial Review: Story 4.5 — Fine-Grained Permissions and Project RBAC

**Date:** 2026-07-09
**Reviewed file:** `_bmad-output/implementation-artifacts/4-5-fine-grained-permissions-and-project-rbac.md`
**Reviewer:** bmad-review-adversarial-general

## Findings

- **[high]** AC-V8's own worked regression example rests on a mechanism that does not exist. It describes "a freshly-promoted org admin (Story 4.2's role-change endpoint, never personally created or been invited to any project)" as the test scenario for the owner/admin bypass. Story 4.2's `PUT /org/users/:userId/projects/:projectId/role` changes a **project** role, not the org role — Story 4.2's own D4 explicitly states "This story introduces **no** org-role-change endpoint at all (org role changes are out of scope entirely — org role is only ever set at registration/invite-acceptance time per Story 4.1's D5)." Confirmed independently: `apps/api/src/modules/auth/service.ts`'s `RegisterResult.role` type is literally `'owner' | 'member'` (no `'admin'`), and Story 4.1's D5 + its own tests confirm every project-invitation acceptance grants `org_memberships.role = 'member'` unconditionally, regardless of the invited *project* role. There is no shipped API path, anywhere in this codebase, that ever produces an org role of `'admin'` for an existing user. This is a real factual error, and a notable one given this story's entire thesis is the org-role-vs-project-role distinction — the AC's own regression example conflates the two axes the story is trying to separate. The regression test itself is still writable (seed a user with `orgRole: 'admin'` directly), but the AC's narrative framing of *how* that state is reached is wrong and should be corrected before it misleads whoever writes the Task 6.1/AC-V8 test into believing a promotion flow exists to exercise end-to-end.

- **[medium]** The Background section's "Confirmed: every other project-scoped route has the identical gap" table, and AC-V4's route list built on it, are wrong about `GET .../credentials/:credentialId/access`. The table groups it with `GET .../credentials/:id`, `.../versions`, `.../dependencies` as "org role `viewer` floor" — but the route's actual `security` config (`credentials/routes.ts`, the `/access` registration) is `allowedRoles: ['owner', 'admin']`, an org-role gate that already excludes every `member`/`viewer` caller before the handler runs. AC-V4 nonetheless includes this route among the eleven where "a member/viewer caller has no `project_memberships` row" is the scenario under test — but a member/viewer can never reach this handler at all today, with or without D1, so the new visibility check inserted there is provably dead code (it can only ever be evaluated for owner/admin callers, for whom D1's bypass makes it unconditionally `true`). This undermines the section's own claim to be "verified by direct inspection of every route below."

- **[medium]** AC-V4 miscounts and under-enumerates its own "every route" list. It states "(11 routes total)," but its own prose names 12: `GET`/`POST /credentials` (2) + `GET /credentials/:id` (1) + `GET /credentials/:id/value` (1) + `POST /credentials/:id/versions` (1) + `GET /credentials/:id/versions` (1) + `POST`/`GET`/`DELETE .../dependencies` (3) + `PATCH /credentials/:id` (1) + `PUT /credentials/:id/tags` (1) + `GET /credentials/:id/access` (1) = 12. Worse, the actual `credentialRoutes()` registration in `credentials/routes.ts` has 15 `:projectId`-scoped routes, and 3 are missing from AC-V4's list entirely: `POST .../credentials/import`, `POST .../credentials/import/confirm`, and `PATCH .../credentials/:credentialId/tags` (the append-mode tag route). The tags omission is low-risk — `PUT` and `PATCH` tags both call the same `handleCredentialTagUpdate()` function, so fixing the listed `PUT` route fixes the omitted `PATCH` one for free — but the two import routes are a genuine gap in the "every route" framing (currently harmless only because they're already `allowedRoles: ['owner','admin']`-gated, same reasoning as the `/access` finding above). AC-V10's "12 call sites" total (1 dashboard + "11" credential routes) inherits the same off-by-one.

- **[medium]** AC-V6 (and Task 4.2, which explicitly says "mirroring Task 2's pattern") implies all four `getOrgDashboardData()` fields can be scoped to a member/viewer's visible projects the same way: "add the join/filter condition." That holds for `totalCredentials` and `expiringWithin30Days` (both built from direct SQL against `credentials`/`projects`), but `projectsWithOverdueRotations` is built from `computeUpcomingRotations(tx, { horizonDays: 0 })` (`dashboard-stats.ts` → `rotation/service.ts`), whose signature only accepts a single optional `projectId`, not a project-id set — there is no SQL join available at that call site to scope it to an arbitrary visible-project set. Implementing this AC as written requires either a post-filter step (fetch the org-wide result, then drop entries outside the caller's visible-project set) or a signature change to `computeUpcomingRotations` that Dev Notes' "resist the urge to over-abstract... write the three call sites directly first" guidance did not anticipate needing for this fourth field.

- **[medium]** AC-P7 mischaracterizes the current state of the exact code it asks to change. It says the page currently shows "whatever generic/fallback error text" for the reveal 403 — but `+page.svelte`'s `revealValue()` catch block already has a specific branch: `if (error instanceof ApiClientError && error.status === 403) { revealError = 'You do not have permission to reveal credential values.' }`. That branch is unreachable today only because the API doesn't return a 403 for this route yet (correctly noted elsewhere in the story) — it is not the generic fallback AC-P7 implies exists. The actual required change is narrower than AC-P7's framing suggests (differentiate by `error.code === 'insufficient_project_role'` rather than "replace the generic handler"), and a developer trusting the AC's description at face value risks adding a duplicate/conflicting 403 branch instead of refining the one that's already there.

- **[low]** A foreseeable UX inconsistency from Group P is never addressed: the reveal button's client-side visibility (`canReveal = canCreateCredential(data.orgRole)`, `+page.svelte`) is purely org-role-based and has no knowledge of the caller's effective *project* role, which the page's load function (`+page.server.ts`) never fetches either. Post-story, a project-role `'viewer'` whose org role is `'member'`+ will still see an enabled "Reveal value" button that is now guaranteed to fail with the new 403 — a foreseeable "sad-path button" the story neither flags as an accepted trade-off nor asks to fix, despite AC-P7's adjacent regression note carefully reasoning through a related (404) case.

- **[low]** AC-V7's backfill `INSERT` has no `org_memberships.status = 'active'` filter, so a deactivated org member also receives a backfilled `'viewer'` row on every project. This is harmless today — `authenticate.ts`'s `loadOrgRole` filters `status = 'active'`, so a deactivated user cannot authenticate at all regardless of any `project_memberships` row — and arguably correct if the user is later reactivated (it preserves what they'd have had under the pre-story org-wide model). But the story's otherwise example-heavy AC-V7 never states or tests this reasoning, despite the story's own author being visibly careful about enumerating edge cases elsewhere (archived projects, zero-membership projects).

- **[low]** Open Question 4 (org owner/admin's unconditional reveal bypass) is left as a deferred design question with zero compensating audit signal. AC-V10/AC-P6 add structured logs for *denials*, but nothing logs the specific scenario Open Question 4 itself names as worth future reconsideration — an org admin exercising the reveal bypass on a project where they also hold an explicit, lower `project_memberships` role. The design choice itself is defensible and consistent with existing precedent (archival, member-management), so this need not block the story, but leaving it purely as prose with no distinguishing audit line means that exact scenario stays unmonitorable even after a future reviewer re-examines it.

- **[low]** Redundant DB round-trip on both Group P routes: `callerCanSeeProject` (AC-V4's gate, which the story's own ordering requires to run first) and `effectiveProjectRole` (AC-P2/AC-P3's gate) each independently call `getProjectMembershipRole` for the identical `(orgId, projectId, userId)` triple on `GET .../value` and `POST .../versions`. The story's "one new helper... reused eleven times" framing (AC-V4) doesn't flag that two of those eleven call sites also pay for a second, duplicate lookup immediately afterward via AC-P1's helper — a small, easily-avoided inefficiency (thread the already-fetched project role through, or have `effectiveProjectRole` accept an optional pre-fetched role) that the Dev Notes' anti-over-abstraction guidance doesn't mention either way.

## Verification Notes (not findings — confirms the story's factual claims against real code)

- ADR-2.1-01, ADR-2.2-03, and ADR-4.4-05 are all cited and represented accurately against the real story files (`2-1-...md`, `2-2-...md`, `4-4-...md`) — context, decision, and consequences all match verbatim or in substance.
- The RLS-vs-application-layer claim is accurate: `check-rls-coverage.ts`'s `EXCLUDED_TABLES` and the `project_memberships_isolation` policy (`0013_projects.sql`) confirm project-level scoping has never been an RLS concern in this codebase, and `make db-migrate` (`Makefile`) runs as the postgres superuser, which bypasses RLS entirely — so AC-V7's migration will succeed regardless of `app.current_org_id` session state, independent of whether the explicit `org_memberships.org_id = projects.org_id` join was strictly necessary for that reason.
- `findDestructiveStatements` (`migration-safety.ts`) genuinely does not flag a plain `INSERT ... SELECT ... ON CONFLICT DO NOTHING` — AC-V7's guard-safety claim is correct.
- The `project_memberships` primary key is `(project_id, user_id)` (`packages/db/src/schema/project-memberships.ts`), matching AC-V7's `ON CONFLICT (project_id, user_id)` target exactly.
- The migration-index-43 collision with sibling story `3-5-credential-expiry-notification-delivery` is real (both stories' worktrees independently target `idx: 43` against the same `_journal.json` state, confirmed highest existing entry is `idx: 42`) — but the coordination notes added near AC-V7, Task 5.1, and the Project Structure Notes section are clear, correctly cross-reference the sibling story by name, and are reciprocally consistent with the matching note already present in 3-5's own story file. No new finding needed here per this review's scope.
- AC count is internally consistent: V1–V10 (10) + P1–P7 (7) = 17 as claimed, no duplicate or contradictory AC numbers found.
- The `roleRank`/`OrgRole` claims (D4, AC-P1) match `secure-route.ts` exactly, and the reveal/version-create routes' current response schemas genuinely lack a `403` today, as claimed.

## Overall

No explicit risk-rating scale is defined by the invoked skill; based on findings severity, this story is close to implementation-ready but should not proceed as-is: the AC-V8 finding (org-role-promotion mechanism that doesn't exist) should be corrected before Task 6 is implemented from it, since it currently points the regression test's narrative at a non-existent user journey. The four `[medium]` findings are all research/enumeration inaccuracies in the ACs themselves (not security regressions) — none of them describes a path where a member/viewer gains unauthorized access — but each would cost real implementer time to discover mid-Task and should be fixed in the story text first. The `[low]` findings are legitimate but genuinely deferrable. The migration backfill's core backward-compatibility logic (org owner/admin bypass, per-project-pair backfill scoped by org, `ON CONFLICT DO NOTHING` preservation of real invitation-granted roles, archived-project inclusion) checks out as correct against the schema and RLS/migration-execution model.
