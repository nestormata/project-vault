# Adversarial Review: Story 8.5 — Rotation Web UI Hardening

- **Date:** 2026-07-06
- **Reviewed file:** `_bmad-output/implementation-artifacts/8-5-rotation-web-ui-hardening.md`
- **Reviewer:** bmad-review-adversarial-general
- **Also-consider scope:** inconsistencies and contradictions between sections of this story; gaps in acceptance criteria or missing edge cases; technical debt implied by the proposed approach; implementation and delivery risks; and security risks — authn/authz, RLS/tenant isolation, secrets handling, injection, input validation, audit/logging gaps — wherever applicable to this story.

## Findings

- **[critical] AC-4's own instructions are internally inconsistent about which of the three dashboard-loader calls actually gets 503-handling, and one of the two named sources has no error-handling mechanism to extend.** The AC's "Given" clause states "a `503` from `listProjects` or `getProjectDashboard` is caught," but the parenthetical implementation guidance says to "add a sibling `503` branch" only to "the existing `getOrgDashboard(...).catch(...)`" and "the `getProjectDashboard` `try`/`catch`" — i.e., `getOrgDashboard` and `getProjectDashboard`, not `listProjects`. Direct inspection of `apps/web/src/routes/(app)/dashboard/+page.server.ts` (current code) confirms `listProjects(fetch)` sits bare inside the `Promise.all([...])` array with **zero** existing catch mechanism of any kind — unlike `getOrgDashboard`, which already has a `.catch()`, and `getProjectDashboard`, which already has a `try`/`catch`. A developer following the parenthetical's literal "add a sibling branch to both" instruction would never add any 503-handling to `listProjects` at all, directly contradicting the AC's own Given clause that names `listProjects` as one of the two possible 503 sources this AC must close.

- **[high] Story-wide claim that resume/abandon share a "60/min" rate-limit cap is factually wrong — the actual code shows 30/min.** The "Why this story exists" re-verification section (line ~39) states `resolutionRateLimit()` is "for resume/abandon, same 60/min shape," and the "Ground-Truth API Surface" table (AC rows for `.../resume` and `.../abandon`) both list "60/min." Direct inspection of `apps/api/src/modules/rotation/routes.ts` (~line 1262) shows `function resolutionRateLimit(key) { return { max: 30, timeWindowMs: 60_000, key } as const }` — 30/min, not 60/min — with an adjacent code comment explicitly contrasting it against "the 60/min checklist-mutation bucket." This directly undermines the story's own repeated claim (Dev Notes: "every numeric... claim... was re-confirmed by direct `Read`/`grep`") and could mislead a developer writing AC-15's test/comments into asserting the wrong shared-bucket relationship between resume/abandon and the checklist mutations.

- **[high] D1's claim that all four touched loaders "already" have the `notFound: true as const` pattern for 404s is only true for 2 of the 4 files.** D1 states the new `vaultSealed` field "mirror[s] the existing `notFound: true as const` pattern each of these files already uses for 404s." Direct inspection shows: `credentials/[credentialId]/+page.server.ts` and `rotations/[rotationId]/+page.server.ts` do have this pattern; but `rotate/+page.server.ts` has **no try/catch and no `notFound` field at all** today (its `load` just calls `listRotations`/`listCredentialDependencies` unguarded), and `dashboard/+page.server.ts` has no `notFound` field either — it only silently swallows 404s to `null` via `.catch()`/`try`-`catch`, never surfacing a discriminant flag. Implementing AC-2 and AC-4 is therefore a bigger lift than D1 implies (introducing new error-handling structure from scratch, not "extending" an existing flag), and a developer relying on D1's blanket claim may under-scope the work for these two files.

- **[high] AC-1 is missing the explicit "check `vaultSealed` before `notFound`" ordering instruction that AC-2, AC-3, and AC-4 all explicitly state, even though the same failure mode applies to it.** AC-2 says the page "checks `data.vaultSealed` **before** its existing... check"; AC-3 says the page "checks `data.vaultSealed` before its existing `data.notFound || !rotation` check"; AC-4 says "`+page.svelte` checks `data.vaultSealed` first." AC-1 states no equivalent ordering requirement. The current template at `credentials/[credentialId]/+page.svelte` reads `{#if data.notFound || !data.credential}` — and per D1, a sealed-vault response nulls `credential` — so a literal implementation of AC-1 without an explicit ordering note (unlike its three sibling ACs) risks silently rendering "Credential not found" for a sealed vault, which is exactly the bug class AC-2/AC-3/AC-4 explicitly guard against.

- **[high] AC-16's fix can trap the admin in a state where the break-glass form cannot be resubmitted or edited, contradicting D5's "accepted friction cost" framing.** AC-16 clears `newValue` in a `finally` block on any error, but never touches `awaitingConfirmText`/`confirmText`. Per the current (and unchanged-by-this-story) template logic in `BreakGlassPanel.svelte`, the new-value `<textarea>` has `disabled={awaitingConfirmText}`, and after a failed submit `awaitingConfirmText` stays `true` — so the textarea is now both empty (per AC-16) **and disabled**, meaning the admin cannot re-paste the value. Meanwhile `confirmText` is untouched and still equals `'CONFIRM'`, so the "Confirm break-glass rotation" button (`disabled={confirmText !== 'CONFIRM' || submitting}`) remains enabled and would immediately resubmit with an **empty** `newValue` if clicked again. The admin's only way out is to notice this and click "Cancel" to fall back through `cancelConfirmation()`. D5 explicitly frames the accepted cost as "the admin must re-paste the value to retry" — but as specified, the field is not editable again without first escaping the confirm step, which AC-16 never mentions.

- **[medium] Tension between D3's single shared error-mapping helper and the Group B Non-Goal's stated rationale for excluding confirm/fail/retry.** The Non-Goal section argues against adding an `mfa_required` branch to `ChecklistItemRow.svelte`'s confirm/fail/retry because the server can never send that error there, calling such a branch "dead code... untestable." But D3/AC-20 mandate a **single shared** `mapRotationMutationError()` function (covering 503/`mfa_required`/429/generic) that AC-13 explicitly requires confirm/fail/retry to adopt (for their 429 handling). That shared function's `mfa_required` branch is therefore compiled into and reachable from confirm/fail/retry's call sites regardless — it's just reached through the shared module instead of being hand-written locally. The story doesn't reconcile this: the "no dead code" rationale for the Non-Goal is really only "no *locally duplicated* dead code," which is a materially different and unstated claim.

- **[medium] The `mfa_required` detection pattern being replicated five more times (`errorMessage.includes('MFA')`) is fragile string-matching, and the story's own shared helper (D3) has strictly more information available to avoid it.** The members-page precedent this story copies verbatim checks for the substring `"MFA"` in the *rendered message* rather than a boolean/code check, so any future generic error message that happens to contain "MFA" (e.g., an unrelated validation string) would incorrectly render the "Enable MFA" link. Since `mapRotationMutationError()` (D3) already knows definitively whether the underlying error was `mfa_required` (it branches on `error.code === 'mfa_required'` to build the message in the first place), the story could have had it return a `{ message, isMfaRequired: boolean }` shape and rendered the link off the boolean — instead it deliberately re-derives the same fragile substring check five times "to match the precedent," propagating rather than fixing a known-fragile pattern.

- **[medium] Missed, closer, in-repo precedent for AC-17's secret-clearing-on-teardown requirement.** AC-17 asks for a new `onDestroy(() => { newValue = ''; reason = '' })` hook in `BreakGlassPanel.svelte`, framed as introducing "a Svelte 5 teardown mechanism." The credential detail page this story also touches for AC-1 (`apps/web/src/routes/(app)/projects/[projectId]/credentials/[credentialId]/+page.svelte`) already has exactly this pattern today: `onDestroy(() => { revealedValue = null; revealVersion = null })`, clearing a revealed secret value on unmount. Neither the "Key Code Patterns to Follow" section nor the References list cites this as the working precedent, despite it being in a file this very story's Task 2 already modifies and being a much closer analogy than anything actually cited for AC-17.

- **[medium] No consideration of repeated-failure telemetry/monitoring for the highest-blast-radius flow this story hardens.** The "also consider... audit/logging gaps" scope is relevant here: this story adds client-side handling for repeated `429`/`mfa_required` responses on break-glass — the flow the story itself calls "the highest-blast-radius path in all of Epic 5" — but nowhere considers whether repeated break-glass rate-limit or MFA-enrollment failures for a given user are worth any additional server- or client-side signal (e.g., to security/ops, who might want to know an admin account is being hammered against the break-glass endpoint during an incident). This is arguably out of this story's stated `apps/web`-only boundary, but the story doesn't even flag it as a known, deferred gap the way it carefully flags other deferred items.

- **[low] `ApiFailure`'s two similarly-named fields (`retryAfter?: number` and `retryAfterSeconds?: number`) are a latent landmine the story doesn't warn about.** `apps/web/src/lib/api/client.ts` declares both fields on `ApiFailure`. The story correctly identifies `retryAfter` as the field `enforceUserRateLimit()` actually populates, but never warns that a developer grepping the type for "retry" will find two candidates and could easily read the wrong (always-`undefined`) one, silently producing the "Try again shortly" fallback for every 429 instead of the intended countdown.

- **[low] AC-5's fake-timer poll test guidance doesn't address the existing visibility-gated poll precedent it must interact with.** Story 5.4 already shipped a "does not refetch while the tab is hidden" test (`apps/web/src/routes/rotation-detail-page.test.ts`) alongside the 15s-interval poll test the story does correctly point new developers toward. AC-5 doesn't mention that `schedulePoll()` no-ops when `document.visibilityState === 'hidden'`, so a sealed-vault banner introduced by this AC will not appear until the tab is refocused if the vault seals while the tab is backgrounded — consistent with D6's "self-heal" framing in spirit, but never stated, leaving it to the implementer to notice.

- **[low] Epic Cross-Story Context repeats the "13 findings" figure for Story 5.5 without acknowledging a known, already-documented discrepancy.** Story 5.4's own adversarial review (`5-4-rotation-workflow-web-ui-adversarial-review.md`) flags a "numeric inconsistency" where `sprint-status.yaml` says Story 5.5 bundled "13" findings but `deferred-work.md`'s enumerated list under the same citation names only 12. This story's Epic Cross-Story Context table (line ~78) repeats "13 findings" as settled fact with no acknowledgment that the source documents it treats as ground truth elsewhere still disagree with each other on this exact number.

- **[low] Group C's 429 test guidance ("mock ... to reject with `429`") does not account for `enforceUserRateLimit()` being a no-op in the default test/dev environment.** `apps/api/src/lib/route-helpers.ts`'s `enforceUserRateLimit()` short-circuits via `isRateLimitEnforced()`, which is disabled by default under `NODE_ENV=test` (and, per existing comments elsewhere in the API codebase, likely dev too) — meaning a developer trying to manually verify AC-11/AC-12 end-to-end against a running local API (rather than a mocked unit test) will not be able to trigger a real 429 without deliberately re-enabling rate limiting first. The story's Vitest-mock-based test plan sidesteps this correctly, but doesn't mention it as a manual-verification caveat, which could cost an implementer time during their own sanity-check pass.

- **[low] The break-glass 429 message's reassurance clause ("this limit exists to prevent runaway automated calls, not to block a real incident response") has no test coverage of its own.** AC-12's test only asserts a substring match on the countdown ("the break-glass-specific message renders with '45 seconds'"), so the deliberate reassurance framing the AC calls out as important is never independently verified by the AC's own test plan, and could silently regress or be dropped during implementation without any test failing.
