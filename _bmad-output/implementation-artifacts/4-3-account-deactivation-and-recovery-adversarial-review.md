# Adversarial Review — Story 4.3: Account Deactivation & Recovery

- **Date:** 2026-07-01
- **Reviewed file:** `_bmad-output/implementation-artifacts/4-3-account-deactivation-and-recovery.md`
- **Reviewer:** bmad-review-adversarial-general
- **Also-consider scope:** inconsistencies/contradictions between sections; gaps in ACs / missing edge cases; technical debt implied by the proposed approach; implementation and delivery risks; security risks (authn/authz, RLS/tenant isolation, secrets handling, injection, input validation, audit/logging gaps).

---

## Findings

- **[HIGH]** AC-15's MFA re-enrollment "mirror" of `verifyEnrollment` (`apps/api/src/modules/auth/mfa.ts:328`) is incomplete in a way that recreates the exact lockout the story exists to fix. The real `verifyEnrollment` always calls `generateRecoveryCodes` + `insertRecoveryCodes` on successful TOTP confirmation, issuing a fresh batch of `mfa_recovery_codes` to the user. D1/AC-15 only describe promoting the pending `mfa_enrollments` row to `status: 'confirmed'` and setting `users.mfaEnrolledAt` — they never mention regenerating recovery codes, and AC-14's completion response schema (`{ email, sessionsRevoked }`) has no field to return them. Since Persona B's entire premise is "lost MFA device **and** exhausted/lost recovery codes," a user who follows the story as written ends up with active MFA but zero recovery codes — set up to hit this exact recovery flow again the next time they lose their device.

- **[HIGH]** AC-11's anti-enumeration guarantee is undermined by an intrinsic, non-subtle timing side channel that the story itself only partially acknowledges. The "hit" path (email exists) performs work the "miss" path never does: an `INSERT` into `account_recovery_tokens`, an `UPDATE` to supersede a prior token (AC-13), a `notificationQueue` enqueue, and — per D2 — **one `audit_log_entries` INSERT per active org membership** the user holds. None of this happens on a miss. AC-9's own edge case admits only "best-effort constant-shape handling, not cryptographic constant-time," but for a user in several orgs the write-count differential (N audit rows vs. zero) makes the timing gap large and easily measurable, not a marginal side-channel — this materially weakens the enumeration resistance the AC claims to deliver.

- **[HIGH]** No task anywhere in the story covers documenting the new required production secret `RECOVERY_TOKEN_HMAC_SECRET` for operators. AC-1 mirrors `INVITATION_TOKEN_HMAC_SECRET`'s production-mode hard-fail validation (env.ts's existing pattern), but Story 4.1 — the story this pattern is copied from — explicitly required updating `.env.example` and listed it in its own file list (`4-1-team-invitations-and-role-assignment.md:617,773`). Story 4.3 has no equivalent requirement in AC-1, the Tasks list, or Project Structure Notes. Since this is a self-hosted product, an operator upgrading in place with an unset secret will hit a hard boot failure with no documented remediation path.

- **[MEDIUM]** The rate-limit helper AC-11 instructs devs to reuse "verbatim from `apps/api/src/modules/auth/routes.ts:213-238`" (`enforceRecoverRateLimit`) is a module-private function with no `export` keyword. The story's own Project Structure Notes leave open the possibility of putting the new routes in a separate `recovery-routes.ts` file "if `routes.ts` is getting unwieldy" — but if that path is taken, the helper cannot be imported without first exporting it, an edit no task calls out. This is a near-certain mid-implementation snag given the story explicitly defers the file-organization decision to implementation time.

- **[MEDIUM]** AC-12's `no_admin_available` boundary is a documented but unquantified anti-enumeration leak: it tells an unauthenticated caller that a specific email belongs to an account whose every org membership lacks a reachable admin — a third response state distinguishable from the generic `202`. The AC calls this "deliberate" and cites a narrow "realistic trigger," but offers no monitoring, no differential rate-limiting for this branch, and no analysis of how much more reachable this state could become once other stories (e.g., a future ownership-transfer or user-deletion feature) exist. The risk is real; the story's confidence that it's rare is asserted, not demonstrated.

- **[MEDIUM]** AC-9 (self-initiated) and AC-10 (admin-initiated) both write to the same `account_recovery_tokens` row space and unconditionally supersede any prior unused/unexpired token for the user (AC-13), regardless of which actor created it. If Alex self-requests a link and Riley independently sends one moments later (plausible — Persona C describes exactly this kind of admin-assist scenario, just for a different trigger), Alex's already-opened tab silently becomes `410 recovery_token_superseded` with no notification of why. No AC or test plan covers this cross-actor race.

- **[MEDIUM]** D2's per-org audit/session-revocation fan-out (AC-9, AC-14, AC-16) is applied inside a single transaction with no stated upper bound, batching strategy, or test coverage for a user belonging to many orgs. This is a latent lock-contention/transaction-duration risk (sequential `revokeAllUserSessionsInOrg` calls plus one audit insert per org, all before commit) that the story never weighs against the correctness rationale it carefully argues for D2 — the "why fan out" case is made, the "what does fan-out cost at scale" case is not.

- **[MEDIUM]** AC-6's mid-request race justification is technically shaky and could mislead a reviewer into skipping real analysis. The story asserts "Postgres transaction isolation means Sam's in-flight request either sees the pre- or post-deactivation state consistently, never a torn read" — but the auth-middleware check ("evaluated once per request, at the start") and the route handler's own subsequent queries are described as separate steps, not one transaction held open for the full request lifecycle. Under ordinary read-committed isolation with separate statements, there is no such single consistent snapshot. The functional outcome (middleware gate is authoritative; next request is blocked either way) is probably fine, but the stated rationale for *why* no additional protection is needed does not hold up and should not be relied upon as-is.

- **[MEDIUM]** No task covers authoring the actual email template content, and AC-10 (admin-initiated) doesn't even specify its own `templateId` — it just says "the same token-creation/supersede/email-enqueue steps as AC-9 run," implying reuse of `auth.recovery_link_created` for both the self-service and admin-triggered cases. An admin-sent recovery email and a self-requested one plausibly need different copy (anti-phishing clarity: "your admin sent you this link" vs. "you requested this link") — this distinction is neither required nor ruled out anywhere in the story.

- **[MEDIUM]** AC-3 blocks an admin from deactivating a peer admin ("admin-vs-admin is blocked too"), but AC-10 places no equivalent hierarchy restriction on `POST /org/users/:userId/recovery/send-link` — any admin/owner can send a password-recovery link to any other admin/owner in the org, including a peer. That is a materially more sensitive action (it is a credential-reset primitive against a peer's account) than the one the story bothers to hierarchy-guard. This asymmetry isn't discussed as a deliberate design choice anywhere, unlike the story's careful D1–D7 treatment of every other ambiguity.

- **[MEDIUM]** AC-13's masked-email format (`al***@example.com`) is underspecified: no rule is given for short local-parts (e.g., a 2-character local part would have its entirety visible with only cosmetic padding), and the domain is left fully unmasked — for an org using a distinctive or low-population email domain, this could still meaningfully narrow identification of the account holder. No test case pins down the exact masking algorithm.

- **[MEDIUM]** Persona A (step 4) references Riley sending Sam "a recovery link if Sam is reactivated by other means," while D4 explicitly states no reactivation mechanism exists anywhere in the product ("`org_memberships.status` can only be flipped back to `'active'` by direct database action today"). The persona narrative dangles a workflow that depends on a capability the story explicitly declines to build and never says what "other means" could concretely refer to today. This is confusing scene-setting at best and should either be cut from the persona or made explicit that it refers only to a hypothetical future capability.

- **[MEDIUM]** RLS/tenant-isolation ambiguity during recovery completion: the recovery flow is deliberately public/token-authenticated (AC-17), meaning it never establishes the normal `secureRoute`/`withOrg()` org-scoped connection context the rest of the codebase's mutations rely on. AC-13/14/15 describe the token lookup as going through the admin connection (correctly, matching the invitation-token precedent), but never clarify whether the subsequent mutation phase — password update, the per-org session-revocation loop, the per-org audit fan-out — switches to an RLS-scoped path per resolved org, or continues on the admin/superuser connection for the entire transaction out of convenience. Given this story's own theme (tenant isolation matters enough to get D3 dedicated to it for deactivation), leaving the completion path's connection/RLS discipline unstated is a real gap, not a stylistic one.

- **[LOW]** Dev Notes / D1 cite `apps/api/src/modules/auth/mfa-enforcement.js` — the actual file is `mfa-enforcement.ts`. A minor citation error, but this story leans heavily on "verified by reading the file directly" claims elsewhere (AC-1, Dev Notes), so small inaccuracies like this reduce confidence that every other file:line citation was actually re-checked at write time.

- **[LOW]** No guidance on cleaning up stale MFA artifacts during recovery re-enrollment. `deletePendingEnrollmentForUser` only removes rows with `status = 'pending'` in `mfa_enrollments`; it does not touch any prior *confirmed* enrollment row or that enrollment's `mfa_recovery_codes`. If a confirmed enrollment somehow still exists at recovery time (edge state), the recovery flow as specified leaves orphaned artifacts behind — harmless functionally, but technical debt the story doesn't acknowledge.

- **[LOW]** AC-19's resolution for the deactivation-race session-revocation call is deliberately non-committal ("must either be skipped entirely by checking status first inside the lock, or be a safe no-op"). That's a reasonable AC-level allowance, but it leaves an actual implementation decision unresolved with no test asserting which behavior is required, inviting two different valid-looking implementations from two different developers.

- **[LOW]** The story never states what happens to an outstanding recovery token if the target user is deleted outright (not merely deactivated) before the token is used. The FK is `ON DELETE CASCADE` on `user_id`, so the row would vanish and the peek endpoint would presumably return `404 recovery_token_not_found` — a reasonable inference, but it is inferred, not stated or tested, despite user deletion being a plausible future admin capability in this product.

- **[LOW]** AC-2's response payload and AC-16's audit payload for deactivation (`{ revokedSessionCount, revokedInvitationCount }`) don't reserve a field for the D7 rotation-check outcome (e.g., an empty `rotationIds: []` placeholder), even though D7 otherwise very carefully documents forward-compatibility for the Epic 5 stub. When Epic 5 ships and the real check is dropped in, this response shape will need an additive change that nothing here anticipates.

- **[LOW]** AC-9's `email` input field has no stated validation beyond "normalized" — no length bound or format check is specified, unlike AC-14's explicit instruction to "reuse the existing password schema/validator, do not invent a new one." Without an equivalent explicit instruction for the email field, there's a real chance the implementer skips reusing registration's email schema and instead accepts unbounded input into normalization, hashing/lookup, and rate-limit-key construction.
