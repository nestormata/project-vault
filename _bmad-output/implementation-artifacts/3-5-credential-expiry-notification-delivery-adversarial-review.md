# Adversarial Review: Story 3.5 — Credential Expiry Notification Delivery

**Date:** 2026-07-09
**Reviewed file:** `_bmad-output/implementation-artifacts/3-5-credential-expiry-notification-delivery.md`
**Reviewer:** bmad-review-adversarial-general

## Findings

- **[critical]** Migration index collision with a sibling in-flight story. AC-W1/Task 1.3 claims migration index **43** as "the next free migration index" and instructs implementers to hand-write `packages/db/src/migrations/0043_credential_expiry_alerts.sql`. The sibling worktree `4-5-fine-grained-permissions-and-project-rbac` (also `status: ready-for-dev`, also verified against the identical `_journal.json` state — highest existing entry `idx: 42`/`0042`) independently claims the **exact same index**: AC-V7 of that story hand-authors `packages/db/src/migrations/0043_project_membership_visibility_backfill.sql` with `idx: 43`. Both stories were planned in parallel against the same base and both believe they own slot 43. Whichever branch merges second will either silently overwrite the journal entry, produce two files racing for one `idx`, or fail `drizzle-kit`/CI migration-ordering checks — and neither story's text acknowledges the other exists. This is exactly the coordination risk the task brief asked to check for, and it is not hypothetical: it is a real, present collision between two concrete ready-for-dev stories. The story must be revised to either explicitly claim/reserve an index range or add a note instructing whoever lands second to renumber before merge — "re-verify against `_journal.json` immediately before implementing" is not sufficient because both stories' authors already did that and got the same answer.

- **[high]** The DLQ-cleanup-vs-in-flight-dispatch race is only closed in one direction. AC-Q2's race edge case correctly requires the new cleanup `UPDATE` to be scoped `WHERE status = 'pending'` so it can't overwrite a row that became `'delivered'` in the interim, and justifies this by claiming it "match[es] `markNotificationDelivered`'s/`claimPendingNotificationEntry`'s own unconditional-except-status-guarded pattern." Direct inspection of `notification-queue-ops.ts` shows this is only true for `claimPendingNotificationEntry` (its `UPDATE` has `AND status = 'pending'`) — **`markNotificationDelivered` and `markNotificationSuppressed` have no status guard at all** (`.where(eq(notificationQueue.id, notificationQueueId))` only). Combined with the fact that `claimPendingNotificationEntry` never transitions status away from `'pending'` while a delivery attempt is in flight (it only bumps `attemptCount`/`lastAttemptAt`), a genuinely slow/hung in-flight delivery (e.g., a stuck SMTP call running past the 30-minute cleanup threshold) can be marked `'failed'` by the cleanup job, and then, when the hung handler eventually finishes, `markNotificationDelivered` will silently overwrite that `'failed'` status back to `'delivered'` with zero guard — contradicting the DLQ cleanup summary log already emitted and undermining the "bounded terminal state" guarantee the story's own Story section promises. The story neither modifies `markNotificationDelivered`/`markNotificationSuppressed` to guard against this reverse race nor flags it as a known limitation the way it correctly does for the pre-existing `expiry-alert-shared.ts` race. This should either be fixed (guard both functions, or have them log/no-op distinctly when overwriting a `'failed'` row) or explicitly documented as an accepted, narrow residual race — right now it's silently absent from the design.

- **[medium]** AC-Q2's log-level requirement is written as an either/or choice ("`'info'` level (or no log at all, matching `runNotificationBackfill`'s existing... convention) — pick one and apply consistently"), which makes the AC itself untestable as literally stated: two different implementations could both claim compliance while producing different, mutually exclusive log behavior. An acceptance criterion should commit to one behavior so the RED test in Task 5.3/5.5 has an unambiguous expected outcome to assert against.

- **[medium]** Coordination gap acknowledged but not otherwise mitigated: `2-9-credential-project-web-ui-completeness` has no story file yet in its own worktree (still backlog), so it's impossible from here to confirm it won't also touch `credentials`' schema or claim migration slot 43 — this is a second, currently-unverifiable instance of the same collision risk as the critical finding above, and the story should say so explicitly rather than only citing `payment-records.ts` as its collision precedent-check.

- **[medium]** AC-D1/AC-D2's batching change is semantically sound for the "no preference row" and "unsubscribed mid-batch" cases the task flagged for scrutiny (defaults still fill correctly per user in the batched path per AC-D1's own examples; a single upfront `SELECT` for N users is, if anything, a *more* internally-consistent snapshot than N sequential per-user `SELECT`s issued at slightly different instants within the same transaction) — but the story never explicitly states this reasoning or tests it as a named regression case ("a preference row is deleted/changed by a concurrent request between when the batch would have fetched user 1 vs user 3"). Given the story goes out of its way to prove no-race-introduced properties elsewhere (AC-W5's documented pre-existing race, AC-Q2's race edge case), the silence here on an equivalent question for AC-D1/D2 is an inconsistency in the story's own standard of rigor, even though the underlying change is actually safe.

- **[low]** AC-Q4's justification that `OperationalEventType` is "a `keyof typeof OperationalEvent` derived type" is imprecise — the real declaration is `(typeof OperationalEvent)[keyof typeof OperationalEvent]` (a union of the string *values*, not of the *keys*). The practical claim (no separate union to hand-maintain) is still correct, but the stated mechanism is wrong and could confuse an implementer who goes looking for a `keyof` type and doesn't find one.

- **[low]** The story asserts credentials' existing indexes (`idx_credentials_project_expires`, `idx_credentials_org`) are sufficient without comment, but the new worker's `fetchRows` filters on `(orgId, isNotNull(expiresAt))` with no `projectId` predicate — there is no composite `(org_id, expires_at)` index on `credentials`, only `(project_id, expires_at)`. This is a non-issue in practice (verified identical to every sibling table's indexing — `cert_records` has the same `(project_id, expires_at)` + `(org_id)`-alone shape and the same gap), so it's a pre-existing convention, not a regression — but the story's AC-W1 could have said so explicitly instead of implying its indexing is a solved question by silence.

- **[low]** Dev Notes' framing of the pre-existing `expiry-alert-shared.ts` concurrent-overlap race as "not the place to fix it" is a defensible scoping call (verified: `runOrgScopedJob` uses a plain transaction with no row locking, so the race is real and does predate this story across all 4 shipped siblings) — but the story's mitigation is entirely a Dev Notes paragraph plus a Task 6.3 promise to add a `deferred-work.md` row. Nothing in the ACs requires that follow-up row to actually exist before the story is marked done, so this could easily slip the same way the four gaps this story itself is closing slipped for months. Consider making Task 6.3's `deferred-work.md` update an explicit, checked sub-item of a Definition-of-Done AC rather than only a task-list checkbox.

- **[low]** AC-W2's payload contract (`{ assetId, projectId, name, expiresAt, daysRemaining, threshold, overdue }`) is internally consistent with `baseExpiryPayload` + the `credentials` schema's `name` column, and matches the sibling workers' shape (`payment-expiry-alert.ts` also uses `name`) — verified correct, no finding, noted only because it was one of the areas flagged for extra scrutiny and is worth recording as a pass rather than silence.

## Overall

No explicit risk-rating scale is defined by the invoked skill; based on findings severity, this story should not proceed to implementation as-is — the critical migration-collision finding is a concrete, verifiable conflict with a second ready-for-dev story in this same reconciliation batch and must be resolved (coordinate index ownership) before Task 1 begins, and the DLQ-cleanup race gap should be addressed or explicitly accepted before Group Q ships.
