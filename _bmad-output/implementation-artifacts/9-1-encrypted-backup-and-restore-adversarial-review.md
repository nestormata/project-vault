# Adversarial Review: Story 9.1 — Encrypted Backup & Restore

- **Date:** 2026-07-05
- **Reviewed file:** `_bmad-output/implementation-artifacts/9-1-encrypted-backup-and-restore.md`
- **Reviewer:** bmad-review-adversarial-general

## Findings

1. **[critical]** AC-9's restore procedure never addresses live-connection safety: running `pg_restore` against `BACKUP_DATABASE_URL` while the API's `vault_app` connections continue to actively serve read/write traffic to the same database means the restore can run concurrently with live transactions, causing lock contention, partial data visibility, referential-integrity violations, or an outright failed restore. No AC or task item requires draining/quiescing live DB connections, putting the API into a maintenance/reject-new-requests mode, or terminating existing `vault_app` sessions (e.g. `pg_terminate_backend`) before `pg_restore` begins. For an operation explicitly described as destructive and whole-instance, this is a severe gap.

2. **[critical]** The single biggest-blast-radius action in the entire system — a destructive whole-instance restore that discards every organization's data — is deliberately routed only through non-tamper-evident operational logging (D6), with compliance-grade audit deferred to a not-yet-written Story 9.4. Shipping this capability without any tamper-evident record of who triggered a full-instance restore (and when) is a governance/security gap that arguably should block this story rather than being accepted as an "interim" measure.

3. **[high]** No filename validation/sanitization is specified anywhere in the ACs or Tasks for the `:filename` path parameter accepted by `POST /admin/backups/:filename/restore` and `.../validate`. Both handlers use this operator-supplied string to locate a file on `BACKUP_STORAGE_PATH` (or an S3 key). Without an explicit allow-list regex (e.g., anchored to the `backup_<timestamp>_<instanceId>.vault` pattern) enforced before file-path construction, this is a path-traversal vector (`../../etc/passwd`-style) for arbitrary file read (validate) or worse.

4. **[high]** D4's mitigation (a) — "`BACKUP_DATABASE_URL` is read from the environment only at the moment `pg_dump`/`pg_restore` is spawned, never held in a long-lived in-memory variable" — is largely illusory as a mitigation. Environment variables are set at container/process start and remain in `process.env` (and therefore trivially readable by any code running in that process, or via `/proc/<pid>/environ` for anyone with host/container access) for the entire lifetime of the API process regardless of when application code chooses to read them. This doesn't meaningfully reduce the exposure window described as the threat.

5. **[high]** No schema-version reconciliation strategy is described for restoring an old backup. `pg_dump`/`pg_restore` of "all tables" implicitly carries schema (DDL) as of the backup's creation time. If an operator restores a backup taken before several intervening Drizzle migrations, the resulting schema can silently diverge from what the currently-deployed application code expects (missing columns/tables added later), breaking the app post-restore in ways not covered by any AC.

6. **[high]** The story states RTO of 2 hours and RPO of 24 hours as the entire justification for this story ("As a platform operator... recover from data loss within the documented 2-hour RTO and 24-hour RPO targets"), yet no AC or integration test in AC-19's exhaustive list actually measures or asserts restore duration against the 2-hour RTO, or backup frequency/success against the 24-hour RPO beyond the unrelated `BACKUP_MAX_AGE_HOURS=25` missed-backup alert. The story's own success metric is unverified by its acceptance criteria.

7. **[high]** AC-1's concurrency resolution for the platform-operator bootstrap race is presented as a menu of options ("e.g., `SELECT ... FOR UPDATE`... or... a unique partial index... catch the unique-violation and retry") rather than a single mandated mechanism. Critically, if the unique partial index approach is chosen, a unique-violation on `is_platform_operator = true` occurring mid-transaction aborts that entire Postgres transaction (including the user's own registration INSERT) unless a `SAVEPOINT` is used — a detail the story never mentions. Implemented naively per the story's literal wording, the "loser" of the race could fail their entire registration instead of "still succeeding as an ordinary registration," directly contradicting AC-1's own requirement.

8. **[medium]** AC-7's advisory lock (`pg_try_advisory_lock(hashtext('backup:snapshot'))`) is described as "acquired at job start and released on completion/failure" with no mention of whether this is a session-level or transaction-level (`pg_advisory_xact_lock`) lock, nor how it behaves under pg-boss's connection pooling. A session-level lock held across a pooled connection that gets returned to the pool (and reused for an unrelated query) before the job explicitly releases it can either leak the lock (blocking all future backups) or be silently released early, defeating the concurrency guard AC-7 depends on.

9. **[medium]** No timeout is specified for the `pg_dump`/`pg_restore` subprocess. A hung subprocess (e.g., a stalled connection that doesn't cleanly error) would hold the AC-7 advisory lock indefinitely, permanently blocking all future scheduled and manual backups until an operator manually intervenes — with no alert defined for "backup job hung," only for "backup job failed" or "backup missed after 25h" (a much slower detection signal).

10. **[medium]** The encryption pipeline (pg_dump → gzip → worker_threads AES-256-GCM) is never specified as streaming. For a multi-GB production database, if the gzipped dump is buffered wholesale before being handed to the `worker_threads` Worker for encryption (as "hands the gzipped bytes to a Worker" reads), this risks OOM-killing the API process on large instances — undermining the very backup job meant to protect against data loss.

11. **[medium]** AC-10's "throwaway temporary database created for this check only" approach requires `CREATE DATABASE`-level privileges on `BACKUP_DATABASE_URL` and disk space equal to a full restore. No AC or task addresses guaranteed cleanup of this temporary database if the validation process crashes mid-run, creating a resource/disk-space leak risk (orphaned temp databases accumulating over repeated validate calls).

12. **[medium]** It's unclear whether `backup_runs` and `admin_alerts` — both newly created platform-level tables — are themselves included in the whole-instance `pg_dump`/`pg_restore` (D2 says "all tables," with no explicit exclusion). If they are included, restoring an old backup silently overwrites the current backup history/instanceId with stale data, potentially causing instanceId discontinuity and loss of the record of backups (and restores) that happened after the snapshot being restored — a self-referential consistency problem the story never addresses.

13. **[medium]** D1's JWT-based `isPlatformOperator` claim is populated "at JWT-verification time from `users.is_platform_operator`" — implying it's read from the DB per-request (good), but this isn't stated unambiguously; if it is instead baked into the JWT payload at issuance (as `orgRole` claims sometimes are, per common JWT patterns), then revoking platform-operator status from a compromised or offboarded operator would not take effect until that operator's existing tokens expire. The story should explicitly state this is a live DB lookup on every request, not a JWT claim, given the severity of the privilege being gated.

14. **[medium]** There is no step-up authentication (e.g., re-entering a password, MFA challenge) required for the destructive restore endpoint beyond a client-supplied `confirmRestore: true` boolean and free-text `reason` string — both trivially satisfied by any request bearing a valid, possibly-stolen platform-operator bearer token. Given the codebase already has MFA infrastructure, requiring a fresh MFA/step-up confirmation for whole-instance restore (the single most destructive action in the API) is conspicuously absent from the ACs.

15. **[low]** AC-9's prose describes operation order as "decrypts the file with `getBackupKey()`, verifies the SHA-256 checksum from the sidecar matches, runs `pg_restore`," implying decryption happens before checksum verification — yet the checksum is computed over the encrypted file (per AC-5: "checksumSha256 (SHA-256 of the final encrypted file)"). Best practice is to validate the checksum of untrusted ciphertext before feeding it to the decryption routine, not after. The story's stated ordering (decrypt-then-checksum) is both internally inconsistent with how the checksum was computed and a minor defense-in-depth regression (attempting decryption on unverified, potentially corrupted or adversarial bytes).

16. **[low]** `admin_alerts.payload` is an unconstrained `jsonb NOT NULL default {}` with no schema/size validation specified anywhere in the ACs. Combined with alerts being created automatically by system jobs (not directly operator-supplied), the risk is low, but nothing in the story bounds payload growth or shape, unlike the tightly-checked `status`/`severity`/`alertType` enums in the same table.

17. **[low]** D7's cross-org alert-recipient resolution loops over every row in `organizations` and calls `resolveRoutingRecipients` per org, once per hourly health check and once per failure event. The story acknowledges this "runs exactly once" today but bakes in an O(n) per-alert query pattern that will need revisiting at Story 9.2 multi-org scale — flagged as accepted technical debt rather than a currently-blocking issue, but worth naming explicitly as such in Dev Notes (it currently isn't).

18. **[low]** AC-19's exhaustive integration-test list (19 items) does not include a test for filename/path-traversal validation on the restore/validate routes (see finding #3), nor a test asserting `backup_runs`/`admin_alerts` are excluded from (or explicitly included in and reconciled after) the whole-instance dump/restore cycle (see finding #12) — both are meaningful coverage gaps in an otherwise very thorough test list.

19. **[low]** The story never states whether `pg_dump` is run in plain-SQL or custom (`-Fc`) format definitively — AC-5 says "in plain-SQL or custom format" as if either is acceptable, but the two formats have materially different restore mechanisms (`psql` vs `pg_restore`), different support for parallel restore, and different corruption-detection characteristics. Leaving this as an implementer's free choice risks inconsistency between what AC-9 documents as the restore tool ("pg_restore/psql") and what's actually produced — the ACs should commit to one format.
