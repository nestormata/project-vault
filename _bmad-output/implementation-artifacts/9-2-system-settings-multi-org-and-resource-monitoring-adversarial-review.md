# Adversarial Review: Story 9.2 — System Settings, Multi-Org & Resource Monitoring

- **Date:** 2026-07-06
- **Reviewed file:** `_bmad-output/implementation-artifacts/9-2-system-settings-multi-org-and-resource-monitoring.md`
- **Reviewer:** bmad-review-adversarial-general

## Findings

- **[critical] No MFA requirement specified on any of the six new platform-operator endpoints, despite them being the most privileged routes in the story.** AC-1/D2's authorization resolution specifies only `requirePlatformOperator()` + `requireOrgScope: false`; the word "MFA"/`requireMfa` does not appear anywhere in the story (verified: zero matches). Yet the pre-existing, *less*-sensitive org-scoped `POST /api/v1/admin/notifications/test` (`apps/api/src/modules/admin/routes.ts`) already sets `security: { requireMfa: true, ... }`. These new routes let a single compromised or coerced platform-operator session read/write instance-wide SMTP credentials (which control where password-reset and security-alert emails go), provision new organizations with owner accounts, and view cross-org resource data — all without a second factor. This is a strict security regression relative to the codebase's own existing precedent for a much lower-stakes endpoint, and no AC or Dev Note discusses or justifies the omission.

- **[critical] Storage-pressure "maintenance mode" (AC-17) has an unaddressed cross-tenant denial-of-service blast radius.** `audit_log_entries` monitoring (D5, AC-15) sums bytes across the entire instance, not per-org. Any single org/user with any write access to audited actions (e.g., scripted rapid credential reveals) can inflate total audit-log storage and push the *whole multi-tenant instance* into 95% maintenance mode (AC-17), silently suspending audit writes for every other org on the instance. Nothing in the story — not D5, not AC-15/16/17, not the Dev Notes — considers per-org rate limiting, fair-use protection, or even acknowledges that one tenant's behavior can degrade the audit guarantees of every other tenant sharing the instance. This is a genuine tenant-isolation gap dressed up as a purely operational "storage capacity" concern.

- **[high] AC-17's maintenance mode blanket-suppresses audit writes for security-critical event types, not just routine ones — verified against the actual chokepoint.** `apps/api/src/lib/audit-or-fail-closed.ts`'s `write*AuditEntryOrFailClosed` wrappers call straight through to the raw `writeHumanAuditEntry`/`writeMachineAuditEntry`/`writeSystemAuditEntry` (`apps/api/src/modules/audit/*.ts`) — the exact functions AC-17/Task 6 names as the interception point. Grep confirms these raw functions are *also* called directly (bypassing the fail-closed wrapper entirely) by `apps/api/src/modules/machine-users/rotation.ts`, `apps/api/src/modules/auth/mfa.ts`, and `apps/api/src/modules/auth/mfa-login.ts`. Placing maintenance-mode suppression at this shared low-level chokepoint means MFA-recovery-code-usage events and machine-key-rotation events are silently dropped during storage pressure — precisely the audit trail you most need intact during an anomaly (which storage pressure itself may be a symptom of, per the DoS finding above). The story never distinguishes "routine" vs. "security-critical" audit categories for suppression purposes.

- **[high] D7/AC-8's claim that this story creates "the first multi-org membership in the instance's history" is factually false, verified against the existing invitation-acceptance code path.** `apps/api/src/modules/invitations/token-routes.ts` (project invitation acceptance) already inserts an `org_memberships` row scoped to the invitation's `orgId` for the accepting user with no check that the user has no other org membership — i.e., a user who accepts project invitations from two different orgs today already ends up with two `org_memberships` rows. D7's own preamble even admits this ("org membership is a side effect of the first project invite a user accepts in that org") one paragraph before D7.3 contradicts it by calling this story "the first place in the codebase a single user can belong to org_memberships rows in more than one org." This factual error undermines the story's own risk-mitigation instruction ("verify no existing query assumes a user has exactly one org") — a dev agent could reasonably (and wrongly) conclude no such audit is actually necessary because the codebase has "never had to handle this before."

- **[high] `maxUsersPerOrg` is defined as an `instancePolicy` limit and alerted on (AC-13) but is never actually enforced anywhere in the ACs or Tasks — unlike its sibling `maxOrgs`, which has a dedicated enforcement AC (AC-10, 422 rejection).** The settings schema and `GET /admin/settings`/`GET /admin/resource-usage` responses present `maxOrgs` and `maxUsersPerOrg` as parallel, equally-authoritative instance limits, but only one is a hard cap; the other is purely advisory (an alert at 80/90/95%, never a rejection of the Nth+1 user join/invite). No AC, Dev Note, or Open Question calls out this asymmetry as an intentional v1 decision — a platform operator configuring `maxUsersPerOrg: 50` would reasonably expect it to be enforced, not merely monitored.

- **[high] Genuine RLS/tenant-isolation gap punted to "confirm at implementation time" rather than resolved by an AC or test.** D7.3 flags that `AuthContext.orgId` is per-session/per-JWT and cautions the dev agent to "search for `.orgId` usages that do a bare `users` join without an explicit `org_memberships` filter" — but this is prose guidance, not a testable AC, and no AC in the story (AC-1 through AC-28) actually verifies that a multi-org user's session correctly scopes to only their JWT's org across *existing* endpoints when `POST /admin/orgs` is the mechanism that (as shown above) doubles down on an already-possible multi-org-membership state. Given a security-sensitive credential vault, shipping a feature that increases the likelihood of multi-org users without a single regression test for cross-org data leakage is a meaningful gap.

- **[medium] AC-22's prescribed concurrency strategy contradicts D3's stated upsert strategy.** D3 resolves persistence as `INSERT ... ON CONFLICT (id) DO UPDATE`. AC-22 instead prescribes "`SELECT ... FOR UPDATE` then `UPDATE` with only the changed columns" and explicitly says this is "not a blind `UPDATE system_settings SET *`" — but a plain `INSERT ... ON CONFLICT DO UPDATE SET col = excluded.col` (listing only the changed columns) already achieves column-scoped, non-clobbering semantics without a separate `SELECT ... FOR UPDATE` step, and conversely `SELECT ... FOR UPDATE` cannot lock a row that doesn't exist yet (D3 states the row does not exist until the first `PUT`). The two prescribed persistence strategies are not obviously reconcilable, and the story never states whether the first-ever `PUT` takes a different code path (`INSERT`) than subsequent ones (`SELECT FOR UPDATE` + `UPDATE`).

- **[medium] AC-10's `maxOrgs` enforcement has no concurrency AC, unlike its sibling limits.** AC-23 explicitly handles the race for two concurrent `POST /admin/orgs` requests targeting the *same new email*, but no AC addresses the equally obvious race of two concurrent `POST /admin/orgs` requests both reading `orgs.current < maxOrgs` as true at the boundary (e.g., both see count=9, limit=10, both proceed) and jointly exceeding the configured limit. A count-then-insert check without a locking/serialization strategy (e.g., an advisory lock or a `CHECK` constraint enforced at the DB level) is a classic TOCTOU race, and the story specifies concurrency handling for the "same email" case but is silent on this one.

- **[medium] Interaction between AC-9's deactivated-owner rejection (409) and AC-23's unique-violation retry-as-existing-user is unspecified.** If two concurrent `POST /admin/orgs` calls race for the same brand-new email, AC-23 says the losing transaction "is caught and retried as 'existing user found'" — but by the time the retry re-queries `users` by email, that user now exists (created by the winning request) and could, in principle, be deactivated by some other concurrent action, or the retry logic might not re-apply AC-9's deactivated check at all. The story doesn't specify whether the AC-23 retry path re-runs the AC-9 validation, meaning the same logical request could non-deterministically return 201 or 409 depending on timing, with no AC covering this interaction.

- **[medium] The permanently-unresolvable `key_age_exceeded` alert (D8/AC-20) risks alert fatigue that undermines trust in the shared `admin_alerts` table used by every other alert type in this story.** Since no rotation-execution endpoint exists (D8, Open Question 3) and `key_rotated_at` can never advance past `initialized_at` within this story's scope, `key_custody_risk` (trigger `key_age_exceeded`) will fire for every instance older than `KEY_ROTATION_MAX_AGE_DAYS` and can *never* clear itself. The story documents this as an "accepted limitation" but proposes no snooze/acknowledge mechanism, and doesn't discuss the risk that operators will learn to ignore a chronically-unresolvable critical alert in a table (`admin_alerts`) that also carries genuinely actionable, time-sensitive alerts (`audit_storage.critical`, `resource.orgs_near_limit`).

- **[medium] AC-14 vs. AC-16's differing cross-org fan-out policy rests on a distinction the story's own Open Question #4 admits is unresolved.** AC-14 (`resource.orgs_near_limit`) explicitly ships with zero notification delivery ("no email fan-out... since there is no 'platform operator's own notification preferences' concept yet"), while AC-16 (`audit_storage.warning`/`.critical`) ships with full cross-org email/Slack/inbox fan-out, justified by "audit storage is instance-wide, affecting every org's ability to write audit entries" vs. "resource.orgs_near_limit... has no per-org relevance at all." Running out of org-provisioning headroom is arguably just as relevant to every org's future health as audit storage capacity, yet the two get materially different treatment. The story's own Open Question #4 flags this as unresolved for v1, but AC-14 still ships as a firm, testable behavior rather than an explicitly flagged placeholder — risking an arbitrary distinction hardening into permanent shipped behavior.

- **[medium] The D5 "100% coverage" claim will silently become false once Story 9.4 ships, with no enforcement mechanism to catch the gap.** D5 states current monitoring "covers 100% of the audit data that actually exists on the instance." The only mitigation for the future gap (once `platform_audit_events` exists) is a prose instruction to "flag this in that story's Dev Notes when 9.4 is written" — a purely manual, easily-dropped cross-story dependency with no failing test, lint rule, or tracked ticket reference forcing 9.4's author to actually extend `AUDIT_LOG_STORAGE_LIMIT_GB` monitoring to the new table.

- **[low] `account_recovery_tokens` reuse (D7) populates `initiatorOrgId` but never mentions populating `initiatorUserId`.** The schema (`packages/db/src/schema/account-recovery-tokens.ts`) has both an `initiatorUserId` (nullable FK to `users`) and `initiatorOrgId` (nullable FK to `organizations`) column, evidently intended to record *who* and *from which org* initiated an admin-driven token. D7's resolution only discusses setting `initiatorOrgId` ("the platform operator's own current org context, or NULL if they have none") and never mentions `initiatorUserId`, leaving the more precise per-operator audit attribution field unset for every org created by this feature — a minor but avoidable audit-trail gap given the schema already supports better attribution.

- **[low] Internal contradiction on whether a platform operator always belongs to an org.** D7.3 hedges with "...or NULL if they have none" (implying a platform operator might belong to zero orgs), while AC-11's edge case asserts organizations are "Never empty in practice — at least one org always exists once the platform operator has registered" (implying the platform operator's own registration always creates/joins an org, per Story 9.1's D1 bootstrap). The story never reconciles whether the platform-operator's own account is guaranteed to have an org context or not, despite this affecting exactly the `initiator_org_id` value discussed in the previous finding.

- **[low] No rate limiting specified for any of the six new endpoints**, including `POST /admin/orgs`, which triggers an outbound email (the new-owner invite flow, D7.4). The pre-existing `POST /admin/notifications/test` sets `rateLimit: { max: 10, ... }`; none of this story's routes mention a rate limit at all. Lower severity than the missing-MFA finding since these routes are platform-operator-gated, but still a departure from the existing pattern for a route that sends unsolicited email to third-party addresses (`ownerEmail`) supplied in the request body, which is itself a mild potential for the endpoint to be used as an email-sending oracle against arbitrary addresses if a platform-operator token is ever compromised or scripted incorrectly.

- **[low] AC-3's `"[configured]"` sentinel special-casing has no escape hatch.** The literal string `"[configured]"` is unconditionally treated as "password omitted" to guard against clients naively echoing back the masked `GET` response. This is reasonable as a footgun-guard, but the story doesn't acknowledge (even to dismiss) the corner case where an operator's actual desired SMTP password legitimately is the literal string `[configured]` — such a password could never be set through this API, silently and permanently.

## Resolution (addressed 2026-07-06, prior to merge)

Both `critical` and all four `high` findings were addressed directly in the story file before this branch was proposed for merge:

- **[critical] Missing MFA** — resolved. AC-1 and D2 now require `security: { requireOrgScope: false, requireMfa: true }` on all six new endpoints, with a dedicated negative example and a route-audit regression assertion. Task 2 updated to match.
- **[critical] Cross-tenant audit-storage DoS** — mitigated, not fully closed. New D10 adds: (a) a `SECURITY_CRITICAL_AUDIT_EVENT_TYPES` allowlist so MFA-recovery and machine-key-rotation events are never suppressed by maintenance mode, and (b) a per-org `topContributingOrgs` breakdown in the 90%/95% alert payload so the responsible org is immediately diagnosable. Full per-org rate limiting/quotas is explicitly out of scope for this story and tracked as Open Question 7 for a dedicated follow-up.
- **[high] Maintenance mode suppressing security-critical events** — resolved by the same D10 allowlist; AC-17 rewritten with a positive example proving MFA-recovery writes bypass suppression.
- **[high] False "first multi-org membership" claim** — corrected. D7 point 3 and AC-8's positive example now acknowledge that project-invitation acceptance (`token-routes.ts`) already permits multi-org membership today, and the risk-mitigation instruction was upgraded from implementation-time prose to a required regression test (AC-23b).
- **[high] `maxUsersPerOrg` unenforced asymmetry** — documented as an intentional v1 decision (D3 point 3, AC-13 note) rather than left as an unflagged gap; hard enforcement tracked as Open Question 6.
- **[high] RLS/tenant-isolation gap punted to prose** — resolved. New AC-23b requires a concrete regression test (query-level, not just end-to-end) proving a multi-org user's JWT-scoped session never leaks cross-org data across projects/credentials/service-endpoints. Referenced in Task 4 and AC-28's integration-test checklist.

All six `medium`/`low` findings remain open and unaddressed — they were judged non-blocking for merge but should be triaged during implementation or a future story pass.
