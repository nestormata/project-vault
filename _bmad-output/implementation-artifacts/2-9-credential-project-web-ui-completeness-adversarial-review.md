# Adversarial Review ‚Äî Story 2.9

**Date:** 2026-07-09
**Reviewed file:** `_bmad-output/implementation-artifacts/2-9-credential-project-web-ui-completeness.md`
**Reviewer:** bmad-review-adversarial-general

## Findings

- **[high] AC-A4's "corrected to the 7 real event types" is itself wrong ‚Äî it silently drops an 8th real `credential.*` event that satisfies the exact filter it's supposed to enumerate.** `packages/shared/src/constants/audit-events.ts` defines 11 `credential.*` event constants. Of those, `credential.bulk_import_initiated`/`_confirmed` are written with `resourceType: 'project'` (see `writeImportBatchAudit` in `apps/api/src/modules/credentials/routes.ts`) and `credential.search` is written with `resourceType: 'credential_metadata'` (`apps/api/src/modules/search/routes.ts`) ‚Äî correctly excluded from AC-A1's `WHERE resource_type = 'credential'` filter. But `credential.version_purged` (`apps/api/src/workers/prune-credential-versions.ts`) IS written with `resourceType: 'credential'` and `resourceId: candidate.credentialId` ‚Äî it satisfies AC-A1's query exactly ‚Äî yet AC-A4's "corrected" 7-item enum omits it. The story never justifies this exclusion (e.g. "system-actor housekeeping events are deliberately excluded from a human-activity feed" would be a defensible design note); as written it just miscounts, and the resulting query will silently under-report retention-pruning activity that already lives in the exact rows it's filtering for.

- **[high] The AC-A1/AC-A3 framing of "erased/pseudonymized" as a single fallback case conflates two backend behaviors that the cited helper itself documents as different.** `apps/api/src/modules/audit/actor-display-name.ts`'s own doc comment says pseudonymization is a *live, resolvable* alias ("a pseudonymized user's historical rows correctly show their alias after the fact") ‚Äî it does NOT hit the `'unknown'` fallback. Only a genuinely erased/hard-deleted token (no row at all) falls back to `'unknown'`. AC-A3's "an actor whose identity has since been erased/pseudonymized still resolves to the existing fallback label" is therefore only true for erasure, not pseudonymization ‚Äî testing this AC with a pseudonymized-actor fixture (as the name half-suggests) would show a real alias, not the fallback string, and a developer implementing strictly from the AC text could write a test that asserts the wrong thing for that case.

- **[medium] Cross-project isolation claim (AC-A1) is correct, but for a reason the story never states, and the review was specifically asked to check this.** `credentials.id` is `uuid('id').primaryKey().defaultRandom()` ‚Äî a single global primary key, not project- or org-scoped ‚Äî and `credentials.projectId` is a `NOT NULL` FK that no route in this codebase ever updates post-creation. So a resource_id collision between two projects' credentials is architecturally impossible, and `resource_id IN (thisProject'sCredentialIds)` is a sound proxy for `project_id = X`. This is good news, but the story asserts the design decision is "documented" without actually spelling out *why* it's safe (global UUID uniqueness + immutable projectId) ‚Äî an implementer or future auditor re-reading AC-A1 has to independently derive the same conclusion this review just did, which is exactly the kind of implicit reasoning the codebase's own "tenant/RLS context" scrutiny convention should have made explicit here.

- **[medium] Coordination risk with 3-5/4-5's shared migration-index collision is real but not caused by this story, and the story doesn't say so.** This story's own Project Structure Notes correctly claims "no migrations." Confirmed: `packages/db/src/migrations/` tops out at `0042` in this worktree, and `3-5-credential-expiry-notification-delivery.md`/`4-5-fine-grained-permissions-and-project-rbac.md` (both accessible sibling worktrees) each explicitly claim `0043` for themselves and already document the mutual collision and a "whoever merges second renumbers" protocol between each other. 2-9 has zero migration-index exposure to that collision since it adds no migration at all ‚Äî but the story is silent on this entirely, leaving a reviewer with no way to distinguish "verified clear" from "never considered" without independently checking sibling worktrees, which is precisely the ambiguity AGENTS.md's cross-story-dependency guidance asks stories to make explicit rather than leave implicit.

- **[medium] Task 1.2 instructs updating a function that has zero call sites in the current codebase, undermining the story's "read the real files, don't assume" rigor claim for exactly the file it cites.** `apps/web/src/routes/(app)/projects/[projectId]/credentials/+page.svelte`'s `filterHref` (line 33) is defined but never invoked anywhere in that file or the rest of the repo (verified via full-repo grep) ‚Äî it appears to be pre-existing dead code, not a live URL-builder wired into pagination or any other control. AC-F1/Task 1.2 tell the implementer to "update `filterHref`... to include `tags`" as if it's load-bearing; at best this is harmless dead-code churn, at worst it signals the story's line-by-line file citations (impressively accurate everywhere else ‚Äî the `+page.svelte` filter-form/empty-state/Clear-link line numbers all check out exactly) were not cross-checked against actual usage for this one function.

- **[medium] Group P's viewer/archived-project citation to "the existing archive-button-hidden-when-archived pattern" doesn't actually exist as described.** `apps/web/src/routes/(app)/projects/+page.svelte` has no button that gets *hidden* when a project is archived ‚Äî the "Archive" button is swapped for an "Unarchive" button (still visible, still clickable, for owners) when `isArchived` is true. The only genuinely hidden-when-archived element is the separate "View credentials" link (`{#if !project.isArchived}`). AC-P3's parenthetical citation is imprecise about which existing pattern it means, which matters because the two patterns imply different UI behavior (element removed vs. element swapped) and AC-P3 explicitly wants the former (edit-tags control removed, no replacement action offered) ‚Äî the analogy it reaches for doesn't cleanly support that.

- **[medium] The "unchanged" 200 response framing in the Background table is worded ambiguously enough to mislead a skim-reader about the wire contract.** "200 with `status:'unchanged'` (still `{data:...}`)" reads as if the HTTP response body contains a `status` field. It doesn't: `updateCredentialLifecycle`'s internal discriminator (`result.status === 'unchanged'`) never reaches the client ‚Äî `routes.ts` returns bare `{ data: result.data }` in both the unchanged and updated cases. A developer skimming just this line (rather than the actual route handler) could reasonably infer an API contract that doesn't exist and build client-side branching logic (e.g. "show a toast only if status !== 'unchanged'") that would never fire, since the client has no way to distinguish the two cases from the response alone.

- **[low] AC-D4's "viewers see the dependent-systems list but Groups L/V hide entirely" contrast is accurate for the routes but slightly overstates the read/write split as unique to Group D.** The GET dependencies route is genuinely `viewer`-minimum while POST/DELETE are `member`-minimum (confirmed in `routes.ts`), so the AC's core claim holds. But Group A's "Recent activity" section (viewer-visible, no write path at all) has an even simpler read/write split than Group D's list-vs-mutate distinction, and AC-D4 frames itself as "a distinct nuance from Groups L/V" without acknowledging Group A sits on the same read-only-for-viewers side of the line ‚Äî a minor completeness gap in the AC's own cross-referencing, not a functional bug.

- **[low] AC-P2's "no special rename affordance needed" claim about `tagDelta` is asserted, not demonstrated against the actual audit payload shape the reader would need to verify it.** The story cites `apps/api/src/lib/tags.ts`'s `tagDelta` by name but never quotes its actual diffing logic; from `updateCredentialTags`'s use of the same helper, the delta is a straightforward set-difference (added = new‚àíold, removed = old‚àínew), so the claim is very likely correct ‚Äî but the AC presents this as a verified fact ("proving 'rename' is just 'replace'") rather than citing the one line of code that proves it, which is a lower rigor bar than most of this story's other claims meet.

- **[low] Group I's "mirroring... the {#if projectId} branch used two lines above" is spatially approximate, not exact.** In the real `OnboardingStep3.svelte`, the credentials-link `{#if projectId}` block is the immediately preceding `<li>` (lines 55‚Äì63), and the Invite `<li>` starts at line 64 ‚Äî "two lines above" undercounts by roughly six lines. Cosmetic, but it's the one citation in an otherwise line-number-precise story that doesn't hold up to a literal reading.

- **[low] The Product Surface Contract's "Surface scope: both" framing for Groups A/G/S is arguably imprecise given Group G ships zero backend changes.** The contract table groups G with A/S as "`both` for the same reason (backend `suggestedActions` logic change + web rendering change)" ‚Äî but Group G (`DashboardPlaceholderGrid`) is purely a Svelte component + `+page.svelte` prop-passing change with no backend touch at all (confirmed: no route/schema citation anywhere in Group G's AC or Dev Notes, unlike A and S which both cite `dashboard-stats.ts` changes). Bundling G into the "both" justification alongside A/S's genuine backend touches is a minor mischaracterization of G's actual surface scope, even though the overall `both` designation for the story as a whole remains correct.

- **[low] No explicit statement that the Lifecycle/Dependent-systems/Add-version forms' fetch calls must scope `projectId`/`credentialId` from the authenticated route params rather than any client-supplied field**, despite this story adding roughly a dozen new state-changing requests. Backend authorization is the real gate and this is almost certainly inherited convention (every existing mutation already works this way), but given how detailed this story is about role-gating specifics elsewhere, its complete silence on this is notable by omission.

- **[low] No mention of what happens if a member submits the Lifecycle form, Dependent-systems form, or Add-version form while another tab has just archived the same project mid-session (TOCTOU between page load and submit) beyond the generic "the UI does not proactively know" disclaimer.** This is explicitly disclaimed as out of scope for the *initial* archived state, but the story doesn't distinguish that documented limitation from the ordinary concurrent-archival race, which is a slightly different (session-freshness) failure mode than "the UI never checks archival status at all."

- **[low] AC-G2's "no project selected" example doesn't reconcile with `CrossProjectEmptyState`'s own content**, which the AC doesn't inspect at all ‚Äî the review can't confirm whether `CrossProjectEmptyState` (rendered immediately above `DashboardPlaceholderGrid` in that branch) already communicates some of the same "nothing here yet" messaging the grid also shows, which would make the combination redundant rather than complementary. The story treats the two components as independent without checking for overlap.
