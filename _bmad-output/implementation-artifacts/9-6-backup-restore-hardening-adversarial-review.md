# Adversarial Review: Story 9.6 — Backup & Restore Hardening

**Date:** 2026-07-07
**Reviewed file:** `_bmad-output/implementation-artifacts/9-6-backup-restore-hardening.md`
**Reviewer:** bmad-review-adversarial-general

## Findings

- **[critical]** The reference implementation of `acquireRestoreLock()` given in D1.4 has an unguarded connection/lock leak. After `pg_try_advisory_lock` succeeds (`locked === true`), the function runs `await getDb().select({ id: backupRuns.id }).from(backupRuns).where(eq(backupRuns.status, 'running')).limit(1)` with **no try/catch/finally** around it. If that query throws for any reason (transient DB error, pool exhaustion on the separate pooled connection, query timeout), the exception propagates out of `acquireRestoreLock()` without ever calling `pg_advisory_unlock` or `reserved.release()`. The reserved connection is checked out of the pool permanently and the session-level advisory lock is held forever (until that leaked connection eventually dies), which — because restore and backup share the same lock key by design (D1.1) — would deadlock **every future restore attempt and every future backup trigger** until the process is restarted. AC-6 only enumerates lock-release guarantees for `restoreFromBackup()`'s five outcomes; it never covers a failure of this pre-check itself, so this failure mode has zero test coverage in AC-19.

- **[high]** Naming inconsistency between D1.4's type (`RestoreLockResult.reason: 'restore_in_progress' | 'backup_in_progress'`) and AC-2's example wire response (`"code": "restore_already_in_progress"`). The two literal strings for the restore-vs-restore case do not match (`restore_in_progress` vs. `restore_already_in_progress`), and nothing in the story states which is authoritative. An implementer following D1.4's code sample literally and a QA engineer writing an integration test against AC-2's literal JSON would produce two different, incompatible string constants.

- **[high]** No accounting for local-disk exhaustion during a prolonged S3 outage. Orphan cleanup (D3.6/AC-16) only deletes `.staged` files older than 24 hours. During an outage longer than 24h, every scheduled/manual backup attempt that fails will leave behind a new full-size encrypted dump on local disk (D3.5 says retries are exhausted then the file is retained), with no AC, size cap, or alert threshold covering cumulative staging-directory growth. The very feature meant to make S3 failures recoverable can turn a remote-service outage into a local-disk-exhaustion incident — a worse failure mode than exists today (today a failed upload leaves nothing on disk at all).

- **[high]** RLS/tenant-isolation assumption for the new `backup_runs` query is unverified within this story. `acquireRestoreLock()`'s post-lock check (D1.4) queries `backup_runs` via plain `getDb()` with no explicit org/tenant context, silently relying on Story 9.1's `EXCLUDED_TABLES` change to `check-rls-coverage.ts` still applying to this new call site. If that assumption is wrong for this particular session/query path, RLS could filter the `running` row to zero results, causing AC-3's core safety guarantee (block restore while a backup is running) to silently no-op instead of failing loudly. No AC or test asserts this specific query is unaffected by RLS/org scoping.

- **[high]** No failure isolation between the two unrelated concerns this story bundles into `runBackupHealthCheck()`: the `backup.missed` auto-resolve logic (D2) and the new hourly orphan-file-cleanup filesystem scan (D3.6). If the filesystem scan throws (permission error, missing mount, ENOSPC), it can abort the function before the alert raise/resolve logic runs, or vice versa — potentially masking the operator's single most important reliability signal (`backup.missed`) behind an unrelated, transient filesystem problem. No AC requires each step to be independently try/caught so one cannot take down the other.

- **[high]** Missing directory-creation/writability handling for `BACKUP_S3_STAGING_PATH`. Neither the design (D3) nor any AC states whether the code creates the staging directory on first use (mkdir -p) or what happens if an operator sets the env var but never provisions/mounts the path. Since staging now happens *before* upload (AC-12), this introduces a brand-new failure mode — the backup fails at the staging step itself — that does not exist in the current code, and it has zero coverage in AC-19's 20-item test list.

- **[medium]** Filename validation (the existing CWE-22 path-traversal guard) runs *after* lock acquisition, per AC-6's own edge case ("this happens after acquireRestoreLock() succeeds"). This means even a malformed or malicious `:filename` pays the full cost of `reserveConnection()` + advisory-lock round trip + a DB query before being rejected — inconsistent with AC-2's stated principle that rejection should be "the cheapest possible path," which the story only applies to lock-contention rejections, not basic input validation. A rapid loop of bad-filename requests against the platform-operator-only endpoint would repeatedly cycle the shared restore/backup lock for no legitimate reason.

- **[medium]** No audit logging of rejected restore attempts. AC-11 adds operational logging only for the alert auto-resolve path. Nothing in AC-1 through AC-7 requires logging the actor identity of a restore request that was rejected with `409` (`restore_in_progress`/`backup_in_progress`). For a secrets-vault product, an admin-triggered full-database-restore attempt — successful or blocked — is exactly the kind of security-sensitive event that should carry an audit trail, and this gap goes unaddressed anywhere in the story.

- **[medium]** The `packages/db/src/index.ts` refactor (Task 1.1: hoist the private `postgres()` client to module scope, add `reserveConnection()`) touches shared low-level infrastructure consumed far beyond the backup module, yet the Testing Standards section scopes coverage only to backup-specific tests (AC-19). No regression pass across other `getDb()` consumers is called for, which is a blast-radius risk disproportionate to the Dev Notes' framing that this story "touches only apps/api/src/modules/backup/**" (a claim the same Dev Notes paragraph then contradicts by listing `packages/db/src/index.ts` as an allowed touch point).

- **[medium]** The Product Surface Contract states "no new request/response shape visible to a client beyond one new `409` case," but the story actually introduces two distinct new `409` payload shapes on the restore endpoint — `restore_in_progress`/`restore_already_in_progress` (AC-2) and `backup_in_progress` (AC-3) — plus a schema union change (Task 1.3). The contract undercounts the client-visible surface change it itself claims to be scoping.

- **[medium]** Race-window mislabeling: because `pg_try_advisory_lock` is non-blocking, if restore's lock attempt lands during the brief moment `acquireBackupSlot()` holds its transaction-scoped xact-lock (the check-then-insert critical section for a new backup trigger), `acquireRestoreLock()` will report `reason: 'restore_in_progress'` even though no restore is actually running — a backup-trigger's transient lock hold gets mislabeled as a competing restore, giving the operator an actively misleading error code for the true cause.

- **[medium]** Orphan cleanup deletes any `.staged` file older than 24h unconditionally, with no coordination against a possible in-progress manual recovery. D3.7 explicitly tells operators their recovery path is to `aws s3 cp` the staged file themselves ("or simply let the next scheduled/manual backup supersede it"), yet nothing prevents the automated 24h sweep from deleting a file an operator is actively investigating or mid-way through manually re-uploading past that window.

- **[medium]** Overlapping hourly health-check ticks are explicitly acknowledged and handled for the alert-resolve logic (AC-9's idempotency edge case) but not for the orphan-cleanup scan added in the very same function: two overlapping ticks could both list the same `.staged` file and both attempt to delete it, with no stated handling for the resulting ENOENT-on-second-unlink race.

- **[medium]** Retryable-vs-non-retryable S3 error classification is incomplete. The story enumerates specific retryable codes (`SlowDown`, `RequestTimeout`, 5xx) and specific non-retryable codes (`InvalidAccessKeyId`, `SignatureDoesNotMatch`, `AccessDenied`, other 4xx), but never states a default for errors outside both lists (DNS failures, generic SDK `NetworkingError`, unknown error shapes) — leaving the implementer to guess whether "unknown" fails fast or retries, with no AC pinning down the default.

- **[low]** `SignatureDoesNotMatch` is bucketed as permanently non-retryable, but this error can also arise from transient server clock skew (a known AWS SigV4 gotcha) that self-corrects after NTP resync — grouping it with genuinely permanent errors like `InvalidAccessKeyId` may cause the system to give up on a condition that would have cleared on retry.

- **[low]** Retry backoff (500ms, then 1500ms) has no jitter. Low real-world impact for a presumably single-tenant self-hosted deployment, but a deviation from standard retry practice that could matter if this pattern is later copied to a multi-instance context.

- **[low]** The `release()` closure's `pg_advisory_unlock` call result is never checked. If it returns `false` (lock not actually held, e.g., due to some unaccounted-for prior release), that would silently mask a lock-lifecycle logic bug rather than surfacing it.

- **[low]** Three independent, coincidentally-close-but-different time thresholds appear in the story with no cross-referencing: AC-8's example uses `BACKUP_MAX_AGE_HOURS=25`, the PRD's RPO target (referenced in Dev Notes/References) is 24h, and the orphan-cleanup window (AC-16) is also 24h. None of these are actually linked, but the similar magnitudes invite operator confusion when tuning any one of them.

- **[low]** Delivery/blast-radius risk from bundling three independently-valuable, differently-risky fixes (restore concurrency lock, alert auto-resolve, S3 staging/retry/cleanup) into a single story with no stated sequencing or independent-shippability plan. A defect found in the most complex piece (S3 retry/staging, ~7 sub-items in D3) could hold up shipping the simplest, lowest-risk piece (the one-line `clearThresholdAlertEpisode` call in D2), and the Task list gives no guidance on splitting the work into separately mergeable units if that becomes necessary.
