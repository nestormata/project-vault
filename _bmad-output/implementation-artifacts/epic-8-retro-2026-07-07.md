## Executive Summary

Epic 8 shipped its four planned stories (8.1 Tamper-Evident Audit Log with HMAC Integrity, 8.2 Audit Log Search/Export/Forwarding, 8.3 Access Reports/Dormant Users/Audit PII Management, 8.4 Data Subject Erasure Request Handling) with strong engineering quality — HMAC-integrity design held up cleanly through every later consumer (including Epic 9's 9.4 prep work), and the GDPR erasure workflow correctly threaded a hard requirement (erase PII, preserve audit-log integrity) through one atomic transaction. Two further stories — 8.5 (Rotation Web UI Hardening) and 8.6 (Epic 7 Completion — Machine User Web UI & Hardening) — also landed under the `8-` key prefix; these are cross-epic closure stories for Epic 5 and Epic 7 respectively, not new Epic 8 scope (confirmed: zero references to either in `epics.md`'s Epic 8 section).

This retro's deeper audit found the epic's real risk isn't in what shipped — it's in a defect class that has now recurred for the fourth time running: **a "flag it in prose, catch it at the next retro" failure mode that the project has asked to be automated twice (Epic 6's P6-1, Epic 7's P7-1) and never built.** It showed up twice more inside Epic 8 itself: (1) 5 of 6 story files drifted out of sync with `sprint-status.yaml` again, and (2) Epic 8's own audit/compliance web UI gap was flagged identically by all four of its own stories (8.1 through 8.4) and still wasn't tracked in `deferred-work.md` going into this retro — the compliance epic carrying an untracked compliance gap.

Unlike previous retros, this one closes with the fix actually assigned to an owner already in motion: Nestor is personally implementing both the status-sync and PSC-enforcement guardrails via in-progress changes to the `pick-story` workflow.

## What Went Well

1. **The HMAC-integrity primitive (8.1) was built once and reused correctly by everything downstream.** `computeAuditHmac()` was consumed as-is by 8.2's export verification and is already referenced as the reuse target for Epic 9's 9.4 (Platform Operator Audit Log) prep — no rewrites needed anywhere.
2. **The GDPR erasure workflow (8.4) correctly solved a genuinely hard constraint**: erase a user's directly-identifying PII while preserving the audit log's own HMAC integrity, in one atomic transaction, with an explicit "what is NOT erased and why" list as its own compliance artifact.
3. **The team caught its own contradiction mid-story rather than shipping it**: 8.4 discovered migration 0001's `enforce_pseudonym_immutability` trigger made re-pseudonymization a true no-op, corrected the story's own AC-E8d wording instead of leaving the mismatch for someone else to find later.
4. **Adversarial review discipline held for 5 of 6 stories** with zero unresolved critical/high findings at each story's `done` sign-off.
5. **Spot-checking a story's own completion claims against live state (a practice Epic 7's retro asked for, N7-2/P7-3) actually happened this time and passed** — 8.6's claims about `main` branch protection and Marketplace-publish issue #112 were independently verified against live GitHub API/`gh` state during this retro and both held up, unlike Epic 7's own 7.2/D13 counterexample.

## What Didn't Go Well / Gaps, Debt & Risks

See the dedicated **Gaps, Inconsistencies & Risks Audit** section below for full evidence-backed detail. Summary for the narrative record:

1. Story-status/sprint-status drift recurred for the fourth time, and the automated check requested twice (Epic 6's P6-1, Epic 7's P7-1) was never built.

## Gaps, Inconsistencies & Risks Audit

### 1. [Critical] Story-status/sprint-status drift recurred a 4th time; the automated guard requested twice was never built
Five of Epic 8's six story files — `8-2-audit-log-search-export-and-external-forwarding.md` (`Status: review`), `8-3-access-reports-dormant-users-and-audit-pii-management.md` (`ready-for-dev`), `8-4-data-subject-erasure-request-handling.md` (`review`), `8-5-rotation-web-ui-hardening.md` (`review`), `8-6-epic-7-completion-machine-user-web-ui-and-hardening.md` (`review`) — disagreed with `sprint-status.yaml`, which marks all six `8-*` keys `done`, at the start of this retro. This is the exact defect class Epic 4's retro fixed once, Epic 6's retro fixed again in one sweep for `6-1/6-2/7-1/7-2/7-3/8-1` (per `epic-6-retro-2026-07-06.md`'s addendum), and whose own action item **P6-1** explicitly said: *"Add an automated CI check... this is the third time this exact drift has been caught manually."* No such check exists in the repo's `Makefile` (confirmed by direct inspection). Epic 7's retro (P7-1) asked for a related enforcement mechanism and it also was never built. This is a violation of `product-surface-contract.md`'s P3 rule and blocks G2 gate condition 1.
**Resolution:** Fixed directly during this retro (all 5 files corrected to `Status: done`, see File List). Structural fix — an actual `make` target diffing story-file headers against `sprint-status.yaml`, wired into `make ci` — is now owned by Nestor personally (P8-1), in progress via `pick-story` workflow changes, with the team's suggestion to add a CI-level safety net specifically for stories that are already `done` in sprint-status but whose header was never touched (a case `pick-story`'s own flow won't catch, since it only fires when a story is *picked up*, not retroactively).

### 2. [Critical] Epic 8's own Product Surface Contract UI gap was flagged 4 times by its own stories and was untracked going into this retro
Stories 8.1, 8.2, 8.3, and 8.4 each contain a `## Product Surface Contract` section explicitly flagging, in near-identical language, that no dedicated web UI exists anywhere in `epics.md` for audit search/export/forwarding configuration, point-in-time access reports, dormant-user admin actions, or the erasure-request workflow — and each one defers resolution to "Epic 8 sprint planning/retrospective." `apps/web/src/routes` has zero routes for any of these surfaces (confirmed via `find`). Unlike the directly analogous gaps in Epic 5 (→ Story 5.4), Epic 6 (→ Story 6-4), and Epic 7 (→ Story 8-6), all of which were converted into `deferred-work.md`'s "Web UI gaps" table once their retros ran, this gap was not in that table as of this retro's start — violating P5-2/P5-3 for a third consecutive epic, this time inside the compliance/governance epic itself. This blocks G2 gate condition 2.
**Resolution:** Scoped and confirmed with Nestor during this retro as closure story **`8-7-epic-8-completion-audit-compliance-web-ui-and-technical-debt`** (backlog), bundling: audit log search/filter UI + export/forwarding/retention config screens (8.1/8.2), point-in-time access-report UI + dormant-user alert admin actions (8.3), and the erasure-request review→confirm→execute flow + compliance-report download (8.4). `epic-8` stays `in-progress` until it lands, matching the 5.4/5.5, 6-4, and 8-6 precedent. To be formally scheduled in `sprint-status.yaml`/`deferred-work.md` as a follow-up step to this retro (see Next Steps) — not written to either file during the ceremony itself, per this project's standing rule that story creation is `create-story`'s job, not the retro's.

### 3. [High] Story 8.6 has no adversarial-review artifact despite shipping destructive state changes
Unlike 8.1 through 8.5, each paired with a dedicated `*-adversarial-review.md` file, `8-6-epic-7-completion-machine-user-web-ui-and-hardening.md` has none — despite shipping API-key revocation, machine-user deactivation, and a rate-limit-affecting dormancy-alert action, all squarely "destructive/irreversible state changes" under the project's own Team Agreement (adopted Epic 4/5, held through Epic 6/7: "A story touching authorization, session lifecycle, or destructive/irreversible state changes gets a formal adversarial review"). Only an ordinary code-review pass is evidenced (commit `7871466`, "require confirmation before revoking a dormant key").
**Resolution:** Retroactive adversarial review confirmed with Nestor during this retro, owned by Dana (QA). Tracked as A8-2, independent of Story 8-7 so it doesn't block 8-7's own UI scope if it surfaces findings needing discussion.

### 4. [Medium] `epic-8` in `sprint-status.yaml` conflates Epic 8's own scope with two unrelated closure stories, undocumented
`epics.md` defines Epic 8 as exactly Stories 8.1–8.4 (confirmed: zero references to `8.5`/`8.6`/`8-5`/`8-6` anywhere in the document). But `8-5` and `8-6` close out Epic 5 and Epic 7 respectively — a deliberate, `deferred-work.md`-documented pattern, not a mistake — yet a reader relying on `epics.md` alone would believe Epic 8 is 4 stories, while `sprint-status.yaml` lists 6 under the same prefix. Additionally, `epic-8:` carries no explanatory inline comment despite (per Finding 2) legitimately needing to stay `in-progress`, unlike the precedent set by `epic-5`/`epic-6`/`epic-7`'s lines.
**Resolution:** Tracked as A8-4 — add the explanatory comment to `epic-8`'s line once Story 8-7 is formally scheduled (mirrors A7-2's fix pattern). No epics.md rewrite proposed; the closure-story pattern itself already document that 8.5/8.6-style stories intentionally live outside their host epic's own numbered scope.

### 5. [Low] `epics.md` still says `audit_events` in ~23 places vs. the shipped, twice-reconciled `audit_log_entries`
Story 7.1's D9 and Story 8.1's D1 both independently document the same naming divergence with explicit rationale ("shipped code wins," matching the project's established pattern for resolving `epics.md`-vs-shipped-code conflicts). This is not silent drift — it's a deliberate, well-reasoned decision — but it was never added to `deferred-work.md`'s "Planning document reconciliation" table alongside the directly analogous `secrets`→`credentials` entry from Epic 2's retro.
**Resolution:** Tracked as A8-5 — add one line to `deferred-work.md`'s reconciliation table. No story needed.

## Team Agreements (carried forward + new)

1. (Carried from Epic 4/5) A story touching authorization, session lifecycle, or destructive/irreversible state changes gets a formal adversarial review — Story 8.6 is this agreement's first confirmed miss, now being corrected retroactively (A8-2).

3. (Carried from Epic 6, P6-2) Don't let a new epic's stories advance while the previous epic's retrospective is still `optional` without an explicit, logged decision to defer it.
4. (New) When a retro action item calls for automated tooling, assign a real owner and a concrete artifact (a script, a `make` target, a checklist gate) — not just a restated ask. Two consecutive retros (6, 7) asked for the same CI check without either landing it.
5. (New) Spot-check at least one Completion Notes claim per story against live/external state before trusting a story's own self-report, especially for stories claiming external-system configuration (branch protection, issue filing, tags) — this retro did it and it held up.
