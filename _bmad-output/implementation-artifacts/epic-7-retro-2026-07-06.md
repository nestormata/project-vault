## Executive Summary

Epic 7 shipped all three planned stories (7.1 Machine User Identity & API Key Management, 7.2 Machine User Authentication & Programmatic Secret Retrieval, 7.3 GitHub Actions CI/CD Integration) with 100% adversarial-review coverage, continuing the discipline established in Epic 5. The hardest security decision in the epic — machine-JWT signing (7.2 D3), flagged **critical** at spec time because the assumed library API didn't exist in the installed version — was resolved cleanly at dev time with a vetted fallback and dedicated forgery/tamper/alg-confusion tests. The 7.3 code-review pass also proactively closed the adversarial review's release-freshness gap.

However, this retro's deeper audit — comparing story-file claims against sprint-status, git history, and live repo state rather than trusting each story's own Completion Notes — surfaced several real, currently-open problems that self-review did not catch:

1. **A story's own "done" claim was factually false and only caught by independent code review, not by the story's authors.** Story 7.2's Dev Agent Record asserts D13 (the cache-activation beacon) was "implemented per the story's resolution." It wasn't — the server-side route didn't exist. Every offline-agent activation beacon was silently swallowed until a later code-review pass (commit `6fb8d92`) caught it as **critical**.
2. **This project's own governance rule (P3) is currently violated for all three Epic 7 stories.** Story-file `Status:` headers all read `review`; `sprint-status.yaml` marks all three `done`. P3 requires these to match on every transition.
3. **Epic 7 cannot legitimately close per this project's own G2 gate**, and — unlike Epic 5's precedent — nothing in `sprint-status.yaml` documents *why* it's held open, so the correct-looking "in-progress" status reads as unexplained rather than deliberate.
4. **The exact process failure Epic 5's retro created rules to prevent has recurred, unaddressed, through both Epic 6 (no retro run) and Epic 7.** The missing machine-user management web UI was flagged as a "genuine planning gap" in both 7.1 and 7.2, exactly like Story 5.1's rotation-UI gap — but unlike that gap, it was never turned into a tracked `deferred-work.md`/backlog entry, violating P5-2 and P5-3 from Epic 5's own retro.
5. **A story's acceptance criterion is admittedly, explicitly unmet, with no tracking artifact created.** Story 7.3's AC-15 requires a named owner + linked tracked issue for the manual GitHub Marketplace publish step. The dev's own notes say this "cannot be considered fully done" without it — and no such issue exists (verified via `gh issue list`), no `vault-action-v*` git tag has ever been pushed, and the branch-protection mitigation the same story calls mandatory is not configured on `main` (verified via GitHub API).

None of this is attributed to any individual — it's a pattern of self-reported completion outrunning verification, three epics running in the same window. See Action Items.

## What Went Well

1. **A critical, explicitly-unresolved-at-spec-time security decision was properly closed at dev time.** 7.2's D3 admitted it couldn't confirm whether `@fastify/jwt`'s `namespace` option existed in the installed version, and left a "roll your own JWT crypto" fallback as a real possibility — exactly the kind of high-risk pattern the adversarial review flagged as **critical**. At dev time, the team correctly discovered the installed `@fastify/jwt@10.1.0` lacked `namespace`, and rather than hand-rolling JWT crypto, used `fast-jwt` (an already-vetted, already-in-the-lockfile transitive dependency) directly, with dedicated tests for tamper, cross-secret forgery, `alg: none` algorithm-confusion, and expiry rejection.
2. **The 7.3 adversarial review's release-freshness gap was proactively fixed, not just noted.** The review flagged that `vault-action-release.yml` never re-verified `dist/` freshness against the actual tagged ref before moving the mutable `v1` tag. The shipped implementation explicitly closes this ("closing the gap the adversarial review flagged" — Completion Notes).
3. **Architecture-conflict-resolution discipline (D1..D13-style tables) held across all three stories**, consistently resolving epics.md-vs-architecture.md-vs-shipped-code conflicts (hashing algorithm, key format, JWT algorithm, error envelope shape, audit event naming) with explicit rationale rather than silent, inconsistent choices — the same discipline Story 4.1 established, still holding two epics later.
4. **A real security fix from outside the epic's own review process was already applied and kept green.** Today's `undici` override (`e17e8ad`) closed the exact transitive-dependency risk 7.3's own Completion Notes flagged and explicitly accepted ("Trivy flagged... pulled in transitively via `@actions/core`") — and the corresponding `vault-action/dist/` rebuild (`bfcdae1`) was done same-day to keep the freshness check green, showing the CI gate actually caught the staleness it exists to catch.
5. **`jscpd` duplication discipline kept paying off**, extracting shared pagination/mapper helpers (7.1), a shared key-validity helper (post-review dedup, 7.2), and shared manual-auth security config (7.2) — the module got cleaner as the epic progressed, matching Epic 5's own "What Went Well" pattern.

## What Didn't Go Well

1. **A story's Dev Agent Record made a false completion claim, and it was caught by a later independent code-review pass, not by the story's own process.** Story 7.2's own notes assert D13 (`machine_cache.activated` beacon) was "implemented per the story's resolution." The server-side `POST /api/v1/machine/cache-activated` route, its route-governance entries, and its audit/alert wiring did not exist — every offline-agent activation beacon was silently swallowed by a generic fetch failure, so FR38's alert and the AC-15 audit row never fired in production use. This was only caught by a subsequent code-review pass (`6fb8d92`, flagged **critical** by "Acceptance Auditor" and **high** by "Blind Hunter"). This means at least one story in this epic reached "done" in `sprint-status.yaml` while its own self-reported Completion Notes were factually wrong about what shipped — a gap between claimed and actual completeness that this epic's own process did not catch internally.
2. **This project's own P3 rule ("story file `Status:` must match `sprint-status.yaml`") is currently violated for all three stories.** `7-1-*.md`, `7-2-*.md`, and `7-3-*.md` all still say `Status: review` at the top of the file; `sprint-status.yaml` marks all three `7-1-...`, `7-2-...`, `7-3-...` keys `done`. This is not a one-off typo — it's uniform across the whole epic, suggesting the sprint-status flip to `done` happened without the corresponding story-file edit in every case.
3. **The exact process failure Epic 5's retro created rules to prevent recurred, twice over.** Epic 5's retro (P5-2, P5-3) established: (a) a "flag it, don't do it" deferral must become a tracked backlog entry, not just prose; (b) a Product Surface Contract "TBD" (missing UI story) must be resolved at story-creation time, not carried to the epic retro. Both 7.1 and 7.2 flagged the missing machine-user management web UI as a "genuine planning gap" and explicitly said it should be "escalated" — but neither Epic 6 (whose optional retro was never run) nor Epic 7 ever turned it into a `deferred-work.md` entry or a scheduled story, unlike the rotation-UI gap that Epic 5's own retro resolved for itself in the same pattern. It's still nowhere in `epics.md` for Epic 8 or Epic 9 either — it will be a permanently curl-only admin feature unless someone acts on it now.
4. **Epic 7 sits at `in-progress` for a legitimate reason, but that reason is undocumented.** Per this project's own `product-surface-contract.md` G2 gate, an epic cannot move to `done` until every PRD user journey is shipped, honestly placeholdered, **or explicitly deferred with a linked follow-up story still tracked** — and no such linked story exists for the machine-user UI gap, so `epic-7: in-progress` is actually correct. But unlike Epic 5's line (`epic-5: in-progress # held open pending 5-4 (rotation workflow web UI, scheduled at retro)`), Epic 7's `sprint-status.yaml` line has no explanatory comment at all — a future reader (or the next retro) has no way to tell "correctly held open per G2" from "someone forgot to flip it."
5. **Story 7.3's own AC-15 sub-requirement is admittedly unmet, and nothing tracks it.** The story requires the Marketplace-publish manual follow-up to be recorded with a named owner and a linked tracked issue — its own Completion Notes say this "cannot be considered fully done per AC-15's letter" without it. No GitHub issue exists (checked), no release tag has ever been pushed, and the mandatory branch-protection mitigation for the mutable `v1` tag (D7) is not configured on `main` (checked via GitHub API — "Branch not protected"). The action is, today, unreleased, unreleasable-safely, and untracked as an open item anywhere outside the story file's own prose.

## Significant Discovery Alert — Risk / Gap / Contradiction / Technical-Debt Audit

Per Nestor's explicit request, this section goes beyond "what the adversarial reviews already said" and cross-checks story claims against live repo state (git log, `gh` CLI, GitHub API, `epics.md`, `deferred-work.md`).

### Process / governance violations (new — found by this retro, not by any prior review)

- **P3 violation:** all three story files (`7-1-*.md`, `7-2-*.md`, `7-3-*.md`) have `Status: review` in their header while `sprint-status.yaml` marks all three `done`. Per `product-surface-contract.md`'s P3 rule, these must match on every transition. *(Epic 7, all 3 stories)*
- **G2 gate — epic correctly blocked, but undocumented:** the epic cannot move to `done` because the machine-user UI gap has no linked follow-up story (G2 condition 2), but `sprint-status.yaml`'s `epic-7` line carries no comment explaining this, unlike `epic-5`'s explicit precedent comment. *(Epic 7)*
- **A story's self-reported Completion Notes were factually wrong about what shipped**, only caught by a later, separate code-review pass (`6fb8d92`) — see "What Didn't Go Well" #1. This is a process-integrity finding, not just a bug: it means "Completion Notes say X was implemented" cannot be taken as verified fact for at least one story in this epic. *(Story 7.2, D13)*
- **P5-2/P5-3 violated:** the machine-user management web UI gap (flagged in both 7.1 and 7.2) was never added to `deferred-work.md`'s "Web UI gaps" table and never scheduled as a story, unlike the directly analogous Epic 5 rotation-UI gap that Epic 5's own retro resolved for itself. *(Epic 7, carried through Epic 6 unaddressed)*
- **Story 7.3's AC-15 is admittedly, explicitly unmet:** no GitHub issue tracks the Marketplace-publish manual follow-up (verified: `gh issue list` returns none matching), no `vault-action-v*` tag has ever been pushed (verified: `git tag -l` empty), and the mandatory branch-protection mitigation for the mutable `v1` tag is not configured on `main` (verified: GitHub API returns "Branch not protected"). The published action does not exist yet in any usable form for an external consumer. *(Story 7.3, D7/AC-15)*

### Security risks (carried from adversarial review, confirmed still open at ship time)

- **Offline-cache concurrent-write race is untested.** 7.2's adversarial review flagged (high) that `packages/agent`'s shared cache file has no documented protection against concurrent CI processes on the same host writing to it simultaneously — directly relevant to the epic's own stated primary use case. The story's own Completion Notes admit "a true multi-process race test was judged lower-value... given time constraints" — an accepted, not a fixed, gap.
- **Per-`keyHash` brute-force rate limit is close to a no-op against the threat it claims to defend**, since HMAC-SHA256 is deterministic and a real guessing attack produces a different hash almost every attempt (adversarial review, high — not revisited or corrected in implementation; real defense is the 256-bit key's entropy, not this limiter).
- **No rate limiting or anomaly detection distinguishes failed vs. successful lookups on the machine credential-value endpoint** — a stolen (but not yet revoked) machine JWT can enumerate credential names within its scoped project at the endpoint's normal throughput, with no separate failed-lookup throttle unlike the key-exchange endpoint's explicit failed-attempt limiter (adversarial review, medium — no evidence in Completion Notes this was addressed).
- **Mutable `v1` tag with no branch protection** (see governance section above) — a real, currently-live supply-chain exposure the moment this action is actually tagged and adopted by any consumer.

### Technical debt (acknowledged in-story, still open)

- **No machine-user deactivation/deletion path exists anywhere in this epic.** `machine_users.deactivatedAt` is schema-only; nothing sets it. Once created, a machine user (and its role/scope) can only be neutralized by revoking its keys one at a time — there is no way to remove a mistakenly-created or wrong-project machine user in v1.
- **Duplicated AES-256-GCM/HKDF cache-crypto now lives in three independently-versioned artifacts** (`packages/crypto`, `packages/agent`, and `packages/vault-action`'s bundled `dist/`) with zero automated mechanism tying a future security fix in one to a mandatory rebuild of the other two — explicitly flagged as technical debt in both 7.2 and 7.3, not resolved.
- **Dormancy-threshold changes have no reconciliation of already-fired alerts** — raising the threshold (e.g. 90→180 days) leaves stale alerts sitting in the admin queue with no defined cleanup path (accepted design gap, both adversarial review and Dev Notes agree).
- **No npm-publish CI pipeline exists** for `packages/agent`, the one workspace package marked `"private": false` — an explicitly flagged open question in both 7.2 and 7.3, unresolved.
- **FR101/FR110 have no PRD-level entry**, only `epics.md` annotations — a pre-existing traceability gap explicitly carried forward, not fixed, by this epic.

## Key Takeaways

1. Epic 7's technical execution was strong — the hardest security call in the epic (machine JWT signing) was resolved correctly under real constraints, and the 7.3 team proactively closed a review-flagged release-safety gap before it was ever exploited.
2. But this epic's *process* regressed on two fronts Epic 5 explicitly fixed: an untracked UI-gap deferral (P5-2/P5-3) and — new this epic — a self-reported completion claim that was simply wrong until an independent review caught it. Both point to the same root cause: agreements written into a retro doc with no mechanism forcing anyone to check them later.
3. Skipping Epic 6's retrospective (marked `optional`, never run) meant the Epic 5 agreements went a full extra epic without anyone checking whether they were holding — this retro is the first checkpoint since Epic 5, and it found real slippage.
4. Epic 7 is currently `in-progress` for a legitimate, rule-based reason (G2's UI-follow-up requirement) — but until today it looked identical to "someone forgot," because the reasoning wasn't written down. That's now fixed for this epic; the underlying process gap (P7-1) is not yet fixed for future epics.
