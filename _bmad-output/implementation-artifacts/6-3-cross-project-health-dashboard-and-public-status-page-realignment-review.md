# Adversarial Review: Story 6.3 Realignment Pass

- **Date:** 2026-07-04
- **Reviewed file:** `_bmad-output/implementation-artifacts/6-3-cross-project-health-dashboard-and-public-status-page.md`
- **Reviewer:** bmad-review-adversarial-general
- **Scope note:** This review targets the **realignment pass** specifically (the 2026-07-04 rewrite that swaps the story's data source from the speculative `payment_records`/`getServiceHealthStatuses` contract to Story 6.2's actual shipped `service_endpoints` table), not the original speculative draft. Findings below focus on whether the rewritten content is internally consistent, whether it accurately reflects what Story 6.2 shipped (verified independently against `6-2-http-endpoint-monitoring-and-availability-alerts.md`), and whether the realignment left any dangling artifacts of the old model.

## Findings

- **[critical]** AC 12 (public unauthenticated `GET /api/v1/status-pages/:token`, Section E) states the response must never include "the underlying `payment_records.name`/`url`" as one of the forbidden internal identifiers. This is a leftover reference to the rejected pre-realignment data model. Under the realigned story, `status_page_services.service_id` is now an FK to `service_endpoints.id` (AC 19), not `payment_records.id` — the underlying record whose `name`/`url` must never leak is a `service_endpoints` row, not a `payment_records` row. This directly contradicts the realignment pass's own header claim that "every other AC/Task affected by the `payment_records` → `service_endpoints` swap is updated in place." An implementer or test author (Task 11's "explicit negative assertion — grep the serialized response for forbidden fields") could reasonably build the privacy check against the wrong table's fields based on this text.

- **[high]** No AC or Task anywhere in the document defines an admin-facing "read current status page configuration" endpoint (e.g., `GET /api/v1/projects/:projectId/status-page`). Sections C–H cover enable (POST), regenerate (POST), public read (GET, unauthenticated), update (PUT), disable (DELETE), and audit — but there is no endpoint for a project owner/org owner to fetch whether a status page is currently enabled and which services/display names are already configured. Task 9's admin UI ("service picker + display-name inputs calling PUT") cannot pre-populate its picker with already-configured services, and Riley's persona journey (which describes opening project settings and seeing existing configuration) has no backing API to load that state from. The public GET (AC 12) deliberately excludes internal `serviceId`s and can't serve this purpose either.

- **[high]** AC 19's migration is documented as landing at "next free number — re-check `_journal.json`, `0029` as of this story's writing." Story 6.2's own migration task (`6-2-...md` Task 2) independently states the identical "`0029` as of this story's writing" for its own three-table migration. Since ADR-6.3-01 makes Story 6.2 completion ("`done`," not just `ready-for-dev`) a hard prerequisite for starting 6.3, migration `0029` will almost certainly already be consumed by Story 6.2's migration by the time anyone works on 6.3 — this isn't a generic "verify before merging" caveat, it's a near-certain collision given the story's own stated sequencing. The realignment pass, which specifically re-examined cross-story consistency with 6.2, did not catch or call out this near-guaranteed collision.

- **[high]** AC 8 (enable a status page, `POST .../status-page`) documents a `409 status_page_already_enabled` for a second enable call, backed by the `UNIQUE(project_id)` constraint as "the backstop." Unlike AC 11 (regenerate), which explicitly walks through and accepts the concurrent-double-call race condition, AC 8 never addresses what happens when two concurrent first-time `POST` calls both pass the "no existing status page" check and then both attempt to insert — one will hit the unique-constraint violation at the DB level. Nothing in AC 8 specifies that this violation must be caught and mapped to the documented `409` rather than surfacing as an unhandled `500`, unlike AC 11's thorough concurrency treatment for the analogous regenerate race.

- **[medium]** ADR-6.3-02/Task 3 instructs: "`status`/`lastCheckedAt` typed exactly as 6.2's `service_endpoints.status`/`lastCheckedAt` columns (import the enum/type from 6.2's schema if exported, do not redeclare a diverging union)." However, Story 6.2's own schema task defines `status` as a plain `text NOT NULL DEFAULT 'healthy'` column with a `CHECK IN ('healthy','degraded','down')` constraint — not a Drizzle enum with an exported TS union type. Nothing in 6.2 commits to exporting a reusable status union type. If none exists when 6.2 ships, 6.3 will have no choice but to declare its own union type — precisely the "diverging union" duplication risk this instruction was trying to avoid, and Task 0's re-verification step doesn't flag this possibility explicitly.

- **[medium]** ADR-6.3-02 cites Story 6.2's `getBatchedProjectServiceHealthStats` (6.2's own Task 8 helper) as "a useful precedent for the batched-by-project query shape this story's Task 3 should mirror." That helper returns aggregated status *counts* grouped by project (a `Map<projectId, {healthy, degraded, down}>`), which is a materially different shape from what AC 1 actually requires: a flat list of individual services (`id`, `name`, `status`, `lastCheckedAt`) grouped by project. Calling this a "useful precedent" for the query *shape* overstates its reusability and could mislead an implementer toward the wrong query pattern before realizing the two needs diverge.

- **[medium]** MFA is required for enable (AC 9) and regenerate (AC 11) — described as protecting actions that "create a new externally-shareable secret" — but not for `PUT .../status-page` (AC 15), which can add, remove, or rename which services and display names are shown on an already-live, publicly shared page. The story never addresses why altering the *content* of an existing public page is considered lower-risk than minting the token itself; a compromised session lacking MFA step-up could still materially change what an unauthenticated external audience sees.

- **[medium]** ADR-6.3-02 instructs: "Before starting Task 2/3 below, confirm `service_endpoints`'s actual shipped columns match this description." Task 2, however, has been entirely removed by this same realignment pass ("Removed (realignment pass, ADR-6.3-02/03)... Nothing to implement here"). The instruction to gate work "before starting Task 2/3" is now a dangling reference to a task with no content.

- **[medium]** AC 15's validation for `displayName` only rejects "empty or exceeds 100 chars" — there is no trim/whitespace-only rejection. A caller could submit a `displayName` consisting entirely of spaces, passing validation but rendering as a blank label on the public, unauthenticated status page (customer-facing content with no visible error to the admin who configured it).

- **[medium]** AC 17's audit-logging section documents a "deleted-row snapshot" requirement for `STATUS_PAGE_DISABLED` (capturing which `status_page_services` display names existed at time of deletion, since the row is hard-deleted with no tombstone) but specifies no equivalent snapshot for `STATUS_PAGE_UPDATED`. Since `PUT` performs a full delete-all-then-insert replace of `status_page_services` (AC 15), an update audit event that doesn't capture the *previous* service list/display names loses exactly the same "what was publicly visible before this destructive overwrite" information that the disable-audit snapshot was explicitly designed to preserve — an inconsistent standard applied to two structurally similar destructive operations.

- **[low]** AC numbering uses two different conventions for evolving the list within the same realignment pass: AC 3 and AC 7 are explicitly retired and skipped ("the number is retired rather than reused, so downstream AC/Task cross-references... don't need renumbering"), while a new AC introduced in the same hardening effort is inserted as "AC 10a" rather than being appended as a new top-level number or folded into an existing AC. The document doesn't explain why one change used retire-and-skip and the other used suffix-insertion, which could cause confusion when counting or referencing "the ACs" as a set.

- **[low]** Known Scope Boundaries accepts, simultaneously, zero audit logging for the public GET and zero enumeration/abuse-metrics instrumentation beyond a bare per-IP rate limit. Taken together, an org admin has no way to ever learn how many times, from where, or how aggressively their externally shared status page link has been accessed or probed. Each omission is individually justified in isolation, but the story doesn't acknowledge the compounding effect of accepting both simultaneously for a public, unauthenticated, externally-shareable surface.

- **[low]** Task 0's residual verification step only mentions adjusting "Task 3's query" if a `service_endpoints` column is renamed during 6.2's actual implementation. It doesn't mention that a rename would also ripple into the AC 19 migration's FK column reference, the Task 5 schema file's column mapping, and any shared Zod type imported per Task 3's guidance — the blast radius of a rename is understated relative to what's actually scoped to be checked.

- **[low]** The realignment-pass header comment's claim that "every other AC/Task affected by the `payment_records` → `service_endpoints` swap is updated in place" is contradicted by the critical finding above (AC 12's stale `payment_records.name`/`url` reference), which suggests the update was done by manual review of the ADRs/major sections rather than a systematic search for every latent reference to the old model — worth a follow-up grep pass before this story is picked up for implementation.
