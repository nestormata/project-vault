## Findings

- **[critical] Group D's central "verified fact" is false — reproduced empirically.** The story claims `packages/db`'s migration/schema code has "no dependency on `@project-vault/crypto`" based on a grep of `packages/db/src/scripts/` and `packages/db/src/lib/` only, then dismisses the two schema files that *do* import it (`schema/credential-versions.ts`, `schema/pending-imports.ts`) as "never touched by `guarded-migrate.ts`." This reasoning is wrong for a `tsc` build unit: `packages/db/tsconfig.json`'s `include` is `src/**/*.ts` (the whole package, not just the migrate script's import graph), and `packages/db/package.json` itself lists `@project-vault/crypto` as a formal (non-dev) `dependencies` entry. I reproduced this directly: with `packages/crypto/dist` absent (its untouched, pre-existing state in this worktree), running `pnpm --filter @project-vault/shared build && pnpm --filter @project-vault/db build` — the *exact* command AC-D1 specifies for the new `db-builder` stage — fails with `error TS2307: Cannot find module '@project-vault/crypto' or its corresponding type declarations` in both files. AC-D1's own "positive example" ("`docker build --target db-builder` ... succeeds and produces an image containing `packages/shared/dist/` and `packages/db/dist/`") is therefore false as specified, and this cascades to break AC-D2 (extends a stage that never successfully builds), AC-D3 (retargeting `migrate` to `db-builder` breaks the `migrate` service outright), and AC-D5's claim that `make bootstrap-docker`/`make docker-up`/`make docker-smoke` "succeed end-to-end exactly as before." (Build artifacts created during this verification were removed via `git clean` before completing this review; the working tree is unmodified.)

- **[high] AC-T4's migration bumps `updated_at` on every non-empty-tagged `credentials`/`projects` row, an unaddressed side effect.** Both tables have a `set_updated_at BEFORE UPDATE` trigger (`packages/db/src/migrations/0014_credentials.sql` line 59, `0013_projects.sql` line 67) that fires on *any* `UPDATE`, including AC-T4's blanket backfill. AC-T4's own "already all-lowercase" example acknowledges the row is "still physically rewritten... acceptable given this is a one-time backfill" but only discusses write cost, never that `updated_at` — a plausible "last modified" signal elsewhere in the product — will jump to migration time for every tagged credential/project in the system, including ones the migration doesn't actually change semantically. This is never mentioned in the ACs, Dev Notes, or the Product Surface Contract table (which describes Group T's only evaluator-visible effect as "typing `Prod` now visibly renders back as `prod`").

- **[medium] AC-T3's "no code change needed" correctness argument is migration-order-dependent, and the story's own sanctioned deploy order can violate it.** AC-T3 claims `tagDelta` needs no change because "`oldTags` and `newTags` are always both-lowercase by the time `tagDelta` runs," reasoning that depends on AC-T4's migration having already run. Dev Notes explicitly states "Group T's migration order does not matter relative to Group T's code changes ... landing the migration without the code change (or vice versa) is safe, just incompletely effective." But landing AC-T1/AC-T2 (code) *before* AC-T4 (migration) — a sequence Dev Notes calls safe — means every pre-existing mixed-case row is still un-migrated while the new lowercase-on-write code is live: exactly the phantom `added`/`removed` diff AC-T3 claims to have eliminated recurs for every such row until the migration also lands. "Incompletely effective" understates that this specific split-deploy order actively reintroduces the bug AC-T3 is written to close.

- **[medium] AC-P2's three newly-named "historical incident" fixture tests add no new detection coverage.** `scanStoryStatusSync` (confirmed present, unmodified logic) already has generic tests for a single mismatch and for multiple simultaneous mismatches (per the story's own description of the existing suite). The CP4-4 shape (one mismatch) and A6-3 shape (three simultaneous mismatches) are structurally identical to what's already covered — AC-P2 relabels existing coverage classes with incident names rather than closing any gap in what the checker can catch. This is legitimate traceability/documentation value, but the story frames it as "hardening" a check with a real gap, when — per the story's own Background section — the check's *logic* is already complete; only the label is new.

- **[medium] Group D's Dev Notes explicitly disclaim cross-group coupling, but AC-D5's own example introduces it.** The story's header comment states groups "are fully self-contained: read only the group you're implementing... even a partial PR (e.g. just Group T) is safe to ship on its own... there are no [cross-group dependencies]." AC-D5's "positive — full smoke path" example nonetheless requires the `migrate` container's logs to show migration `0043` "(once AC-T4 exists)" among the applied set — folding a Group T deliverable into a Group D acceptance example, in tension with the story's own independence claim (harmless as a forward-reference, but a real inconsistency in a story that repeatedly emphasizes strict group isolation).

- **[low] The story's own diligence missed the cheapest signal that would have caught the Group D bug.** Before writing any code or running any build, `packages/db/package.json`'s `dependencies` block already lists `@project-vault/crypto: workspace:*` as a formal, non-dev dependency — a one-line read that contradicts the "no dependency on crypto" framing well before the deeper `tsc`-unit reasoning is needed. The story's "exhaustive" grep was scoped to two subdirectories and never checked the package manifest or attempted the actual build command it prescribes.

- **[low] No fallback or troubleshooting path exists for when the `db-builder` split fails to build.** Task 1.3 says only "confirm both succeed" — there is no guidance for what an implementer should do upon hitting the `TS2307` failure this review reproduces (e.g., whether to also build `crypto` inside `db-builder`, which would defeat the stated optimization goal, or exclude `credential-versions.ts`/`pending-imports.ts` from the `db` package's build unit via a separate tsconfig/entrypoint). As written, a developer following this story hits a wall with no specced way out.

- **[low] AC-P1/AC-P3 are close to tautological given how the check already behaves.** `scanStoryStatusSync` already scans every file under `_bmad-output/implementation-artifacts/`, so this story's own file was already going to be checked with zero extra code — AC-P3's "dogfooding" framing implies a special new safety property being added, when it is simply an instance of behavior the check already has for every other story file. Low risk, but oversells the marginal value of Group P's most "story-specific" AC.

- **[low] Tag ordering is silently non-deterministic across the migration boundary, while write-path ordering has an explicit contract.** AC-T1 specifies `dedupeTags` preserves "first-occurrence order" for every new write, and AC-T6 asserts an exact order (`['prod', 'staging']`) for the HTTP round-trip. AC-T4's backfill uses `jsonb_agg(DISTINCT lower(elem))`, which provides no ordering guarantee over existing rows' pre-migration order. The story dismisses this ("tag order was never a documented contract... only tag set membership is meaningful") but never surfaces to the Product Surface Contract or persona journey that an existing user's tag list can be silently reordered the moment this migration runs — a real, if minor, user-visible discontinuity the "Alex" persona journey (which only describes the case-collapse behavior) doesn't mention.

- **[low] Verification rigor is inconsistent between groups making the same class of claim.** Group C's dead-code claim (AC-C1) is backed by a single, literal, copy-pasteable verification command (`grep -rl "PlaceholderSection\|placeholder-copy\|getPlaceholderSection" apps/web/src`) — independently re-run for this review and confirmed to return exactly the three files claimed. Group D and Group T's comparably strong claims ("zero matches," "exhaustive list," "zero existing test fixtures use non-lowercase tags" — also independently verified true) are asserted in prose without an equivalent single reproducible command, making Group D's now-disproven claim (see first finding) harder to have caught by inspection alone.
